ภาพรวมชุดวิชา 99316 การวิเคราะห์และออกแบบระบบเชิงวัตถุ

ชุดวิชานี้เกี่ยวกับอะไร

ชุดวิชา 99316 "การวิเคราะห์และออกแบบระบบเชิงวัตถุ" (Object-Oriented Analysis and Design — OOAD) สอนวิธีพัฒนาซอฟต์แวร์ด้วยแนวคิด "เชิงวัตถุ" (Object-Oriented) ซึ่งมองระบบเป็นกลุ่มของ "วัตถุ" (Object) ที่มีทั้งข้อมูลและพฤติกรรมอยู่ด้วยกัน แทนการมองเป็นลำดับขั้นตอนแบบเชิงโครงสร้าง (Structured) หัวใจของวิชาคือการใช้ภาษาภาพมาตรฐาน "ยูเอ็มแอล" (UML) เขียนแผนภาพเพื่อวิเคราะห์และออกแบบระบบ ตั้งแต่เก็บความต้องการไปจนถึงทดสอบและส่งมอบ

ภาพใหญ่: วงจรการพัฒนาเชิงวัตถุ (OOA → OOD → OOP → OOT)

ให้จำลำดับ 4 กิจกรรมหลักของการพัฒนาเชิงวัตถุ ซึ่งเป็นโครงของทั้งวิชา

  • OOA — การวิเคราะห์เชิงวัตถุ (Object-Oriented Analysis) ตอบคำถามว่าระบบต้อง "ทำอะไร" (What) เน้นเข้าใจปัญหาและความต้องการ เช่น เขียนแผนภาพยูสเคส (Use Case) และค้นหาคลาสในโดเมนปัญหา
  • OOD — การออกแบบเชิงวัตถุ (Object-Oriented Design) ตอบว่าจะ "ทำอย่างไร" (How) ออกแบบคลาส ความสัมพันธ์ สถาปัตยกรรม และฐานข้อมูลถาวร
  • OOP — การเขียนโปรแกรมเชิงวัตถุ (Object-Oriented Programming) แปลงแบบเป็นโค้ดด้วยภาษาเชิงวัตถุ
  • OOT — การทดสอบเชิงวัตถุ (Object-Oriented Testing) ตรวจสอบความถูกต้องของระบบ

เสาหลัก 4 ประการของแนวคิดเชิงวัตถุ

ทุกอย่างในวิชานี้ตั้งอยู่บนคุณสมบัติ 4 ข้อ ที่ต้องจำให้แม่น

  • การห่อหุ้ม (Encapsulation) รวมข้อมูลและพฤติกรรมไว้ในวัตถุเดียว และซ่อนรายละเอียดภายใน
  • การสืบทอด (Inheritance) คลาสลูกรับคุณสมบัติจากคลาสแม่ ใช้ซ้ำได้
  • พหุสัณฐาน (Polymorphism) ชื่อเมท็อดเดียวกันทำงานต่างกันได้ตามวัตถุ
  • นามธรรม (Abstraction) ดึงเฉพาะสาระสำคัญของสิ่งหนึ่งมาเป็นคลาส

UML คือเครื่องมือกลางของวิชา

ยูเอ็มแอล (UML — Unified Modeling Language) ดูแลโดยองค์กร OMG พัฒนาจากผลงานของ "Three Amigos" (Booch, Rumbaugh/OMT, Jacobson/OOSE) ใช้เขียนแผนภาพมาตรฐาน 9 ชนิด แบ่งเป็นกลุ่มเชิงโครงสร้าง (Structural) และเชิงพฤติกรรม (Behavioral) แผนภาพที่ออกสอบบ่อยในวิชานี้ ได้แก่ ยูสเคส (Use Case), คลาส (Class), ลำดับ (Sequence), การสื่อสาร (Communication), สถานะ (State Machine), กิจกรรม (Activity), คอมโพเนนต์ (Component) และดีพลอยเมนต์ (Deployment)

แผนที่ 15 หน่วย (แบ่งเป็น 4 กลุ่ม)

อ่านเป็นกลุ่มจะเห็นภาพและจำง่ายขึ้น

  • กลุ่มที่ 1 พื้นฐานแนวคิดเชิงวัตถุ (หน่วย 1–3): ภาพรวม OOAD (หน่วย 1), แนวคิดเรื่องวัตถุและคลาส (หน่วย 2), การวิเคราะห์เชิงนามธรรม (หน่วย 3)
  • กลุ่มที่ 2 กระบวนการพัฒนาและ UML (หน่วย 4–6): กระบวนการพัฒนา/Unified Process (หน่วย 4), ยูเอ็มแอล (หน่วย 5), ยูสเคส (หน่วย 6)
  • กลุ่มที่ 3 แผนภาพและการวิเคราะห์/ออกแบบ (หน่วย 7–11): แผนภาพคลาส (หน่วย 7), การวิเคราะห์เชิงวัตถุ (หน่วย 8), การออกแบบเชิงวัตถุ (หน่วย 9), แผนภาพปฏิสัมพันธ์ (หน่วย 10), แผนภาพสถานะและกิจกรรม (หน่วย 11)
  • กลุ่มที่ 4 การทดสอบ เครื่องมือ และกรณีศึกษา (หน่วย 12–15): การทดสอบระบบ (หน่วย 12), เครื่องมือวิศวกรรมซอฟต์แวร์ (หน่วย 13), กรณีศึกษาวิเคราะห์/ออกแบบ (หน่วย 14), การออกแบบ UI และสถาปัตยกรรม (หน่วย 15)

วิธีใช้เอกสารนี้ให้ได้ผลก่อนสอบ

อ่านภาพรวมนี้ให้เห็นภาพใหญ่ก่อน แล้วค่อยลงรายละเอียดทีละหน่วย แต่ละหน่วยจัดเป็น 5 ส่วนเหมือนกันหมด คือ ภาพรวม → แนวคิดสำคัญ → เนื้อหาหลัก → คำศัพท์และตัวย่อ → ประเด็นที่ควรเน้นเพื่อสอบ วิชานี้เน้น "แผนภาพ" มาก จึงควรฝึกอ่านและจำสัญลักษณ์/ความสัมพันธ์ของแต่ละแผนภาพให้ได้ เมื่ออ่านครบแล้วให้ย้อนมาทบทวนที่ "อภิธานศัพท์รวม" และ "ชีตทบทวนด่วนก่อนสอบ" ท้ายเล่ม

ภาพรวม

หน่วยนี้เป็นพื้นฐานของทั้งชุดวิชา ปูความเข้าใจว่า "วิศวกรรมซอฟต์แวร์เชิงวัตถุ" (Object-Oriented Software Engineering: OOSE) คืออะไร และทำไมแนวทางเชิงวัตถุ (Object-Oriented Approach) จึงเข้ามาแทนที่แนวทางเชิงโครงสร้าง (Structured Approach) แบบดั้งเดิม เนื้อหาครอบคลุมความหมายของหลักการเชิงวัตถุ (Object Oriented: OO) คุณสมบัติสำคัญ 5 ข้อของวัตถุ ขั้นตอนการวิเคราะห์/ออกแบบเชิงวัตถุ ตลอดจนเครื่องมือหลักคือ ยูเอ็มแอล (UML) และเคสทูล (CASE Tools) และจบด้วยตัวอย่างระบบงานสารสนเทศทางธุรกิจ

แนวคิดสำคัญ

หัวใจของหน่วยนี้คือ "การมองทุกสิ่งในโลกเป็นวัตถุ (Object)" ที่มีความสัมพันธ์ (Relationship) และปฏิสัมพันธ์ (Interaction) ต่อกัน แล้วนำมุมมองนี้มาประยุกต์กับการพัฒนาซอฟต์แวร์ แทนที่จะแยกข้อมูลกับกระบวนการออกจากกันแบบเชิงโครงสร้าง แนวคิดเชิงวัตถุจะรวมข้อมูล (Data) กับวิธีการ (Method) ไว้ในวัตถุเดียวกัน ทำให้นำกลับมาใช้ใหม่ได้ (Reusable) ลดเวลาและค่าใช้จ่ายในการพัฒนา และรองรับระบบที่ซับซ้อนและเปลี่ยนแปลงบ่อยได้ดีกว่า

เนื้อหาหลัก

1.1 แนวคิดเกี่ยวกับการวิเคราะห์และออกแบบระบบเชิงวัตถุ

วิศวกรรมซอฟต์แวร์ (Software Engineering: SE) หมายถึง การนำหลักวิชาการด้านวิศวกรรมมาดูแลกระบวนการผลิตซอฟต์แวร์ ตั้งแต่ขั้นตอนแรกจนถึงการบำรุงรักษาหลังใช้งาน เพื่อให้ได้ซอฟต์แวร์คุณภาพสูงสุดภายใต้ข้อจำกัดด้านเวลาและต้นทุน ศัพท์นี้ปรากฏครั้งแรกในการประชุม NATO ปี ค.ศ. 1968 เพื่อแก้ "วิกฤติการณ์ซอฟต์แวร์ (Software Crisis)"

  • ส่วนประกอบของวิศวกรรมซอฟต์แวร์ มี 2 ส่วน คือ 1) วิศวกรรมระบบ (System Engineering) ศึกษา/วิเคราะห์ระบบที่ซับซ้อน และ 2) วิศวกรรมการผลิต (Development Engineering) แปรความต้องการของระบบให้กลายเป็นซอฟต์แวร์
  • คุณลักษณะที่ดีของกระบวนการ SE เช่น เข้าใจง่าย (Understandability) ชัดเจน (Visibility) มีเครื่องมือสนับสนุน (Supportability) ยอมรับได้ (Acceptability) เชื่อถือได้ (Reliability) แข็งแรง (Robustness) ปรับเปลี่ยนได้ (Maintainability) และรวดเร็ว (Rapidity)

เปรียบเทียบ 2 แนวทางพัฒนาซอฟต์แวร์ (ออกสอบบ่อย):

  • *แนวทางเชิงโครงสร้าง (Structured Approach)* เน้นที่กระบวนการ (Process) แบ่งระบบเป็นระบบย่อย (Sub-System) เป็นลำดับชั้น มีโครงสร้างควบคุมพื้นฐาน 3 แบบ คือ ตามลำดับ (Sequence) เลือกตามเงื่อนไข (Decision) และทำซ้ำ (Loop) ระเบียบวิธีที่นิยมคือ SSAD (Structured System Analysis and Design) คิดค้นโดย Yourdan & Demarco ปี ค.ศ. 1978 ข้อเสียคือต้องวิเคราะห์ข้อมูลและพฤติกรรมแยกกัน ใช้เวลานาน ต้นทุนสูง นำกลับมาใช้ใหม่ได้น้อย และเสี่ยงต่อความต้องการที่เปลี่ยนแปลง
  • *แนวทางเชิงวัตถุ (Object-Oriented Approach)* เน้นที่วัตถุ (Objects) คิดค้นโดย Grady Booch, James Rumbaugh และ Ivar Jacobson มองทุกอย่างเป็นวัตถุที่มีข้อมูลและพฤติกรรม จัดกลุ่มเป็นคลาส (Class) แต่ละคลาสมีสถานะ (State) พฤติกรรม (Behavior) และห่อหุ้ม (Encapsulate) คุณสมบัติไว้ ติดต่อกันด้วยข่าวสาร (Message) ความเสี่ยงต่ำ นำกลับมาใช้ใหม่ได้สูง เหมาะกับระบบใหญ่ที่เปลี่ยนแปลงบ่อย แต่ต้องใช้ผู้เชี่ยวชาญสูง

กระบวนการ OOSE มี 4 ขั้นตอน: 1) การวิเคราะห์เชิงวัตถุ (Object Oriented Analysis: OOA) 2) การออกแบบเชิงวัตถุ (Object Oriented Design: OOD) 3) การพัฒนา/เขียนโปรแกรมเชิงวัตถุ (Object Oriented Programming: OOP) และ 4) การทดสอบเชิงวัตถุ (Object Oriented Testing: OOT) โดย OOA เริ่มจากวิเคราะห์ความต้องการ (Requirement Analysis) เน้น "อะไร" ที่ระบบต้องทำ (ยังไม่เน้น "อย่างไร") ใช้แผนภาพยูสเคส (Use Case) ส่วน OOD กำหนดรายละเอียดเชิงเทคนิค เช่น สถาปัตยกรรม (2-Tier, 3-Tier, Multi-Tier) เพื่อนำไปเขียนโปรแกรมจริง

คุณสมบัติสำคัญ 5 ข้อของวัตถุ (Alan Kay) — จุดออกสอบสำคัญ:

  1. เอกลักษณ์ (Identity) — วัตถุแต่ละตัวมีลักษณะเฉพาะประจำตัว แม้สร้างจากคลาสเดียวกันและมีคุณลักษณะเหมือนกันก็ถือเป็นคนละวัตถุ
  2. การห่อหุ้ม (Encapsulation) — มองวัตถุเป็นกล่องดำ (Black box) วัตถุอื่นไม่ต้องรู้การทำงานภายใน เรียกใช้ได้ผ่านการส่งข่าวสารตามรูปแบบที่ตกลงกัน
  3. การซ่อนรายละเอียด (Information Hiding) — ปกปิดข้อมูลและวิธีการภายใน เข้าถึงได้ผ่านเมธอดเท่านั้น ใน UML ใช้สัญลักษณ์ + (Public), - (Private), # (Protected)
  4. การรับทอด (Inheritance) — กำหนดคุณสมบัติวัตถุใหม่โดยรับทอดจากวัตถุที่มีอยู่แล้ว ช่วยให้โครงสร้างเป็นระบบ ลดเวลาและค่าใช้จ่าย
  5. พอลิมอร์ฟิซึม (Polymorphism) — การพ้องรูป วัตถุที่มีพฤติกรรมชื่อเดียวกัน เช่น calculateArea() แต่วิธีการกระทำภายในต่างกันตามชนิดของวัตถุ

นอกจากนี้ยังอธิบาย ซอฟต์แวร์เชิงวัตถุ (Object-Oriented Software) ว่าคือกลุ่มของวัตถุที่จัดการข้อมูลของตนเองและสื่อสารกันด้วยข่าวสาร (Message) ในลักษณะไคลเอนต์-เซิร์ฟเวอร์ (Client-Server)

1.2 เครื่องมือการวิเคราะห์และออกแบบระบบเชิงวัตถุ

ยูเอ็มแอล (Unified Modeling Language: UML) คือ ภาษาสัญลักษณ์รูปภาพมาตรฐานสำหรับสร้างแบบจำลอง (Modeling) เชิงวัตถุ เปรียบเป็น "แบบพิมพ์เขียว" ของระบบงาน

  • *ประวัติ:* เกิดจากสงครามเมธอด (Method Wars) ของ 3 วิธี คือ OOSE (Jacobson), OMT (Rumbaugh), Booch Method (Booch) ทั้งสามคน ("Three Amigos") รวมแนวคิดที่ Rational Software และเสนอต่อ OMG จนได้ UML 1.1 ในปี ค.ศ. 1997
  • *ข้อดี:* รองรับตลอดวงจรพัฒนา ใช้กับหลายโดเมน ไม่ผูกติดภาษาหรือแพลตฟอร์ม ใช้ร่วมกับกรรมวิธีใดก็ได้

แผนภาพ UML แบ่งเป็น 2 ประเภทใหญ่: 1) แผนภาพเชิงโครงสร้าง/เชิงสถิต (Structural/Static Diagram) และ 2) แผนภาพเชิงพฤติกรรม/เชิงพลวัต (Behavioral/Dynamic Diagram) รวม 9 ชนิด ได้แก่ แผนภาพยูสเคส (Use Case), แผนภาพคลาส (Class), แผนภาพวัตถุ (Object), แผนภาพลำดับ (Sequence), แผนภาพประสานงาน (Collaboration), แผนภาพสถานะ (State Chart), แผนภาพกิจกรรม (Activity), แผนภาพคอมโพเนนต์ (Component), และแผนภาพการดีพลอยเมนต์ (Deployment)

เคสทูล (CASE Tools) คือ เครื่องมือช่วยให้นักวิเคราะห์ระบบเขียนแผนภาพต่างๆ ได้ง่ายขึ้น ตั้งแต่การวิเคราะห์ ออกแบบ พัฒนา ทดสอบ ไปจนถึงสร้างคำสั่งโปรแกรมให้อัตโนมัติ

1.3 ตัวอย่างระบบงานสารสนเทศทางธุรกิจ

ยกตัวอย่างระบบสำคัญ 2 ระบบ คือ 1) ระบบการขายปลีก มีกิจกรรมซื้อสินค้าจากผู้ผลิต ขายปลีกแก่ลูกค้า (ขายหน้าร้านทันที หรือส่งสินค้าให้ลูกค้า) และดูแลสินค้าคงคลัง และ 2) ระบบบริหารศูนย์บริการรถยนต์ ตั้งแต่ลูกค้าโทรนัด ส่งรถ ตรวจสอบอาการ เบิกอะไหล่ ซ่อม คิดค่าบริการ จนถึงชำระเงินและรับรถคืน ใช้เป็นความรู้พื้นฐานสำหรับหน่วยต่อๆ ไป

ประเด็นที่ควรเน้นเพื่อสอบ

  • คุณสมบัติ 5 ข้อของวัตถุตาม Alan Kay (Identity, Encapsulation, Information Hiding, Inheritance, Polymorphism) — นิยามและตัวอย่างของแต่ละข้อ
  • เปรียบเทียบแนวทางเชิงโครงสร้าง vs เชิงวัตถุ ในแง่ จุดเน้น (Process vs Objects) ความเสี่ยง การนำกลับมาใช้ใหม่ และความเหมาะสม
  • ความหมายของ วิศวกรรมซอฟต์แวร์ และ OOSE พร้อม 4 ขั้นตอน (OOA, OOD, OOP, OOT)
  • ผู้คิดค้นแนวทางเชิงวัตถุ: Booch, Rumbaugh, Jacobson ("Three Amigos") และที่มาของ UML
  • แผนภาพ UML 9 ชนิด และการแบ่งเป็น 2 ประเภท (เชิงโครงสร้าง/สถิต กับ เชิงพฤติกรรม/พลวัต)
  • สัญลักษณ์ระดับการเข้าถึง (Visibility) ใน UML: + Public, - Private, # Protected
  • โครงสร้างควบคุมพื้นฐานเชิงโครงสร้าง 3 แบบ: Sequence, Decision, Loop
  • ความหมายของ เคสทูล (CASE Tools) และบทบาทช่วยงานวิเคราะห์/ออกแบบ
  • ความแตกต่างระหว่างความสัมพันธ์ (Relationship — มองไม่เห็น ต้องตีความ) กับ ปฏิสัมพันธ์ (Interaction — สังเกตเห็นได้)
  • ประโยชน์ของหลักการเชิงวัตถุ: ลดความซับซ้อน ยืดหยุ่น แก้ไข/เพิ่มเติมง่าย นำกลับมาใช้ใหม่ได้

หน่วยที่ 2: ความรู้เบื้องต้นเกี่ยวกับวัตถุ

ภาพรวม

หน่วยนี้เจาะลึก "วัตถุ (Object)" และ "คลาส (Class)" ซึ่งเป็นแก่นของแนวคิดเชิงวัตถุ (Object-Oriented Concept) อธิบายว่าวัตถุคืออะไร ต่างจากคลาสอย่างไร มีองค์ประกอบอะไรบ้าง (คุณลักษณะและพฤติกรรม) และกระบวนการแปลงวัตถุในโลกจริงเข้าสู่โลกคอมพิวเตอร์ผ่านนามธรรม (Abstraction) และการสร้างวัตถุตัวแทน (Instantiation) จากนั้นต่อยอดสู่คุณสมบัติเชิงวัตถุ 4 ข้อ และความสัมพันธ์ระหว่างคลาส 4 รูปแบบ

แนวคิดสำคัญ

แนวคิดเชิงวัตถุ (Object Orientation) มาจากคำว่า Object (วัตถุ) + Orientation (การปรับทิศทาง/แนวคิด) หมายถึงการใช้วัตถุเป็นหลักในการพิจารณาความจริงภายใน "ขอบเขตของปัญหา (Problem Domain)" ที่เราสนใจ จุดสำคัญที่สุดคือต้องแยกให้ออกว่าอะไรคือ "วัตถุ" (สิ่งที่เจาะจง มีตัวตน ทำงานได้) และอะไรคือ "คลาส" (กลุ่ม/แม่พิมพ์ที่เป็นนามธรรม ทำงานเองไม่ได้) คลาสเปรียบเสมือน Template ที่ใช้สร้างวัตถุตัวแทน (Object Instance) ขึ้นมาทำงานจริงในระบบคอมพิวเตอร์

เนื้อหาหลัก

2.1 แนวคิดเชิงวัตถุ

วัตถุและคลาส (เรื่องที่ 2.1.1):

  • วัตถุ (Object) คือ สิ่งที่เป็นรูปธรรมหรือนามธรรมในโลกจริงที่นำมากำหนดในโลกคอมพิวเตอร์ได้ ประกอบด้วย คุณลักษณะ (Attribute) และพฤติกรรม (Behavior) แบ่งเป็น 2 ประเภท คือ 1) วัตถุรูปธรรม/จับต้องได้ (Tangible Objects) เช่น คน รถยนต์ เสื้อผ้า และ 2) วัตถุนามธรรม/จับต้องไม่ได้ (Intangible Objects) เช่น กฎหมาย ภาพยนตร์ เวลา สี
  • ขอบเขตของปัญหา (Problem Domain / โดเมน) คือ ขอบเขตที่เราสนใจการทำงานของวัตถุที่ทำงานร่วมกัน เนื่องจากเราสนใจทุกสิ่งในโลกพร้อมกันไม่ได้ จึงต้องกำหนดกรอบ โดเมนหนึ่งมีวัตถุได้ตั้งแต่ 2 ตัวขึ้นไป และวัตถุหนึ่งอยู่ได้หลายโดเมน
  • ความสัมพันธ์ (Relationship) คือ ความเกี่ยวข้องระหว่างวัตถุตั้งแต่ 2 ตัวขึ้นไป มักมองไม่เห็น ต้องตีความ (เช่น เป็นแม่-ลูก เป็นเจ้าของ) ส่วน ปฏิสัมพันธ์ (Interaction) คือ การกระทำที่เกิดขึ้นระหว่างวัตถุ มักสังเกตเห็นได้ (เช่น เปิด หยิบ กิน) — *ตัวอย่าง:* "มานีเปิดตู้เสื้อผ้าแล้วหยิบกระโปรง" → วัตถุ = มานี, ตู้เสื้อผ้า, กระโปรง; ความสัมพันธ์ = มานีเป็นเจ้าของตู้/กระโปรงอยู่ในตู้; ปฏิสัมพันธ์ = เปิด, หยิบ
  • คลาส (Class) คือ กลุ่มของวัตถุที่ได้จากการให้แนวคิด (Concept) มีคุณลักษณะ/พฤติกรรมคล้ายกัน เรียกอีกชื่อว่า Abstract Objects เปรียบเป็นแม่พิมพ์ (Template) คลาสเป็นนามธรรม ทำกิจกรรมเองไม่ได้ ("รถวิ่งบนถนน" เป็นเหตุการณ์นามธรรม แต่ "รถของนายสมชายวิ่งบนถนนแจ้งวัฒนะ" เป็นเหตุการณ์จริง เพราะระบุวัตถุเจาะจง)
  • นามธรรม (Abstraction) = กระบวนการให้แนวคิดกับวัตถุจนเกิดเป็นคลาส; การสร้างวัตถุตัวแทน (Instantiation) = กระบวนการสร้างวัตถุ (Object/Instance) จากคลาส แผนภาพสำคัญแสดงการแปลงจาก Real World (Object Domain) → Abstraction → Class Domain (Abstract Level) → Instantiation → Computer World

องค์ประกอบของวัตถุและคลาส (เรื่องที่ 2.1.2):

  • คุณลักษณะ (Attribute) = ข้อมูลที่ใช้อธิบายและเก็บค่าของวัตถุ เช่น รถมี Attribute ยี่ห้อ รุ่น สี
  • พฤติกรรม (Behavior) = สิ่งที่วัตถุทำได้ ในการเขียนโปรแกรมเชิงวัตถุเรียก เมธอด (Method) เช่น รถยนต์มีพฤติกรรม สตาร์ตเครื่อง ขับเคลื่อน หยุดรถ เปลี่ยนเกียร์ — สิ่งสำคัญคือการกำหนด Attribute/Behavior ต้องสอดคล้องกับขอบเขตปัญหา/ความต้องการของระบบ (เช่น ระบบซื้อขายรถสนใจ ยี่ห้อ/รุ่น/ราคา ส่วนระบบโรงงานผลิตสนใจส่วนประกอบ)

2.2 คุณสมบัติและความสัมพันธ์

คุณสมบัติเชิงวัตถุ 4 ข้อ (เรื่องที่ 2.2.1) — จุดออกสอบหลัก:

  1. เอกลักษณ์ของวัตถุ (Object Identity) — สิ่งที่ใช้ระบุวัตถุให้จำเพาะ ไม่ซ้ำกัน (Uniqueness) วัตถุในคลาสเดียวกันต้องมีเอกลักษณ์ต่างกัน บางคุณลักษณะ เช่น รหัสนักศึกษา หรือ ISBN ใช้เป็น Identity ได้
  2. การห่อหุ้มและการซ่อนข้อมูล (Encapsulation & Data Hiding) — กำหนดขอบเขตการเปิดเผยข้อมูล วัตถุอื่นเรียกใช้ได้เฉพาะที่อนุญาต แบ่งการมองเห็น/เข้าถึง (Visibility and Accessibility) เป็น 3 แบบ: Private (มองไม่เห็นจากภายนอก เข้าถึงผ่านพฤติกรรมสาธารณะเท่านั้น), Protected (มองเห็นเฉพาะคลาสที่รับทอดมา), Public (มองเห็น/แก้ไข/เรียกใช้ได้จากทุกวัตถุภายนอก) — มองภายใน (Inside view) vs ภายนอก (Outside view) ประโยชน์คือความปลอดภัย ป้องกันผิดพลาด และบำรุงรักษาง่าย (Maintainability)
  3. พอลิมอร์ฟิซึม (Polymorphism) — มาจาก Poly (มากมาย) + Morph (รูปแบบ) วัตถุมีพฤติกรรมชื่อเดียวกันแต่รายละเอียดการกระทำต่างกัน เช่น "รูปสี่เหลี่ยม" กับ "รูปวงกลม" ต่างคำนวณพื้นที่ได้แต่วิธีต่างกัน หรือเฮลิคอปเตอร์/เครื่องบิน/บอลลูน/จรวด ต่างบินได้แต่วิธีต่างกัน ใช้คู่กับการรับทอด ช่วยเรื่อง Software Interoperability
  4. การรับทอด (Inheritance) — ความสามารถถ่ายทอดคุณลักษณะ/พฤติกรรมจากคลาสพ่อแม่ (Parent class / Superclass) สู่คลาสลูก (Child Class / Subclass) ซึ่งคลาสลูกเพิ่มคุณลักษณะ/พฤติกรรมใหม่ได้ Superclass มีความทั่วไป (Generalization) มากกว่า Subclass ที่มีความจำเพาะ (Specialization) มากกว่า แบ่งเป็น 2 แบบ: 1) การรับทอดเดียว (Single Inheritance) มีคลาสพ่อแม่เดียว และ 2) การรับทอดหลายส่วน (Multiple Inheritance) มีคลาสพ่อแม่มากกว่า 1 — ประโยชน์: หลีกเลี่ยงความซ้ำซ้อน (Avoiding redundancies) ลดขนาด/ความซับซ้อนของโค้ด นำกลับมาใช้ใหม่ได้

ส่วนประกอบของคลาสและความสัมพันธ์ระหว่างคลาส (เรื่องที่ 2.2.2):

คลาสมีส่วนประกอบหลัก คือ คุณลักษณะ (Attribute) และ พฤติกรรม/การดำเนินการ ซึ่งในคอมพิวเตอร์เรียก โอเปอเรชัน (Operation) เขียนแสดงด้วยแผนภาพคลาส (Class Diagram)

ความสัมพันธ์ระหว่างคลาส (Class Relationship) 4 รูปแบบหลัก:

  1. แบบทั่วไป (Generalization) — การรับทอดแบบ "เป็นประเภทหนึ่งของ" คลาสหนึ่งสืบทอดคุณลักษณะจากอีกคลาส เช่น คลาสที่อยู่อาศัย (Dwelling) มีคลาสย่อย บ้าน/คอนโด/ทาวน์โฮม
  2. แบบเป็นส่วนหนึ่ง (Aggregation) — คลาสหนึ่งมีส่วนประกอบเป็นคลาสย่อย โดยถ้านำส่วนย่อยออกไป คลาสหลักยังคงสภาพอยู่ได้ (เช่น ถาดใส่ไข่กับตู้เย็น, กระดานกระโดดน้ำกับสระว่ายน้ำ)
  3. แบบเป็นองค์ประกอบ (Composition) — คล้าย Aggregation แต่ส่วนย่อยขาดไม่ได้ ถ้านำออกคลาสหลักจะสูญเสียความเป็นคลาสนั้น (เช่น ล้อรถกับรถยนต์, ปีกกับเครื่องบิน) แสดงด้วยรูปสี่เหลี่ยมข้าวหลามตัดสีดำ
  4. แบบเกี่ยวข้องกัน (Association) — ความสัมพันธ์ส่วนใหญ่ในระบบ เป็นการทำงานร่วมกัน "is related to" เช่น คนเป็นเจ้าของรถ คนอาศัยในบ้าน คนเลี้ยงสัตว์

*ตัวอย่างระบบโรงพยาบาล:* บุคลากร→แพทย์/พยาบาล/เจ้าหน้าที่/รปภ. เป็น Generalization; ผู้ป่วย→ผู้ป่วยใน/ผู้ป่วยนอก เป็น Generalization; อาคารกับบุคลากร/ผู้ป่วย เป็น Association

  • องค์ประกอบของวัตถุ: คุณลักษณะ (Attribute) และพฤติกรรม (Behavior/Method)

  • ระดับการเข้าถึงข้อมูล 3 แบบ: Private, Protected, Public และความหมายของแต่ละแบบ

  • ความหมายของ ขอบเขตของปัญหา (Problem Domain) และทำไมต้องกำหนดกรอบความสนใจ

หน่วยที่ 3: การวิเคราะห์เชิงนามธรรม

ภาพรวม

หน่วยนี้ต่อยอดจากหน่วยที่ 2 โดยเจาะลึก "แนวคิด (Concept)" และ "นามธรรม (Abstraction)" ซึ่งเป็นกระบวนการสร้างคลาสจากวัตถุในโลกจริงเข้าสู่โลกคอมพิวเตอร์ หัวใจของหน่วยคือ ประเภทของนามธรรม 4 แบบ ที่ใช้ในการวิเคราะห์เพื่อสร้างคลาสและความสัมพันธ์ระหว่างคลาส พร้อมทั้งสอนขั้นตอนการวิเคราะห์อย่างเป็นระบบ และปิดท้ายด้วยตัวอย่างวิเคราะห์ระบบงานจริง 2 ระบบ (ตู้จำหน่ายตั๋วอัตโนมัติ และศูนย์บริการรถยนต์)

แนวคิดสำคัญ

"แนวคิด (Concept)" คือ ความคิดรวบยอดเกี่ยวกับสิ่งที่เราสนใจ เพื่อสรุปรายละเอียดสำคัญและใช้เป็นแนวทางสร้างวัตถุในคอมพิวเตอร์ ส่วน "นามธรรม (Abstraction)" คือ กระบวนการให้แนวคิดกับวัตถุต่างๆ ในโลกจริงภายในขอบเขตที่กำหนดเพื่อสร้างคลาส โดยจะให้ความสำคัญเฉพาะคุณลักษณะที่เกี่ยวข้องกับบริบทและละทิ้งคุณลักษณะที่ไม่เกี่ยวข้อง คลาส Person ในระบบจัดหางานกับระบบสินค้าออนไลน์จึงมีคุณลักษณะต่างกันได้ตามความต้องการของระบบ

เนื้อหาหลัก

3.1 แนวคิดและหลักการเชิงนามธรรม

ความหมายของแนวคิดและนามธรรม (เรื่องที่ 3.1.1):

  • แนวคิด (Concept) = ความคิดรวบยอดเกี่ยวกับสิ่งหนึ่งที่สนใจ เพื่อสรุปรายละเอียดสำคัญและใช้สร้างวัตถุในคอมพิวเตอร์
  • นามธรรม (Abstraction) = กระบวนการให้แนวคิดเกี่ยวกับวัตถุในโลกจริงภายในขอบเขตที่กำหนดเพื่อสร้างคลาส โดยเน้นคุณลักษณะที่เกี่ยวข้องกับบริบทและเพิกเฉยคุณลักษณะที่ไม่เกี่ยวข้อง จึงเกิดการจัดกลุ่มที่เรียก Abstraction Objects หรือ Class
  • ต้องแยก คลาส vs วัตถุ ให้ชัด (เช่น "อาหาร/เก้าอี้/ตัวการ์ตูน" เป็นคลาส; "ข้าวผัดไก่/เก้าอี้สีแดงของสมศรี/โดเรมอน" เป็นวัตถุ)
  • ความสัมพันธ์ระหว่างคลาสกับวัตถุ คือ "การเป็นสมาชิกของ (is a member of)" และวัตถุในคลาสย่อย (Subclass) ก็เป็นสมาชิกของคลาสใหญ่ด้วย
  • กระบวนการนามธรรมเพื่อสร้างคลาส: รวบรวมกลุ่มวัตถุ → สังเกตคุณสมบัติ/พฤติกรรมร่วม → กำหนดแนวคิด → สร้างคลาส (เช่น สร้างคลาส "ยานพาหนะ/Vehicle" จากแนวคิด "เคลื่อนที่ได้และโดยสารได้")

ประเภทของนามธรรม (เรื่องที่ 3.1.2) — แก่นสำคัญที่สุดของหน่วย แบ่งเป็น 4 ประเภท:

  1. นามธรรมแบบการจำแนก (Classification Abstraction) — กระบวนการให้แนวคิดเพื่อแยกวัตถุเป็นประเภทต่างๆ ทำให้ได้ "คลาสพื้นฐาน (Fundamental classes)" ถือเป็น นามธรรมที่สำคัญที่สุด เพราะเป็นจุดเริ่มต้นสร้างคลาส แสดงความสัมพันธ์แบบ "is a member of"
  2. นามธรรมแบบการเป็นส่วนหนึ่ง (Aggregation Abstraction) — หรือนามธรรมแบบการรวมกลุ่ม นำคลาสพื้นฐานมารวม/ประกอบกันให้เกิดคลาสที่ใหญ่/ซับซ้อนขึ้น แสดงความสัมพันธ์แบบ "is a part of" ส่วนประกอบที่สำคัญและขาดไม่ได้เรียก Composition (เช่น ฝาประตู/ชั้น/ช่องแช่แข็งของตู้เย็น) ส่วนประกอบที่ไม่สำคัญแยกออกได้ (เช่น อาหาร/ถาดใส่ไข่)
  3. นามธรรมแบบทั่วไป (Generalization Abstraction) — กำหนดคุณสมบัติร่วมของหลายคลาสเพื่อสร้างคลาสใหญ่ (Superclass) ตามลำดับชั้นคลาส (Class hierarchy) แสดงความสัมพันธ์แบบ "is a kind of" กระบวนการย้อนกลับคือ นามธรรมจำเพาะพิเศษ (Specialization Abstraction) ที่สร้างคลาสย่อย (Subclass) และรับทอดคุณสมบัติจาก Superclass (เช่น คลาส Shape → Rectangle/Circle/Triangle → Square/Diamond)
  4. นามธรรมแบบเกี่ยวข้องกัน (Association Abstraction) — สร้างความสัมพันธ์ระหว่างคลาสที่อยู่ระดับเดียวกัน มีความสำคัญเท่าเทียมกัน ไม่มีคลาสใดเป็นองค์ประกอบของอีกคลาส แสดงความสัมพันธ์แบบ "is related to" (เช่น คนขับรถยนต์ คนอาศัยในบ้าน คนเลี้ยงสัตว์)

สรุปความสัมพันธ์ของแต่ละนามธรรม (ตารางจำง่าย): Classification → is a member of | Aggregation → is a part of | Generalization → is a kind of | Association → is related to

3.2 หลักการวิเคราะห์เชิงนามธรรม

นามธรรมแบบการจำแนก (เรื่องที่ 3.2.1): ถ้าคลาสพื้นฐานในขั้นนี้ผิดพลาด การทำ Aggregation/Generalization/Association ต่อไปจะผิดตามไปด้วย มี 7 ขั้นตอน:

  1. กำหนดขอบเขตของปัญหา 2. หาคำนาม (เพื่ออธิบายวัตถุและคลาส) 3. แยกประเภทของคำนาม (ว่าเป็นวัตถุหรือคุณลักษณะ) 4. ให้แนวคิด (เพื่อให้ได้คลาสพื้นฐาน) 5. ระบุวัตถุ (ว่าวัตถุใดอยู่ในคลาสใด) 6. ระบุรายละเอียดของคลาส (คุณลักษณะและพฤติกรรม) 7. วาดแผนภาพแสดง Classification — *ตัวอย่าง:* "รถยนต์โตโยต้าคันหนึ่ง สีแดง มี 2 ประตู" → วัตถุ = รถยนต์; คุณลักษณะ = สี, จำนวนประตู; คลาส = รถยนต์

นามธรรมแบบการเป็นส่วนหนึ่ง (เรื่องที่ 3.2.2): ตอบคำถามว่าคลาสใดรวมกันเกิดคลาสใหม่ได้ หรือคลาสใดแบ่งเป็นส่วนย่อยได้ ขั้นตอนไม่ซับซ้อน เพียงพิจารณากลุ่มวัตถุที่เป็นส่วนประกอบของคลาสหลักแล้ววาดแผนภาพ (เช่น รถยนต์ประกอบด้วยล้อ/เครื่องยนต์/เบาะ/พวงมาลัย; เครื่องปรับอากาศประกอบด้วยคอมเพรสเซอร์/พัดลม/รีโมท)

นามธรรมแบบทั่วไป (เรื่องที่ 3.2.3): นำคลาสที่คล้ายกันหรือมีคุณสมบัติร่วมมาจัดเป็นคลาสเดียว (General) กระบวนการย้อนกลับ Specialization ทำให้จำเพาะเจาะจงขึ้น — *ตัวอย่าง:* คลาสรถยนต์ (Superclass มี ล้อ/เครื่องยนต์) → รถบรรทุก (+บรรทุกของได้) / รถเก๋ง (+4 ประตู) / รถตู้ (+12 ที่นั่ง) / รถสปอร์ต; คลาสสัตว์เลี้ยง → สัตว์เลี้ยงไว้เป็นอาหาร/สัตว์เลี้ยงในบ้าน

นามธรรมแบบเกี่ยวข้องกัน (เรื่องที่ 3.2.4): อธิบายความสัมพันธ์ที่ไม่ใช่ Aggregation หรือ Generalization เช่น การใช้งาน การขับเคลื่อน การเป็นเจ้าของ ช่วยกำหนดพฤติกรรมที่เรียกใช้ระหว่างคลาส และขอบเขตการเข้าถึงข้อมูล (ใช้ + อนุญาตเข้าถึง, - ไม่อนุญาต; พฤติกรรมลงท้ายด้วย () ) — *ตัวอย่าง:* ระบบเรียนการสอน คลาสนักศึกษา/อาจารย์/รายวิชา/งานทะเบียน/สถานศึกษา มีความสัมพันธ์ ศึกษาอยู่ที่/สอน/ลงทะเบียน/บันทึก

การจับคู่นามธรรมกับ abstraction (โจทย์ออกสอบบ่อย):

  • "สิ่งมีชีวิต แบ่งเป็น คน สัตว์ พืช" → Generalization/Specialization
  • "แมวสีแดง แมวสีขาว แมวสีดำ" → Classification
  • "คนนั่งเครื่องบิน" → Association
  • "โรงพยาบาลประกอบด้วย หมอ พยาบาล คนไข้ ห้องผ่าตัด" → Aggregation
  • "คนมีอวัยวะ แขน ขา ลำตัว ศีรษะ" → อธิบายด้วย Aggregation ได้ (แต่ใช้ Classification/Generalization/Association ไม่ได้)

3.3 ตัวอย่างการวิเคราะห์เชิงนามธรรม

ระบบงานตู้จำหน่ายตั๋วอัตโนมัติ (เรื่องที่ 3.3.1): วิเคราะห์ด้วยนามธรรมทั้ง 4 แบบ ได้คลาสพื้นฐาน คือ ลูกค้า, เจ้าหน้าที่, ตู้จำหน่ายตั๋ว, บัตรโดยสาร — Aggregation: บัตรโดยสารเป็นส่วนหนึ่งของตู้; Generalization: ลูกค้า+เจ้าหน้าที่ รวมเป็นคลาส "ผู้กระทำ"; Association: ลูกค้าใช้งานเครื่องเพื่อซื้อตั๋ว / เจ้าหน้าที่ดูแลซ่อมบำรุง

ระบบบริหารศูนย์บริการรถยนต์ (เรื่องที่ 3.3.2): วิเคราะห์ได้คลาส ลูกค้า, พนักงาน, รถยนต์, ช่างซ่อม, อะไหล่, พนักงานบัญชี — Aggregation: อะไหล่เป็นส่วนหนึ่งของรถยนต์; Generalization: พนักงาน/ช่างซ่อม/พนักงานบัญชี รวมเป็นคลาส "พนักงาน" (มีคลาสย่อยพนักงานบริการ/พนักงานซ่อมบำรุง/พนักงานบัญชี); Association: ลูกค้าติดต่อพนักงาน เป็นเจ้าของรถ เรียกช่างซ่อม เบิกอะไหล่

*หมายเหตุสำคัญ:* แผนภาพในหน่วยนี้เป็นแผนภาพอย่างง่ายเพื่อความเข้าใจ การวาดแผนภาพคลาส (Class Diagram) มาตรฐานจะศึกษาเพิ่มเติมในหน่วยที่ 7

ประเด็นที่ควรเน้นเพื่อสอบ

  • นามธรรม 4 ประเภท (Classification, Aggregation, Generalization, Association) — นิยาม ความสัมพันธ์ที่บ่งชี้ (is a member of / is a part of / is a kind of / is related to) และตัวอย่าง (ออกสอบแน่นอน)
  • นามธรรมแบบการจำแนก (Classification) เป็นนามธรรมที่สำคัญที่สุด เพราะสร้างคลาสพื้นฐาน
  • 7 ขั้นตอนของ Classification Abstraction (กำหนดขอบเขต → หาคำนาม → แยกประเภทคำนาม → ให้แนวคิด → ระบุวัตถุ → ระบุรายละเอียดคลาส → วาดแผนภาพ)
  • ความสัมพันธ์ระหว่าง Generalization กับ Specialization (กระบวนการย้อนกลับกัน)
  • ความต่างระหว่าง Aggregation (แยกออกได้) กับ Composition (แยกออกไม่ได้)
  • ความหมายของ แนวคิด (Concept) vs นามธรรม (Abstraction)
  • การ จับคู่ข้อความกับประเภทนามธรรม (เช่น "คอมพิวเตอร์มีส่วนประกอบ..." = Aggregation, "โทรศัพท์เป็นเครื่องใช้ไฟฟ้าประเภทหนึ่ง" = Classification + Generalization)
  • เทคนิคแยกคำนามในประโยค: คำนาม = วัตถุหรือคุณลักษณะ (และในหน่วย 1 คำกริยา→Operation, คำวิเศษณ์→Attribute)
  • คลาส Person ในแต่ละระบบมีคุณลักษณะต่างกันตามบริบท/ความต้องการของระบบ
  • คลาสพื้นฐานที่ได้จากตัวอย่างระบบตู้จำหน่ายตั๋ว และระบบศูนย์บริการรถยนต์

หน่วยที่ 4: กระบวนการพัฒนาระบบเชิงวัตถุ

ภาพรวม

หน่วยนี้อธิบายว่าการสร้างระบบงานคอมพิวเตอร์มี 2 แนวทางหลัก คือ การพัฒนาแบบดั้งเดิม/เชิงโครงสร้าง (Structural/Traditional) กับการพัฒนาเชิงวัตถุ (Object Oriented) แล้วเปรียบเทียบความแตกต่างทั้งด้านการวิเคราะห์ ออกแบบ และขั้นตอน จากนั้นเจาะลึก "กระบวนการยูนิไฟด์" (Unified Process) ซึ่งเป็นกระบวนการพัฒนาเชิงวัตถุที่ทำงานแบบวนรอบเพิ่มพูนและมีสถาปัตยกรรมเป็นศูนย์กลาง ประกอบด้วย 2 มิติ คือ มิติเวลา (ช่วงระยะเฟส) และมิติกระแสงาน (Workflows) โดยใช้ภาษา UML เป็นเครื่องมืออธิบายแบบจำลอง

แนวคิดสำคัญ

กระบวนการพัฒนาระบบแบบดั้งเดิมทำได้ยากและไม่ยืดหยุ่นเมื่อระบบใหญ่ขึ้น เพราะทุกส่วนงานเกี่ยวโยงกัน การพัฒนาเชิงวัตถุจึงเกิดขึ้นเพื่อปรับเปลี่ยนตามความต้องการของผู้ใช้ได้ง่าย โดยมองระบบเป็น "วัตถุ" (Object) ที่รวมข้อมูล/คุณลักษณะ (Attribute) กับการปฏิบัติ/พฤติกรรม (Behavior) ไว้เป็นหน่วยเดียว ตามหลักการแบ่งแยกส่วน (Decomposition) โดยพิจารณาขอบเขตของปัญหา (Problem Domain) ในโลกจริง (Real World) จุดสำคัญที่ต้องเข้าใจคือ กระบวนการยูนิไฟด์เป็นตัว "บ่งบอกว่างานหนึ่งๆ ณ ช่วงเวลาหนึ่ง ควรใช้แผนภาพ UML ใด"

เนื้อหาหลัก

4.1 กระบวนการพัฒนาระบบ

  • กระบวนการเชิงโครงสร้าง (SAD) เริ่มจากการวิเคราะห์และออกแบบเชิงโครงสร้าง (Structural Analysis and Design) มี 12 ขั้นตอน เด่นที่: กำหนดปัญหา/โครงการ → วิเคราะห์ความต้องการ → ออกแบบแผนภาพการไหลข้อมูล (DFD) → คำอธิบายกระบวนการ → พจนานุกรมข้อมูล (Data Dictionary) → แผนภาพ ER → จัดทำฐานข้อมูล → ออกแบบหน้าจอ → เขียนโปรแกรม → ทดสอบ → ถ่ายโอนงาน → แผนภาพดีพลอยเมนต์
  • กระบวนการเชิงวัตถุ (OOAD) เริ่มจาก Object Oriented Analysis and Design มี 12 ขั้นตอนคล้ายกัน แต่ส่วนที่ต่างคือ: วิเคราะห์ความต้องการ โดยใช้ยูสเคส (Use case) → ระบุคลาส (Class Identification) → ออกแบบแผนภาพคลาส (Class Diagram) → แปลงแผนภาพคลาสเป็น RDBMS → จัดทำแผนภาพความสัมพันธ์ (Sequence, Activity ฯลฯ) (ตารางที่ 4.1 เปรียบเทียบ SAD กับ OOAD)
  • วัฏจักรการพัฒนาระบบงาน (SDLC) สรุปรวบยอดจาก 12 ขั้นตอนเหลือ 7 ขั้นตอนหลัก: 1) การวางแผน (Planning) 2) การวิเคราะห์ (Analysis) 3) การออกแบบเชิงตรรกะ (Design) 4) การพัฒนาโปรแกรม (Implementation) 5) การทดสอบ (Testing) 6) การดำเนินงานและดีพลอยเมนต์ (Operation & Deployment) 7) การบำรุงรักษา (Maintenance)
  • แบบจำลอง (Model) การพัฒนาระบบ มี 2 แบบสำคัญ:

- แบบจำลองน้ำตก (Waterfall model) ดำเนินแต่ละขั้นต่อเนื่องเหมือนน้ำตกไหลจากที่สูงลงต่ำ แก้ไขย้อนกลับยาก ต้องเริ่มตั้งแต่แรก เหมาะกับการพัฒนาดั้งเดิม/เชิงโครงสร้าง

- แบบจำลองวนรอบเพิ่มพูน (Iterative and Incremental model) แบ่งงานเป็นรอบงาน (Iteration) ทำส่วนที่ชัดเจนก่อนแล้ววนกลับมาเพิ่มเติม ระบบถูกเพิ่มพูน (Incremental) ให้โตขึ้นเรื่อยๆ จัดการความเสี่ยงสูงตั้งแต่ต้น รองรับการเปลี่ยนความต้องการได้ดี เหมาะกับการพัฒนาเชิงวัตถุ

4.1.2 ความแตกต่างของ SAD และ OOAD

  • ด้านการวิเคราะห์ (Analysis): SAD เน้นความต้องการเชิงโครงสร้าง (DFD, Structured English, Decision Table/Tree, ER Analysis) ส่วน OOAD เน้นวิศวกรรมความต้องการด้วย ยูสเคสโมเดล (Use Case Model) และ อ็อบเจ็กต์โมเดล (Object Model) (หาคลาส/ความสัมพันธ์, Object Interaction, Sequence & Collaboration Diagram, State Machine Diagram)
  • ด้านการออกแบบ (Design): SAD ออกแบบฐานข้อมูล + นอร์มอลไลเซชัน + GUI/ฟอร์มรายงาน; OOAD ออกแบบฐานข้อมูลเชิงกายภาพ + สถาปัตยกรรมระบบ + คลาส (รวม/แยก/ลบคลาส) + คอมโพเนนต์ + GUI
  • ด้านขั้นตอน: SAD ใช้วัฏจักรแบบน้ำตก; OOAD ใช้กระบวนการยูนิไฟด์แบบวนรอบเพิ่มพูน
  • ข้อสรุป (ตารางที่ 4.4): SAD เน้นกระบวนการ (Process) ความเสี่ยงสูง การนำกลับมาใช้ต่ำ สมบูรณ์/แพร่หลาย เหมาะกับโครงการที่ความต้องการนิ่ง; OOAD เน้นวัตถุ (Objects) ความเสี่ยงต่ำ การนำกลับมาใช้สูง กำลังเติบโต เหมาะกับโครงการใหญ่ที่ความต้องการเปลี่ยนแปลง

4.2 กระบวนการยูนิไฟด์ (Unified Process)

  • นิยาม: กระบวนการพัฒนาระบบงานและซอฟต์แวร์ ใช้อธิบายแนวทางสร้าง ติดตั้ง และบำรุงรักษาระบบ แบ่งเป็น 2 มิติ:

1. มิติทางด้านเวลา (Time) หรือ ช่วงระยะเฟส (Phase) — แกนนอน

2. มิติทางด้านกระแสงาน (Workflows) — แกนตั้ง

  • ช่วงระยะเฟส (Lifecycle phases) มี 4 เฟส:

1. ระยะการเตรียมงาน (Inception phase) — เริ่มต้น นิยามขอบเขตโครงการ สร้าง Business case ผลลัพธ์: Vision document, ยูสเคสโมเดลเบื้องต้น, Project glossary, Business model, Prototype เบื้องต้น (ตรงกับ Phase Planning ของ UML)

2. ระยะการจัดทำรายละเอียด (Elaboration phase) — วางแผนโครงการ จัดทำรายละเอียดความต้องการ สร้างสถาปัตยกรรม ผลลัพธ์: ยูสเคสโมเดลที่สมบูรณ์ขึ้น, คำอธิบายสถาปัตยกรรมซอฟต์แวร์ (ตรงกับ Phase Analysis and Design)

3. ระยะการก่อสร้าง (Construction phase) — พัฒนาและทดสอบระบบ ผลลัพธ์: ซอฟต์แวร์ที่ใช้งานได้ + คู่มือ (ตรงกับ Implementation) ใช้ทรัพยากร (resource) มากที่สุด

4. ระยะการถ่ายโอนระบบ (Transition phase) — ส่งมอบ อบรม ติดตั้ง ช่วยเหลือผู้ใช้

  • รอบงาน (Iteration): แต่ละเฟสแบ่งเป็นรอบงาน แต่ละรอบทำทั้งกระแสงานหลักและสนับสนุน ได้ผลลัพธ์เป็นเวอร์ชันที่ดีขึ้นเรื่อยๆ
  • มิติกระแสงาน (Workflows) มี 9 กระแสงาน แบ่งเป็น:

- กระแสงานหลัก (Core Workflows) 1-6: 1) การสร้างแบบจำลอง (Modeling) 2) ความต้องการ (Requirement) 3) การวิเคราะห์และออกแบบ (Analysis and Design) 4) การสร้างระบบงาน (Implementation) 5) การทดสอบ (Testing) 6) การติดตั้งระบบใหม่ (Deployment)

- กระแสงานสนับสนุน (Supporting Workflows): 7) การบริหารจัดการการปรับแต่ง (Configuration Management) 8) การบริหารจัดการโครงการ (Project Management) 9) การจัดเตรียมสภาพแวดล้อม (Environment)

  • สถาปัตยกรรมศูนย์กลาง (Architecture-Centric): สถาปัตยกรรมซอฟต์แวร์อยู่เป็นศูนย์กลาง กำหนดในรูปส่วนประกอบ (Components) และส่วนเชื่อมต่อ (Connectors) แบ่งเป็น 5 มุมมอง: 1) มุมมองตรรกะ (Logical view) 2) มุมมองอิมพลีเมนเทชัน (Implementation view) 3) มุมมองกระบวนการ (Process view) 4) มุมมองถ่ายทอดงาน (Deployment view) 5) มุมมองกรณีใช้งาน (Use-case view)
  • โครงสร้างคงที่ (Static Structure) ของกระบวนการยูนิไฟด์ ประกอบด้วย 4 ส่วน (ตอบคำถาม who/how/what/when):

1. ผู้ปฏิบัติงาน (Worker) = who

2. กิจกรรม (Activity) = how

3. ชิ้นงาน (Artifact) = what (เช่น Model, Document, Source code, Executables)

4. กระแสงาน (Workflows) = when

หน่วยที่ 5: ยูเอ็มแอล (UML)

ภาพรวม

หน่วยนี้ปูพื้นเรื่อง "การจำลองแบบระบบ" (Modeling) ว่าคือการสร้างแบบจำลองที่แสดงสาระสำคัญ (abstraction) ของระบบ เพื่อสื่อสารและทำความเข้าใจระบบในแง่มุมต่างๆ จากนั้นเข้าสู่หัวใจของหน่วย คือ ยูเอ็มแอล (UML – Unified Modeling Language) ซึ่งเป็นมาตรฐานการจำลองแบบระบบซอฟต์แวร์ด้วยวิธีการเชิงวัตถุ ดูแลโดยองค์กร OMG หน่วยนี้อธิบายแผนภาพ UML ทั้ง 9 แผนภาพ ทั้งกลุ่มที่อธิบายการทำงานของระบบและกลุ่มที่อธิบายสถาปัตยกรรมของระบบ พร้อมการจัดประเภทแผนภาพ 2 แบบ

แนวคิดสำคัญ

การจำลองแบบ (Modeling) = การกำหนดสาระสำคัญของระบบ (system abstraction) โดยให้ความสำคัญเฉพาะคุณลักษณะที่จำเป็น ละเว้นรายละเอียดที่ไม่เกี่ยวข้อง แบบจำลองมี 2 ลักษณะ คือ แบบใช้ข้อความ (text model) ซึ่งตีความได้ต่างกัน กับ แบบใช้ภาพ/สัญลักษณ์ (visual model) เช่น แผนภาพ ซึ่งเข้าใจตรงกันและไม่กำกวม UML จึงเป็น visual model มาตรฐาน ที่รวมแนวคิดจากหลายระเบียบวิธีให้เป็นสัญกรณ์ (notation) เดียว ทำให้ผู้เกี่ยวข้องทุกฝ่าย (stakeholder) เข้าใจระบบตรงกัน

เนื้อหาหลัก

5.1 การจำลองแบบระบบ

  • แนวคิดการจำลองแบบมี 2 รูปแบบหลัก: 1) วิเคราะห์/ออกแบบเชิงโครงสร้างตามฟังก์ชันการทำงาน (functional decomposition) 2) วิเคราะห์/ออกแบบเชิงวัตถุตามวัตถุและคลาส (object-oriented decomposition)
  • แบบจำลองอธิบายระบบซอฟต์แวร์ 3 ชนิด: 1) แบบจำลองวัตถุ (Object model) – ระบบมีวัตถุอะไร สัมพันธ์กันอย่างไร 2) แบบจำลองฟังก์ชัน (Functional model) – หน้าที่/ฟังก์ชันการทำงานคืออะไร 3) แบบจำลองพลวัต (Dynamic model) – ระบบตอบสนองต่อเหตุการณ์/สถานะอย่างไร
  • การแยกส่วนเชิงฟังก์ชัน ได้แบบจำลอง 2 ประเภท: แบบจำลองกระบวนการ (Process Model เช่น DFD) และแบบจำลองเชิงข้อมูล (Data-oriented Model เช่น ERD ของ Peter Chen ปี 1976)
  • คุณสมบัติสำคัญเชิงวัตถุ 4 ข้อ: 1) เอกลักษณ์ (Identity) 2) การห่อหุ้ม (Encapsulation / information hiding) 3) การรับทอด (Inheritance) 4) การพ้องรูป/พอลิมอร์ฟิซึม (Polymorphism)
  • วัตถุ (object) = เอนทิตีรวมกับพฤติกรรม วัตถุห่อหุ้ม (encapsulate) ทั้งข้อมูลและฟังก์ชัน พฤติกรรมเรียกรวมว่า เมธอด (methods); วัตถุแบ่งเป็น 2 ประเภท: วัตถุคงอยู่ชั่วครู่ (Transient) และวัตถุคงอยู่ตลอดเวลา (Persistent)

5.1.3 แผนภาพระบบ (เปรียบเทียบโครงสร้าง vs วัตถุ)

  • แผนภาพเชิงโครงสร้างที่นิยม: 1) แผนภาพบริบท (Context diagram) 2) แผนภาพการไหลของข้อมูล (DFD) 3) แผนภาพความสัมพันธ์ระหว่างเอนทิตี (ERD)
  • การเทียบเคียง (ข้อสอบชอบถาม): แผนภาพบริบท ≈ แผนภาพยูสเคส | DFD ระดับ 1 ≈ แผนภาพลำดับ | ERD ≈ แผนภาพคลาส

5.2 แผนภาพยูเอ็มแอล

  • นิยาม UML: มาตรฐานคุณภาพตาม ISO สำหรับจำลองแบบระบบซอฟต์แวร์ เป็นมาตรฐานที่ไม่มีการคุ้มครองความเป็นเจ้าของ (non-proprietary) ดูแลโดย OMG (Object Management Group)
  • ที่มาจาก 3 ระเบียบวิธี: 1) OMT (Object Modeling Technique) ของ James Rumbaugh 2) Booch method ของ Grady Booch 3) OOSE (Object-oriented Software Engineering) ของ Ivar Jacobson
  • แผนภาพ UML 9 แผนภาพ:

1. แผนภาพยูสเคส (Use case diagram) – แบบจำลองฟังก์ชัน แสดงพฤติกรรมตามฟังก์ชันที่ผู้ใช้มองเห็น (แอ็คเตอร์ + ยูสเคส + ขอบเขตระบบ)

2. แผนภาพคลาส (Class diagram) – แบบจำลองวัตถุ แสดงคลาสและความสัมพันธ์ เป็นโครงสร้างสถิต (static structure) มี แอตทริบิวต์ (attribute) และ เมธอด (method)

3. แผนภาพการสื่อสาร (Communication diagram) – การทำงานร่วมกันของวัตถุ/แอ็คเตอร์ เน้นวิธีการที่วัตถุติดต่อกัน

4. แผนภาพลำดับ (Sequence diagram) – ลำดับการทำงาน/ปฏิสัมพันธ์ของวัตถุตามเวลา ลงรายละเอียดภายในแต่ละยูสเคส

5. แผนภาพกิจกรรม (Activity diagram) – ฟังก์ชัน/เมธอดการทำงาน นิยมเขียนแบบ สวิมเลน (swim lane)

6. แผนภาพสถานะ (State diagram) – สถานะ (state) ของวัตถุที่เปลี่ยนแปลงตามเหตุการณ์

7. แผนภาพแพ็กเกจ (Package diagram) – จัดกลุ่มคลาสที่ทำงานร่วมกันเป็นแพ็กเกจ

8. แผนภาพคอมโพเนนต์ (Component diagram) – โครงสร้างเชิงกายภาพ ประกอบด้วยคอมโพเนนต์ใดบ้าง

9. แผนภาพดีพลอยเมนต์ (Deployment diagram) – สถาปัตยกรรมเชิงกายภาพ (physical architecture) ความสัมพันธ์ฮาร์ดแวร์กับซอฟต์แวร์

  • การจัดประเภทแผนภาพ UML 2 แบบ:

- แบบที่ 1 ตามลักษณะแผนภาพ: (ก) แผนภาพเชิงโครงสร้าง/เชิงสถิต (Structural/Static) = คลาส, วัตถุ, แพ็กเกจ, คอมโพเนนต์, ดีพลอยเมนต์ (ข) แผนภาพเชิงพฤติกรรม/เชิงพลวัต (Behavioral/Dynamic) = ยูสเคส, ลำดับ, กิจกรรม, สถานะ, การสื่อสาร

- แบบที่ 2 ตามการจำลองแบบ: (ก) แผนภาพการทำงานของระบบ = ยูสเคส, การสื่อสาร, คลาส(วัตถุ), ลำดับ, สถานะ, กิจกรรม (ข) แผนภาพสถาปัตยกรรมของระบบ = แพ็กเกจ, คอมโพเนนต์, ดีพลอยเมนต์

  • องค์ประกอบแผนภาพคลาสและความสัมพันธ์ (สำคัญมาก):

- ความเกี่ยวข้องกัน (Association) – เส้นตรงเชื่อมคลาส มีการระบุจำนวน (multiplicity เช่น 1, 1..*, *)

- การรวมกลุ่ม (Aggregation) – ความสัมพันธ์แบบ "part-of" ใช้สัญลักษณ์ สี่เหลี่ยมข้าวหลามตัดโปร่ง (เช่น Computer ↔ CPU, Memory)

- การเป็นองค์ประกอบ (Composition) – การรวมกลุ่มที่เหนียวแน่น เมื่อคลาสหลักถูกทำลาย องค์ประกอบถูกทำลายตามด้วย ใช้ สี่เหลี่ยมข้าวหลามตัดทึบ (เช่น SimpleWatch ↔ Display, Battery)

- การรับทอด (Inheritance) – ความสัมพันธ์ "is-kind-of" คลาสลูกรับทอดแอตทริบิวต์/การดำเนินการจากคลาสพ่อแม่ ใช้ สามเหลี่ยมโปร่ง

  • ความสัมพันธ์ยูสเคส 2 ลักษณะ (เริ่มในหน่วยนี้ ขยายในหน่วย 6):

- แบบขยาย (extend relationship) – ยูสเคสที่เรียกใช้ในบางกรณี/หน้าที่กรณีพิเศษ (exceptional) สัญลักษณ์ <<extend>>

- แบบรวม (include relationship) – ยูสเคสที่ทำงานร่วมกับยูสเคสหลักเสมอ แยกออกเพื่อนำกลับมาใช้ซ้ำ (reuse) สัญลักษณ์ <<include>>

ประเด็นที่ควรเน้นเพื่อสอบ

  • UML มี 9 แผนภาพ — ต้องท่องได้ทั้ง 9 และบอกได้ว่าแต่ละแผนภาพอธิบายอะไร
  • การจัดประเภท 2 แบบ: เชิงโครงสร้าง vs เชิงพฤติกรรม / แผนภาพการทำงาน vs แผนภาพสถาปัตยกรรม — รู้ว่าแผนภาพใดอยู่กลุ่มใด (เช่น คลาส = โครงสร้าง, ยูสเคส = พฤติกรรม)
  • จับคู่แผนภาพโครงสร้างกับวัตถุ: Context ≈ Use case, DFD ≈ Sequence, ERD ≈ Class (ออกสอบบ่อย)
  • แยกสัญลักษณ์ความสัมพันธ์ในแผนภาพคลาส: Aggregation (ข้าวหลามตัดโปร่ง) vs Composition (ข้าวหลามตัดทึบ) vs Inheritance (สามเหลี่ยมโปร่ง) vs Association (เส้นตรง + multiplicity)
  • แยก <<include>> vs <<extend>>: include = เรียกใช้เสมอ/เพื่อ reuse ; extend = เรียกใช้บางกรณี/กรณีพิเศษ
  • UML ดูแลโดย OMG เป็นมาตรฐาน ISO และมาจาก 3 ระเบียบวิธี (OMT-Rumbaugh, Booch, OOSE-Jacobson)
  • 3 แบบจำลองอธิบายระบบ: Object / Functional / Dynamic model จับคู่กับแผนภาพได้ (เช่น ยูสเคส=ฟังก์ชัน, คลาส=วัตถุ, ลำดับ/สถานะ/กิจกรรม=พลวัต)
  • คุณสมบัติเชิงวัตถุ 4 ข้อ: Identity, Encapsulation, Inheritance, Polymorphism
  • จำได้ว่า เมธอด (method) = ฟังก์ชันการทำงานของคลาส และ แอตทริบิวต์ (attribute) = คุณลักษณะ/ข้อมูลของคลาส

หน่วยที่ 6: ยูสเคส (Use Case)

ภาพรวม

หน่วยนี้เจาะลึกเครื่องมือสำคัญในการเก็บความต้องการของผู้ใช้แบบเชิงวัตถุ คือ ยูสเคส (Use case) ครอบคลุมความหมาย คุณสมบัติ องค์ประกอบของ แผนภาพยูสเคส (Use case diagram) วิธีเขียน คำอธิบายยูสเคส (Use case description) และที่สำคัญที่สุดคือ ความสัมพันธ์ของยูสเคส ทั้ง 3 กลุ่ม (ระหว่างแอ็คเตอร์, ระหว่างยูสเคส, ระหว่างแอ็คเตอร์กับยูสเคส) ปิดท้ายด้วยตัวอย่างการวิเคราะห์และเขียนแผนภาพยูสเคสจริง (ศูนย์บริการรถยนต์ และการขายสินค้าออนไลน์)

แนวคิดสำคัญ

แบบจำลองยูสเคส (Use case model) ถูกพัฒนาโดย Ivar Jacobson ปี ค.ศ. 1986 เป็นแบบจำลองที่อธิบายการทำงานของผู้ใช้ประเภทต่างๆ ที่เข้ามามีปฏิสัมพันธ์กับระบบ ใช้แสดงขอบเขตของระบบและทำให้ผู้ใช้มีส่วนร่วมในการออกแบบตั้งแต่ช่วงแรก หลักการเขียนยูสเคสคือ "ต้องทำให้ง่ายที่สุด" เพราะผู้ใช้มีความรู้คอมพิวเตอร์น้อย แบบจำลองยูสเคสมี 2 ส่วนหลักเสมอ คือ 1) แผนภาพยูสเคส และ 2) คำอธิบายยูสเคส

6.1.1 แนวคิดและคำศัพท์เกี่ยวกับยูสเคส

  • ยูสเคส (Use case) = กิจกรรม/หน้าที่/ความต้องการของระบบที่แสดงกระบวนการทางธุรกิจ เป็นกิจกรรมที่รับรู้/เข้าใจได้ชัดเจน สัญลักษณ์คือ วงรี ตั้งชื่อด้วย คำกริยา (เช่น ออกใบเสร็จรับเงิน/Receipt, ขับรถ/drive car)
  • แอ็คเตอร์ (actor) = ผู้ใดหรือสิ่งใดที่มีบทบาทเกี่ยวข้องกับกระบวนการ (เช่น นักศึกษา ลูกค้า พนักงาน หรือฐานข้อมูล/ธนาคาร) อยู่นอกระบบ สัญลักษณ์คือ รูปกระดูกคน ตั้งชื่อด้วย คำนาม และพิจารณาที่ บทบาท (role) ไม่ใช่ตัวบุคคล
  • คุณสมบัติของยูสเคส 3 ข้อ: 1) ยูสเคสถูกกระทำด้วยแอ็คเตอร์ 2) ยูสเคสรับข้อมูลจากแอ็คเตอร์และส่งข้อมูลให้แอ็คเตอร์ 3) ยูสเคสรวบรวมคุณลักษณะความต้องการภายในระบบ (สรุปความต้องการทั้งหมด)
  • แนวทางสร้างยูสเคส 4 ขั้น: 1) จำแนกแอ็คเตอร์ (ใคร/สิ่งใดนำข้อมูลเข้า แสดงผล มีปฏิสัมพันธ์) 2) จำแนกยูสเคส 3) สร้างแผนภาพยูสเคส 4) จัดทำเอกสารคำอธิบายยูสเคส

6.1.2 แผนภาพยูสเคสและคำอธิบายยูสเคส

  • เงื่อนไขสำคัญ: Precondition (เงื่อนไขก่อนเข้าสู่ระบบ) และ Postcondition (เงื่อนไขหลังออกจากระบบ)
  • องค์ประกอบของแผนภาพยูสเคส 4 อย่าง: 1) แอ็คเตอร์ (actor) 2) ยูสเคส (use case) 3) ขอบเขตระบบ (system boundary) – กรอบสี่เหลี่ยม แอ็คเตอร์อยู่นอก ยูสเคสอยู่ใน 4) ความสัมพันธ์ (relationship) – เส้นเชื่อม
  • ขั้นตอนสร้างแผนภาพยูสเคส 5 ขั้น: 1) เริ่มการทำงาน (starting) ด้วยแอ็คเตอร์ 2) เกิด Precondition 3) ทำงาน (working) ด้วยยูสเคส 4) เกิด Postcondition 5) จบการทำงาน (ending) ด้วยแอ็คเตอร์ที่รับผลลัพธ์
  • โครงสร้างคำอธิบายยูสเคส (Use case description) ส่วนสำคัญ:

- USE CASE / ID (ชื่อและรหัสยูสเคส)

- Brief description (คำอธิบายโดยย่อ)

- Primary actors (แอ็คเตอร์หลัก – ตัวเริ่มต้นยูสเคส) / Secondary actors (แอ็คเตอร์รอง)

- Preconditions (เงื่อนไขก่อนหน้า)

- Main flow (การทำงานหลัก – ตอบ Who/What/When/Where)

- Postconditions (เงื่อนไขภายหลัง)

- Alternative Flows (ทางเลือก – เรียกใช้เมื่อ main flow เกิด error หรือ interrupt)

  • หมายเหตุ: ในหน่วย 5 ระบุการอธิบายยูสเคสมี 6 ส่วนหลัก (Unique name, Participating actors, Entry/Precondition, Flow of events/Main flow, Exit/Postcondition, Special requirements)

6.1.3 ความสัมพันธ์ของยูสเคส (หัวใจของหน่วย)

แบ่งเป็น 3 กลุ่ม:

1. ความสัมพันธ์ระหว่างแอ็คเตอร์กับแอ็คเตอร์

  • แบบทั่วไป (Generalization) = การถ่ายทอด/รับทอด (inheritance) แอ็คเตอร์ลูก (child actor) รับคุณสมบัติจากแอ็คเตอร์พ่อแม่ (parent actor) โดยพ่อแม่เกิดก่อน เช่น นักศึกษา → นักศึกษาภาคปกติ/ภาคค่ำ/ภาคอินเตอร์; พนักงาน → พนักงานประจำ/พนักงานรายวัน
  • แบบโครงสร้าง/เชื่อมโยง (Association) = แอ็คเตอร์กระทำร่วมกันหรือส่งต่อไปอีกแอ็คเตอร์หนึ่ง เช่น ลูกค้าโรงแรมแจ้งผ่านพนักงานเพื่อจองห้อง

2. ความสัมพันธ์ระหว่างยูสเคสกับยูสเคส

  • แบบทั่วไป (Generalization) = ยูสเคสลูก (child use case) รับทอดคุณสมบัติจากยูสเคสพ่อแม่ (parent use case) เช่น ขับรถ → หยุดรถ/กลับรถ; Send → Forward/Reply (และ Reply → Reply all)
  • แบบรวม (Include relationship) สัญลักษณ์ <<include>> = นำขั้นตอนที่ซ้ำซ้อนมาสร้างเป็นยูสเคสแยก (sub use case) เพื่อให้ยูสเคสอื่นเรียกใช้ได้ ถูกเรียกใช้ทุกครั้งที่ Base Use case ทำงาน เช่น เข้าสู่ระบบ (Login) ถูก include โดย ลงทะเบียนเรียน/สมัครเรียน/ตัดเกรด; หัวลูกศรชี้ไปยังยูสเคสที่ถูกรวม
  • แบบขยาย (Extend relationship) สัญลักษณ์ <<extend>> = นำยูสเคสเดิมมาเพิ่มการทำงานเฉพาะเจาะจง มีสิ่งกระตุ้นจึงทำงานต่าง ถูกเรียกใช้เฉพาะบางกรณี เช่น กลับรถ ← กลับรถทางซ้าย/ทางขวา; หัวลูกศรชี้ไปยังยูสเคสพื้นฐาน (base use case)

3. ความสัมพันธ์ระหว่างแอ็คเตอร์กับยูสเคส (Association Relationship)

  • แอ็คเตอร์ให้ข้อมูลเข้าระบบเพื่อให้ยูสเคสทำงาน และแอ็คเตอร์รับผลลัพธ์ เป็นเส้นเชื่อมระหว่างแอ็คเตอร์กับยูสเคส

> ความแตกต่างที่ออกสอบบ่อยที่สุด: แบบขยาย (extend) = ยูสเคสที่ถูกเรียกใช้เฉพาะบางกรณีเท่านั้น แต่แบบรวม (include) = ถูกเรียกใช้ทุกครั้งที่ Base Use case ดำเนินกิจกรรม

6.2 ตัวอย่างการเขียนแผนภาพยูสเคส

  • ตัวอย่างศูนย์บริการรถยนต์: แอ็คเตอร์ = ลูกค้า, พนักงานบริการ, พนักงานบัญชี, ช่างซ่อม (สามอย่างหลังเป็น child actor ของพนักงาน); ยูสเคส = บริการส่วนหน้า, ตรวจสอบอาการรถ, ซ่อมรถ (extend/child ของตรวจสอบอาการรถ), รายงานการซ่อม, การชำระเงิน — แต่ละยูสเคสเขียนคำอธิบาย (ID, Brief description, Primary/Secondary actors, Preconditions, Main flow, Postconditions, Alternative Flows)
  • ตัวอย่างการขายสินค้า/จำหน่ายตั๋วออนไลน์: ยูสเคส ค้นหาตั๋ว → กรอกข้อมูล → จองตั๋ว (include กรอกข้อมูล) → รับตั๋ว/ชำระเงิน (include จองตั๋ว); แสดงให้เห็นการใช้ <<include>> ในระบบจริง

ประเด็นที่ควรเน้นเพื่อสอบ

  • แบบจำลองยูสเคสพัฒนาโดย Ivar Jacobson (ค.ศ. 1986) — ออกสอบเรื่องบุคคล/ปี
  • องค์ประกอบแผนภาพยูสเคส 4 อย่าง: actor, use case, system boundary, relationship
  • สัญลักษณ์ที่ต้องจำ: ยูสเคส = วงรี, แอ็คเตอร์ = รูปกระดูกคน, ขอบเขต = กรอบสี่เหลี่ยม
  • การตั้งชื่อ: ยูสเคสใช้คำกริยา / แอ็คเตอร์ใช้คำนาม
  • แยก include vs extend (สำคัญสุด): <<include>> ถูกเรียกทุกครั้งเพื่อ reuse (ลูกศรชี้ยูสเคสที่ถูกรวม) ; <<extend>> ถูกเรียกบางกรณี/กรณีพิเศษ (ลูกศรชี้ base use case)
  • ความสัมพันธ์ 3 กลุ่ม: actor↔actor, use case↔use case, actor↔use case — และรู้ว่าแต่ละกลุ่มมีแบบใดบ้าง
  • Generalization = การรับทอด ทั้งระหว่างแอ็คเตอร์ และระหว่างยูสเคส (parent เกิดก่อน child)
  • รูปแบบ Use case description: จำหัวข้อ ID, Brief description, Primary/Secondary actors, Preconditions, Main flow, Postconditions, Alternative Flows
  • Precondition vs Postcondition: เงื่อนไขก่อน vs หลังการทำงานของยูสเคส
  • 5 ขั้นตอนสร้างแผนภาพยูสเคส: starting → precondition → working → postcondition → ending
  • ฝึกวิเคราะห์โจทย์จริง (ATM ถอนเงิน, ศูนย์บริการรถยนต์, ขายตั๋วออนไลน์) ให้เขียนทั้งแผนภาพและคำอธิบายได้

หน่วยที่ 7: แผนภาพคลาส (Class Diagram)

ภาพรวม

แผนภาพคลาส (Class Diagram) เป็นแผนภาพหนึ่งในยูเอ็มแอล (UML) ที่ใช้แสดง "โครงสร้างเชิงสถิตย์" ของระบบ คือแสดงองค์ประกอบของคลาส กลุ่มของคลาส และความสัมพันธ์ระหว่างคลาส หน่วยนี้แบ่งเป็น 2 ตอน คือ ตอนที่ 7.1 ความรู้เบื้องต้น (องค์ประกอบ สัญลักษณ์ และความสัมพันธ์) และตอนที่ 7.2 การสร้างแผนภาพคลาสจริงตั้งแต่การหาคลาสจนถึงการลากความสัมพันธ์ เป้าหมายคือให้นักศึกษาอธิบายองค์ประกอบ/สัญลักษณ์ และสร้างแผนภาพคลาสได้

แนวคิดสำคัญ

  • คลาส (Class) คือการจัดกลุ่มของวัตถุที่มีคุณลักษณะ (attribute) และพฤติกรรม (behavior) คล้ายกันเข้าด้วยกัน คลาสเป็น "นามธรรม (Abstract)" จึงทำกิจกรรมเองไม่ได้ แต่ใช้เป็นแม่พิมพ์ (template) เพื่อสร้างวัตถุตัวแทน (Instantiation) ที่เรียกว่า Object หรือ Instance ในโลกจริง
  • ความสัมพันธ์ในแผนภาพคลาสเป็น ความสัมพันธ์เชิงสถิตย์ (Static Relationship) คือความสัมพันธ์ที่มีอยู่เป็นปกติและไม่เปลี่ยนแปลง (เช่น เจ้าของบัญชีเป็นเจ้าของบัญชีเงินฝาก) ต่างจาก ความสัมพันธ์เชิงพลวัติ (Dynamic Relationship) ที่เกิดจากกิจกรรม (เช่น ฝากเงิน ถอนเงิน) ซึ่งจะไม่ปรากฏในแผนภาพคลาส
  • ในการสร้างแผนภาพคลาสช่วงวิเคราะห์ คลาสต้องอยู่ในโลกความจริง อาจจับต้องได้ (Tangible) หรือจับต้องไม่ได้ (Intangible) และยังไม่ต้องคำนึงคลาสฝั่งคอมพิวเตอร์ เช่น หน้าจอ หรือส่วนต่อประสานผู้ใช้

เนื้อหาหลัก

7.1.1 ความหมาย องค์ประกอบ และสัญลักษณ์ของแผนภาพคลาส

คลาสแบ่งองค์ประกอบเป็น 3 ส่วน ได้แก่ 1) ชื่อคลาส (Class Name) 2) แอตทริบิวต์ (Attributes) 3) โอเปอเรชัน (Operations/Method) เขียนในกล่องสี่เหลี่ยม 3 ช่อง (ชื่ออยู่บนสุด แอตทริบิวต์อยู่กลาง โอเปอเรชันอยู่ล่าง)

หลักการตั้งชื่อ:

  • ชื่อคลาส: ใช้คำนาม ขึ้นต้นด้วยตัวพิมพ์ใหญ่ ถ้ามี 2 คำให้ขึ้นต้นใหญ่ทั้งคู่ ไม่มีช่องว่างคั่น (เช่น Human, GoldMember)
  • ชื่อแอตทริบิวต์: ใช้คำนาม นิยมตัวพิมพ์เล็ก ถ้ามี 2 คำให้คำที่ 2 ขึ้นต้นใหญ่ (เช่น age, studentName)
  • ชื่อโอเปอเรชัน: ใช้คำกริยา นิยมตัวพิมพ์เล็ก ตามด้วย () เสมอ แม้ไม่มีพารามิเตอร์ (เช่น checkPasswd()); set นำหน้าเพื่อกำหนดค่า, get นำหน้าเพื่ออ่านค่า

ระดับการมองเห็นและการเข้าถึง (Visibility and Accessibility) มี 3 แบบ:

  • + สาธารณะ (public) — คลาสอื่นมองเห็นและใช้งานได้ (โอเปอเรชันมักเป็น public เสมอ)
  • # ป้องกัน (protected) — คลาสอื่นมองเห็นได้แต่ใช้งานไม่ได้
  • - ส่วนตัว (private) — คลาสอื่นมองเห็นและใช้งานไม่ได้ (แอตทริบิวต์นิยมเป็น private)

รูปแบบการเขียนแอตทริบิวต์: Visibility attributeName: Attribute Type โดยประเภทอยู่หลังโคลอน (:) แบ่งเป็น Primitive Data Type (PDT เช่น int, string) และ Abstract Data Type (ADT เช่น วัตถุเมาส์ จอภาพ)

รูปแบบการเขียนโอเปอเรชัน: Visibility OperationName(Parameter List: Type): Return Type โดย 1 โอเปอเรชันมี Return Type ได้มากสุด 1 ตัวหรือไม่มีก็ได้ (ตัวอย่าง + getAge(y:Integer):Integer)

7.1.2 ความสัมพันธ์ระหว่างคลาส (ประเด็นเน้นมาก)

พิจารณา 3 ด้าน ได้แก่ ความสัมพันธ์เชิงนามธรรม, ดีกรีของความสัมพันธ์ และจำนวนวัตถุที่สัมพันธ์กัน

(ก) ความสัมพันธ์เชิงนามธรรม (Abstract Class Relationship) — 4 รูปแบบหลัก:

  1. การเกี่ยวข้องกัน (Association) — คลาสอยู่ระดับเดียวกัน สำคัญเท่ากัน ไม่มีคลาสใดเป็นองค์ประกอบหรือสืบทอดกัน เชื่อมด้วยเส้นตรง มีชื่อความสัมพันธ์ (Association Name) บนเส้น และหัวลูกศรบอกทิศทาง (เช่น Customer กับ Order "จัดทำโดย/Issued by")
  2. การเป็นส่วนหนึ่ง (Aggregation) — ความสัมพันธ์ต่างระดับแบบ "is part of" คลาสย่อยคือ Part Class คลาสหลักคือ Whole Class จุดสำคัญคือ เมื่อลบ Whole Class แล้ว Part Class ยังอยู่ได้ สัญลักษณ์คือ สี่เหลี่ยมข้าวหลามตัดโปร่ง (โล่ง) ที่ปลายฝั่งคลาสหลัก (เช่น Car ประกอบด้วย Engine, Seat, Wheel)
  3. การเป็นองค์ประกอบ (Composition) — คล้าย Aggregation แต่ เข้มงวดกว่า คือ เมื่อลบ Whole Class แล้ว Part Class อยู่ไม่ได้ (หายไปด้วย) สัญลักษณ์คือ สี่เหลี่ยมข้าวหลามตัดสีทึบ (เต็ม) (เช่น Person ประกอบด้วย Hand, Leg)
  4. การเป็นปกติทั่วไป (Generalization) — การสืบทอดคุณสมบัติแบบ "is a kind of" อยู่ในรูปลำดับชั้น (class Hierarchy) คลาสบนคือ superclass คลาสล่างคือ subclass โดย subclass รับ attribute/operation จาก superclass และเพิ่ม/แก้ไขได้ สัญลักษณ์คือ เส้นที่มีปลายลูกศรสีขาว (โปร่ง) ชี้จาก subclass ไปยัง superclass (เช่น Employee → FullTime, PartTime)

(ข) ดีกรีของความสัมพันธ์ (Degree of Relationship) — จำนวนคลาสที่ร่วมสัมพันธ์ 3 ระดับ:

  • Unary (1 คลาส) — สัมพันธ์กับตัวเอง (เช่น Person "married to" Person)
  • Binary (2 คลาส) — พบมากที่สุด (เช่น Employee "Assigned to" Position)
  • Ternary (3 คลาส) — สัมพันธ์ 3 คลาส มักเกิดคลาสตัวกลางมาเชื่อม (เช่น Part, Vendor, Warehouse ผ่านคลาส Ship)

7.2.1 การสร้างแผนภาพคลาสเบื้องต้น (4 ขั้นตอน)

  1. พิจารณาหาคลาสจากขอบเขตของปัญหา (Problem Domain) และยูสเคส — วิเคราะห์โดย 1) หาคำนาม 2) แยกประเภทคำนาม (อาจเป็นวัตถุ/คลาส/แอตทริบิวต์) 3) หาคำกริยา (จะกลายเป็นโอเปอเรชัน); กรณีมียูสเคสให้ใช้ส่วน Main Flow และ Exception Flow มาวิเคราะห์
  2. ใช้นามธรรมแบบการจำแนก (Classification Abstraction) — หาว่ามีวัตถุใดอยู่ในขอบเขตและควรจัดในคลาสใด ขั้นนี้ "สำคัญที่สุด" เพราะให้คลาสพื้นฐาน ถ้าผิดจะกระทบการสร้างความสัมพันธ์ทั้งหมด
  3. กำหนดคุณสมบัติให้แอตทริบิวต์ 3 ด้าน — (3.1) คีย์หลัก (Primary Key) ใช้สัญลักษณ์ <<PK>> ค่าต้องไม่ซ้ำ (3.2) ประเภทแอตทริบิวต์ (String, Integer, Date, Currency, Double, Boolean) (3.3) ระดับการมองเห็น (แอตทริบิวต์นิยมกำหนดเป็น private)
  4. เติมโอเปอเรชันพื้นฐาน — นำคำกริยาที่ได้มาพิจารณาว่าอยู่คลาสใด (เช่น ดึงข้อมูลนักศึกษา(), คำนวณเงินค่าลงทะเบียน())

7.2.2 การสร้างความสัมพันธ์ระหว่างคลาส (ใช้ตัวอย่างระบบลงทะเบียน)

  • Aggregation: คลาสวิชากับคลาสการลงทะเบียน — 1 การลงทะเบียนมี 1..7 วิชา (เป็นส่วนหนึ่งของ)
  • Association: นักศึกษากับการลงทะเบียน เป็น one-to-many (นักศึกษา 1 คน 1..* การลงทะเบียน; การลงทะเบียน 1 ครั้งมาจากนักศึกษา 1 คน)
  • Composition: นักศึกษากับประวัติการเรียน — ถ้านักศึกษาพ้นสภาพ ประวัติการเรียนหายไปด้วย จึงเข้มงวด (one-to-many)
  • ระบบนี้ไม่มีความสัมพันธ์แบบ Generalization — ย้ำว่าไม่จำเป็นต้องมีครบทุกความสัมพันธ์

> หมายเหตุสำคัญ: แผนภาพคลาสจากหน่วยที่ 7 ยัง "ไม่สมบูรณ์" จะถูกปรับปรุงอีกครั้งเมื่อเขียนแผนภาพปฏิสัมพันธ์ (หน่วย 10) แผนภาพสถานะและกิจกรรม (หน่วย 11)

  • แยก Aggregation vs Composition ให้ชัด: ทั้งคู่เป็น "is part of" แต่ Aggregation = ลบคลาสหลักแล้วส่วนประกอบยังอยู่ (ข้าวหลามตัดโปร่ง) ส่วน Composition = เข้มงวดกว่า ลบคลาสหลักแล้วส่วนประกอบหายไปด้วย (ข้าวหลามตัดทึบ)
  • จำสัญลักษณ์ความสัมพันธ์ทั้ง 5/4 แบบ: Association (เส้นตรง+ชื่อ), Aggregation (ข้าวหลามตัดโปร่ง), Composition (ข้าวหลามตัดทึบ), Generalization (ลูกศรปลายสีขาว/โปร่งชี้จาก subclass ไป superclass)
  • อ่านค่า Multiplicity ให้ถูก: ตัวเลขอยู่ปลายคลาส "ฝั่งตรงข้าม"; จำความหมาย 1, 0..1, 0..*, 1..*, ช่วง 2..4; เขียนโจทย์เป็นสัญลักษณ์ได้ (เช่น นักศึกษาลง 1..6 วิชา, วิชามีนักศึกษา 0..* คน)
  • ทิศทางการอ่าน Multiplicity: มีลูกศรทึบ = อ่านซ้ายไปขวา; ไม่มีลูกศร = อ่านฝั่งไหนก่อนก็ได้
  • องค์ประกอบ 3 ส่วนของคลาส และความหมายของ + - # ออกสอบแน่นอน รวมถึงอ่านความหมายโอเปอเรชันแบบ + getAge(y:Integer):Integer
  • 4 ขั้นตอนการสร้างแผนภาพคลาสเบื้องต้น และจำว่า "ใช้นามธรรมแบบการจำแนก" เป็นขั้นที่สำคัญที่สุด
  • เทคนิควิเคราะห์หาคลาส: คำนาม → วัตถุ/คลาส/แอตทริบิวต์, คำกริยา → โอเปอเรชัน
  • ดีกรีความสัมพันธ์ 3 ระดับ (Unary/Binary/Ternary) และ Binary พบบ่อยที่สุด; Ternary มักสร้างคลาสตัวกลาง

  • ในหน่วยนี้ "การวิเคราะห์ระบบ" กับ "การวิเคราะห์ความต้องการของระบบ" มีความหมายเหมือนกัน คือการขยาย/แตกความต้องการของผู้มีส่วนได้ส่วนเสียให้ชัดเจนว่าระบบใหม่ต้องมีความสามารถและคุณลักษณะอะไรบ้าง
  • ผู้มีส่วนได้ส่วนเสีย (Stakeholder) หมายถึงบุคคล กลุ่มบุคคล องค์กร ฮาร์ดแวร์/ซอฟต์แวร์นอกระบบ หรือสิ่งใดที่ได้ประโยชน์ทางตรง/อ้อมจากระบบ ส่วน ผู้ใช้ระบบ (User) คือสิ่งที่อยู่นอกระบบและมีปฏิสัมพันธ์กับระบบ; ผู้ใช้ระบบเป็น "สับเซ็ต" ของผู้มีส่วนได้ส่วนเสีย (เช่น สปอนเซอร์ที่ขอให้ใส่โลโก้เป็นผู้มีส่วนได้ส่วนเสียแต่ไม่ใช่ผู้ใช้)

8.1.1 แนวคิดการเก็บรวบรวมความต้องการ

ขั้นตอนก่อนเก็บความต้องการ (การร้องขอพัฒนาระบบ): 1) ผู้ต้องการระบบส่งข้อมูลการร้องขอ (ระบุปัญหา ความจำเป็น ความสามารถที่ต้องการ งบประมาณ ประโยชน์) ไปยังผู้บริหาร 2) ผู้บริหารพิจารณา หากอนุมัติส่งต่อหัวหน้าโครงการและทีม (รวมนักวิเคราะห์ระบบ) 3) นักวิเคราะห์ระบบนำข้อมูลมาออกแบบการเก็บรวบรวมความต้องการ

ความต้องการ 3 ประเภท (ออกสอบบ่อย):

  • ความต้องการเชิงธุรกิจ (Business Requirement) — เป้าหมายทางธุรกิจที่จะได้เมื่อใช้ระบบใหม่ เช่น เพิ่มส่วนแบ่งตลาด ลดเวลาทำงาน ลดต้นทุน เป็นจุดอ้างอิงว่าระบบตอบโจทย์ธุรกิจหรือไม่
  • ความต้องการเชิงฟังก์ชัน (Functional Requirement) — ฟังก์ชันที่ระบบต้องมีเพื่อให้ผู้ใช้ทำงานสำเร็จ เช่น "ระบบรองรับการเข้าระบบ", "ระบบรองรับการลงทะเบียนเรียน", "ค้นหาวิชา" (ตอบว่า "ใคร ทำอะไร")
  • ความต้องการไม่เชิงฟังก์ชัน (Non-functional Requirement) — คุณลักษณะ ประสิทธิภาพ ความปลอดภัย ข้อจำกัด ส่วนต่อประสานภายนอก เช่น "ตอบสนองไม่เกิน 5 วินาที", "รองรับ Google Chrome 65+", "ใส่โลโก้ทุกหน้า", "ทำงานแบบออนไลน์"

การระบุรายละเอียดกระบวนการทำงาน ต้องมีข้อมูล "ใคร / ทำอะไร / เมื่อใด / อย่างไร" — "ใคร"=ผู้ใช้ระบบ, "ทำอะไร"=ความต้องการเชิงฟังก์ชัน, "เมื่อใด"=จุดเริ่มงาน, "อย่างไร"=ขั้นตอนการทำงาน (ช่วยให้วิเคราะห์ครอบคลุมและพบผู้เกี่ยวข้องเพิ่ม) เทคนิคในแบบฝึกคือดูข้อความร้องขอแล้วระบุว่าเป็นความต้องการประเภทใด และยัง "ขาด" อะไร (เช่น ขาดขั้นตอน ใคร เมื่อใด อย่างไร)

8.1.2 วิธีการเก็บรวบรวมความต้องการ

ขั้นตอนการเก็บรวบรวม: 1) ระบุผู้มีส่วนได้ส่วนเสีย 2) เลือกเทคนิค 3) ดำเนินการเก็บ 4) จัดทำเอกสาร SRS

แบ่งวิธีเป็น 2 กลุ่ม:

(ก) วิธีดั้งเดิม (Traditional):

  1. การสัมภาษณ์ (Interview) — ควรวางแผนล่วงหน้า ถามเป็นกลางหลีกเลี่ยงคำถามนำ ตั้งใจฟัง/จดบันทึก สัมภาษณ์กลุ่มหลากหลาย; คำถามมี 2 แบบคือ ปลายปิด (มีตัวเลือก เช่น ใช่-ไม่ใช่, หลายตัวเลือก, จัดอันดับ) และ ปลายเปิด (เปิดกว้าง ไม่มีตัวเลือก)
  2. การสังเกตขณะปฏิบัติงาน (Observation) — สะท้อนความจริงตรงกว่าการสัมภาษณ์ (เพราะสัมภาษณ์เป็นการเล่าอดีตที่อาจข้ามขั้นตอน) แต่มีข้อจำกัด: ผู้ถูกสังเกตอาจเกร็ง/ระวังเกินไป, จำกัดด้านสถานที่/เวลา/จำนวนคน, บางองค์กรไม่อนุญาต (ความลับ)
  3. การเก็บข้อมูลจากเอกสารการทำงาน — สะท้อนกฎ/หลักเกณฑ์/รูปแบบข้อมูล (เช่น รหัสชุดวิชา มสธ. ยาว 5 หลัก) ประเภทเอกสาร: ด้านนโยบาย, กระบวนการทำงาน, แบบฟอร์ม, รายงาน; ข้อควรระวังคือเอกสารอาจล้าสมัยหรือขัดแย้งกับการปฏิบัติจริง
  4. แบบสอบถาม (Questionnaire) — เหมาะเมื่อต้องการข้อมูลจากคนจำนวนมากที่กระจายหลายพื้นที่ และคำตอบเป็นเชิงตัวเลขไม่ต้องวิเคราะห์ลึก นิยมคำถามปลายปิด (สรุปเป็นตัวเลขง่าย)

(ข) วิธีสมัยใหม่ (Modern):

  1. การประชุมร่วมกับผู้มีส่วนได้ส่วนเสีย (JAD) — นำเจ้าของงาน ผู้ใช้ นักวิเคราะห์ นักออกแบบ ผู้สร้างระบบ มาประชุมพร้อมกันเพื่อกำหนดความต้องการ ช่วยอภิปรายข้อขัดแย้งทันทีและสร้างความรู้สึกเป็นเจ้าของระบบ ผลลัพธ์คือชุดเอกสารรายละเอียดระบบปัจจุบันและความต้องการระบบใหม่
  2. การพัฒนาแอปพลิเคชันอย่างรวดเร็ว (RAD: Rapid Application Development) — สร้างต้นแบบ (Prototype) ให้ผู้ใช้ทดลองและปรับปรุงซ้ำจนพอใจ ขั้นตอน 4 ระยะ: วางแผนความต้องการ → ออกแบบ → พัฒนาต้นแบบ → คัทโอเวอร์ (Cutover คือยอมรับเป็นทางการ แปลงข้อมูล ทดสอบ อบรมผู้ใช้)

8.2.1 แผนภาพยูสเคสจากการวิเคราะห์ความต้องการ

นำข้อมูลจากการร้องขอระบบ + ข้อมูลจากการเก็บรวบรวมความต้องการ มาวิเคราะห์เพื่อระบุ แอ็คเตอร์ (Actor) และ ยูสเคส (Use Case) แล้วเขียนเป็นแผนภาพยูสเคส โดย เน้นความต้องการเชิงฟังก์ชันเท่านั้น ส่วนความต้องการไม่เชิงฟังก์ชันจะ "ไม่นำมาเขียน" ในแผนภาพยูสเคส (เพราะบอกคุณสมบัติระบบ ไม่ได้บอกว่าใครทำอะไร)

หลักการระบุยูสเคส: พิจารณา "งาน" ที่แอ็คเตอร์ต้องทำกับระบบ โดยเป็นงานสำคัญและจำเป็น แล้วหาความสัมพันธ์ของแอ็คเตอร์และยูสเคส

ความสัมพันธ์ที่พบในตัวอย่าง:

  • <<include>> — ยูสเคสหนึ่งเรียกใช้อีกยูสเคสเสมอ (เช่น "สั่งซื้อสินค้า" include "เข้าระบบ")
  • <<extend>> — ส่วนขยายที่อาจเกิด (เช่น "เข้าระบบ" extend "สมัครสมาชิก")
  • ความสัมพันธ์แบบทั่วไป (Generalization) ระหว่างแอ็คเตอร์ — มี Parent Actor และ Child Actor (เช่น "บุคคลทั่วไป" เป็น parent, "สมาชิก" เป็น child เมื่อมีความสัมพันธ์นี้แล้วไม่ต้องลากเส้นซ้ำจากสมาชิกไปค้นหาสินค้า); ยูสเคสก็มีความสัมพันธ์ทั่วไปได้ (เช่น "เข้าระบบ" แตกเป็น "เข้าระบบโดยสมาชิก" กับ "เข้าระบบโดยพนักงาน")

ข้อสังเกตจากตัวอย่าง: งานที่เกิดระหว่างผู้ใช้กับบุคคลภายนอกโดยไม่ผ่านระบบ (เช่น ชำระเงินกับธนาคาร) หรือเป็นการทำงานภายในของระบบ (เช่น ระบบยกเลิกการสั่งซื้อใน 24 ชม. อัตโนมัติ) "ไม่ต้องเขียน" ในยูสเคสความต้องการของระบบ เพราะยูสเคสแสดงเฉพาะ "งานที่แอ็คเตอร์ทำกับระบบ"

8.2.2 เอกสารรายละเอียดความต้องการซอฟต์แวร์ (SRS)

เอกสารที่ระบุความต้องการของผู้ใช้และของระบบอย่างเป็นทางการ เพื่อให้ผู้ร้องขอระบบ ผู้ออกแบบ ผู้พัฒนา ผู้ทดสอบ และหัวหน้าโครงการเข้าใจตรงกัน ใช้มาตรฐาน IEEE 830 (ค.ศ. 1998) ซึ่งเป็นมาตรฐานทั่วไปที่ปรับใช้ได้ มี 9 องค์ประกอบ:

1) บทนำ 2) คำอธิบายโดยรวม 3) ข้อกำหนดด้านอินเทอร์เฟซระหว่างระบบกับสิ่งแวดล้อมภายนอก 4) คุณสมบัติของระบบ 5) ความต้องการอื่นๆ 6) ข้อกำหนดความต้องการอื่นๆ 7) ภาคผนวก ก: อภิธานศัพท์ 8) ภาคผนวก ข: แบบจำลองการวิเคราะห์ 9) ภาคผนวก ค: รายชื่อที่กำหนด

ประเด็นที่ควรเน้นเพื่อสอบ

  • แยกความต้องการ 3 ประเภทให้ขาด: เชิงธุรกิจ (เป้าหมายธุรกิจ) vs เชิงฟังก์ชัน (ระบบทำอะไรได้-ใคร/ทำอะไร) vs ไม่เชิงฟังก์ชัน (คุณลักษณะ/เวลาตอบสนอง/รองรับเบราว์เซอร์/โลโก้) — ข้อสอบมักให้ข้อความแล้วถามว่าเป็นประเภทใด
  • เกณฑ์วิเคราะห์ "ใคร/ทำอะไร/เมื่อใด/อย่างไร" และวิเคราะห์ว่าข้อความร้องขอยัง "ขาด" อะไร
  • แยก Stakeholder กับ User: ผู้ใช้เป็นสับเซ็ตของผู้มีส่วนได้ส่วนเสีย (จำตัวอย่างสปอนเซอร์)
  • เทคนิคเก็บความต้องการ 6 วิธี (ดั้งเดิม 4 + สมัยใหม่ 2): จำจุดเด่น/ข้อจำกัดของแต่ละวิธี โดยเฉพาะ "สังเกตสะท้อนจริงกว่าสัมภาษณ์" และ "แบบสอบถามเหมาะกับคนจำนวนมาก/กระจายพื้นที่"
  • JAD vs RAD: JAD = ประชุมรวมผู้เกี่ยวข้อง; RAD = สร้างต้นแบบซ้ำ (4 ระยะ จบที่ Cutover)
  • คำถามปลายปิด vs ปลายเปิด ใช้ได้ทั้งสัมภาษณ์และแบบสอบถาม (แบบสอบถามนิยมปลายปิด)
  • การสร้างแผนภาพยูสเคสใช้เฉพาะความต้องการเชิงฟังก์ชัน ส่วนไม่เชิงฟังก์ชันไม่นำมาเขียน — ข้อนี้ออกสอบบ่อย
  • ความสัมพันธ์ <<include>> vs <<extend>> และความสัมพันธ์ทั่วไป (Generalization) ระหว่างแอ็คเตอร์ (Parent/Child)
  • งานที่ไม่ผ่านระบบ หรือทำงานภายในระบบอัตโนมัติ ไม่เขียนเป็นยูสเคส (ยูสเคส = งานที่แอ็คเตอร์ทำกับระบบ)
  • SRS ตามมาตรฐาน IEEE 830 (1998) มี 9 องค์ประกอบ และมีไว้ให้ผู้เกี่ยวข้องเข้าใจตรงกัน
  • จำว่า "การวิเคราะห์ระบบ" = "การวิเคราะห์ความต้องการของระบบ" คือการขยาย/แตกความต้องการ

หน่วยที่ 9: การออกแบบระบบเชิงวัตถุ

ภาพรวม

หน่วยนี้ต่อจากการวิเคราะห์ (หน่วย 8) เข้าสู่ขั้น "การออกแบบระบบเชิงวัตถุ (OOD: Object Oriented Design)" คือการนำผลลัพธ์ (แผนภาพต่างๆ) จากการวิเคราะห์มาปรับปรุงและเพิ่มรายละเอียดเชิงเทคนิค เปรียบเสมือนทำ "พิมพ์เขียว" ของระบบให้พร้อมส่งโปรแกรมเมอร์ แบ่งเป็น 3 ตอน: 9.1 หลักการออกแบบ (สถาปัตยกรรมระบบ/แอปพลิเคชัน/ฐานข้อมูล) 9.2 การออกแบบด้วยแผนภาพคอมโพเนนท์และดีพลอยเมนท์ 9.3 กรณีศึกษาระบบบทเรียนคอมพิวเตอร์ช่วยสอน (CAI) ผ่านอินเทอร์เน็ต

แนวคิดสำคัญ

  • ความต่างของการวิเคราะห์กับการออกแบบ (ออกสอบบ่อยมาก): การวิเคราะห์ (OOA) ถามว่า "ปัญหาที่ต้องแก้คืออะไร (What is the problem to be solved?)" โดย ไม่คำนึงถึงเครื่องคอมพิวเตอร์ ส่วนการออกแบบ (OOD) ถามว่า "ปัญหานั้นจะถูกแก้ด้วยวิธีใด (How to solve the problem?)" โดยทำให้ใช้งานได้จริงในคอมพิวเตอร์
  • การออกแบบระบบ (System Design) คือการนำความต้องการมาเป็นแบบแผน/พิมพ์เขียว นักวิเคราะห์ต้องออกแบบส่วนนำเข้า ส่วนส่งออก ฐานข้อมูล เครือข่าย ระบบความปลอดภัย และส่วนต่อประสานผู้ใช้
  • ในเชิงวัตถุ ถือว่า ทุกส่วนประกอบเป็นวัตถุ (Object) รวมถึงฮาร์ดแวร์ด้วย จึงออกแบบซอฟต์แวร์และฮาร์ดแวร์ได้โดยไม่ต้องเปลี่ยนหลักการ

9.1.1 ความรู้เกี่ยวกับการออกแบบระบบเชิงวัตถุ

ประเภทการออกแบบ 2 แบบ:

  • การออกแบบเชิงตรรกะ (Logical Design) — ออกแบบลักษณะการทำงานตามทางเลือกที่เลือก ยังไม่ระบุคุณลักษณะอุปกรณ์ เน้นรูปแบบรายงาน การนำเข้า/ผลลัพธ์ (สัมพันธ์กับขั้นวิเคราะห์มาก ใช้แผนภาพ E-R แสดงความสัมพันธ์ข้อมูล)
  • การออกแบบเชิงกายภาพ (Physical Design) — ระบุคุณลักษณะอุปกรณ์ เทคโนโลยี ภาษาโปรแกรม ฐานข้อมูล เครือข่าย ผลคือ "ข้อมูลเฉพาะของการออกแบบ (System Design Specification)" ส่งให้โปรแกรมเมอร์

การเปลี่ยนผ่านจาก OOA สู่ OOD คือการนำแผนภาพมา ปรับปรุง (Refine) ให้ใช้ในคอมพิวเตอร์ได้ง่ายขึ้น และเพิ่ม Diagram อื่นเพื่ออธิบายแนวทางแก้ปัญหา

หลักการออกแบบเชิงวัตถุ (OOD) แบ่งเป็น 4 ส่วน (จำให้ได้):

  1. การทำรีไฟน์เมนท์ (Refinement) — ปรับปรุงแผนภาพจากการวิเคราะห์ให้ดี/มีประสิทธิภาพขึ้น (ทำได้กับทุกแผนภาพ: ยูสเคส คลาส ลำดับ สถานะ)
  2. การออกแบบสถาปัตยกรรมซอฟต์แวร์ (Application Architecture Design)
  3. การออกแบบสถาปัตยกรรมฮาร์ดแวร์ (System Architecture Design)
  4. การออกแบบฐานข้อมูล (Persistent Data Design)

แผนภาพ UML ที่สร้างในขั้นนี้คือ แผนภาพคอมโพเนนท์ (Component Diagram) และ แผนภาพดีพลอยเมนท์ (Deployment Diagram) ส่วนแผนภาพคลาสและแผนภาพลำดับจะถูกเพิ่มรายละเอียดเชิงเทคนิค

หน้าที่ของนักออกแบบระบบ: แบ่งระบบเป็นระบบย่อย ระบุปัญหา/ความต้องการ เลือกวิธีเก็บข้อมูล เลือกวิธีพัฒนาโปรแกรม จัดการข้อผิดพลาด กำหนดลำดับความสำคัญ ฯลฯ

9.1.2 การออกแบบสถาปัตยกรรมระบบ (System Architecture Design)

ออกแบบทั้งฮาร์ดแวร์/ซอฟต์แวร์และความเกี่ยวข้องภายในระบบ แบ่งเป็น 4 ประเภท:

  1. แบบรวมศูนย์ (Centralized) — ใช้เมนเฟรมเป็นศูนย์กลาง (Host) มีเทอร์มินัล (Terminal) เชื่อม; ข้อดี: Code/Debug ง่าย; ข้อเสีย: คอขวดที่เซิร์ฟเวอร์ อัปเกรดยาก ราคาแพง
  2. แบบไฟล์เซิร์ฟเวอร์ (File Server) — ใช้พีซีเป็นเซิร์ฟเวอร์แชร์ไฟล์/เครื่องพิมพ์/พื้นที่เก็บข้อมูล
  3. แบบไคลเอ็นต์เซิร์ฟเวอร์ (Client-Server) — แบ่งประมวลผลระหว่างเซิร์ฟเวอร์กับไคลเอ็นต์ ประสิทธิภาพสูง; ข้อเสีย: พัฒนายาก ต้อง Debug ข้ามเครือข่าย อาจต้องมี Middleware
  4. แบบไคลเอ็นต์เซิร์ฟเวอร์เทียร์ (Tier) — ใช้เซิร์ฟเวอร์มากกว่า 1 ตัวช่วยงานเฉพาะด้าน เช่น ทรีเทียร์ (3-Tier) = Client + Application Server + Database Server; โฟร์เทียร์ (4-Tier) = Client + Web Server + Application Server + Database Server (รองรับอินเทอร์เน็ต) — แบ่งเลเยอร์ตรรกะเป็น Presentation Logic / Application Logic / Data Access Logic

9.1.3 การออกแบบสถาปัตยกรรมแอปพลิเคชัน + การออกแบบฐานข้อมูล

  • ข้อมูลแบบถาวร (Persistent Data) = ข้อมูลที่ยังอยู่แม้ปิดโปรแกรม เก็บได้หลายรูปแบบ (file, Database, Network DB, Hierarchical DB)
  • ในเชิงวัตถุไม่มีฐานข้อมูลถาวรเฉพาะ จึงใช้ ฐานข้อมูลเชิงสัมพันธ์ (Relational Database) ทดแทน (ผู้ริเริ่มโมเดลคือ Dr. Edgar F. Codd) จำเป็นต้อง "แปลงแผนภาพคลาสเป็นฐานข้อมูลเชิงสัมพันธ์"
  • แนวคิดวัตถุกับฐานข้อมูลมี 2 แบบ: OODBMS (Object Oriented DBMS) และ ORDBMS (Object Relational DBMS)
  • ส่วนประกอบรีเลชัน: แถว (Row/Tuple) จำนวนแถว = Cardinality; คอลัมน์ (Column/Attribute) จำนวนคอลัมน์ = Degree; มี Primary Key และ Foreign Key
  • การแมประหว่างคลาสกับตาราง: Class → Table, Object → Row, Attribute → Column (ไม่กล่าวถึง Method เพราะไม่ใช่ข้อมูล) ใช้ภาษา DDL (Data Definition Language) สร้างตาราง โดย Primary Key กำหนดเป็น NOT NULL เสมอ
  • หลักการแปลงความสัมพันธ์เป็นตาราง: One-to-One, One-to-Many (นำ PK ฝั่ง Max=1 ไปเป็น FK ฝั่ง Max>1), Many-to-Many (สร้าง Associate Table), Aggregation/Composition (Composition ตั้ง FK เป็น NOT NULL และเป็นส่วนหนึ่งของ PK เพื่อบังคับว่า Part อยู่ไม่ได้ถ้าไม่มี Whole), Generalization (Sub Class ใช้ PK เหมือน Super Class และเป็น FK อ้างอิงกลับ; มีรูปแบบ Total-Overlapping, Total-Exclusive, Partial-Exclusive, Partial-Overlapping)

9.2.1 การแบ่งระบบใหญ่ออกเป็นระบบย่อย (System Decomposition)

ระบบซับซ้อนนิยมแบ่งเป็นระบบย่อย (Subsystem) ในหน่วยนี้ใช้วิธี ดริลดาวน์เอ็นจิเนียริง (Drill Down Engineering) คือขุดเจาะระบบย่อยทีละส่วนจนละเอียดที่สุดหรือจนทีมเห็นสมควร แบ่งเป็น 3 ส่วนหลัก:

  1. ระบบย่อยส่วนติดต่อผู้ใช้ (Presentation Logic Subsystem) — User Interface, Output, Input
  2. ระบบย่อยส่วนการทำงาน (Working/Business Logic Subsystem) — การคำนวณ ดึง/บันทึกข้อมูล ที่ทำใน CPU
  3. ระบบย่อยส่วนฐานข้อมูล (Database Logic Subsystem) — รายการข้อมูลที่จัดเก็บ สัมพันธ์กับ Persistent Data Design

9.2.2 แผนภาพคอมโพเนนท์ (Component Diagram)

เป็น แผนภาพเชิงสถิตย์ (Static Diagram) จำลองลักษณะทางกายภาพของระบบเชิงวัตถุ แสดงส่วนประกอบซอฟต์แวร์ (Software Component) และความสัมพันธ์ มีความสัมพันธ์กับแผนภาพคลาส เพราะ 1 คอมโพเนนท์ประกอบด้วยคลาสตั้งแต่ 1 คลาสขึ้นไป

สัญลักษณ์คอมโพเนนท์ (รูปสี่เหลี่ยมมีสี่เหลี่ยมเล็ก 2 อันด้านซ้าย) ใช้แทน 4 อย่าง: 1) โปรแกรมที่ประมวลผลได้ (Executable Program เช่น .exe) 2) แฟ้มซอร์สโค้ด (Source Code) 3) ฐานข้อมูล (Database) 4) ชุดข้อมูลในฐานข้อมูล (Table)

สัญลักษณ์ความสัมพันธ์: 1) การเรียกใช้/ขึ้นต่อกัน (Calls, Uses, Dependency) 2) การเชื่อมโยงทั่วไป (General Connection เช่นผ่านเครือข่าย/อินเทอร์เน็ต) 3) การเป็นส่วนประกอบ (Aggregation)

ประโยชน์: เห็นภาพรวมระบบ แบ่งงานให้ทีมรับผิดชอบเป็นส่วนๆ ได้

9.2.3 แผนภาพดีพลอยเมนท์ (Deployment Diagram)

ใช้ทำ System Architecture Design (ฝั่งฮาร์ดแวร์) แสดงสถาปัตยกรรมเชิงกายภาพ (Physical Architecture) ขณะระบบทำงานจริง ได้แก่ หน่วยประมวลผล (Processor) อุปกรณ์ (Devices) และซอฟต์แวร์ที่ออกแบบไว้

สัญลักษณ์หลัก:

  • ใช้สัญลักษณ์ "ลูกบาศก์ (Cube)" แทน โหนด (Node) = อุปกรณ์ฮาร์ดแวร์ ชื่อโหนดเป็น Unique Identity และมี Path Name บอกประเภทฮาร์ดแวร์
  • ใช้ สเตอริโอไทป์ (Stereotype) เขียนในเครื่องหมาย << >> (เช่น <<PC>>, <<Web Browser>>, <<Protocol>>, <<DB Server>>)
  • Communication Path = เส้นเชื่อมต่อระหว่างอุปกรณ์ โดยระบุรูปแบบ เช่น <<LAN>>, <<TCP/IP>>, <<HTTP>>

องค์ประกอบ: ฝั่งซอฟต์แวร์ = Data Storage, Data Access Logic, Application Logic; ฝั่งฮาร์ดแวร์ = Server Computer, Client Computer, Connecting Network

ตัวอย่างกรณีศึกษา: ระบบเรียกดูเอกสารเครือข่าย (LAN/Intranet + ODBC + SQL Server), ระบบขายผ่าน Tablet (Internet ไร้สาย + TCP/IP), ระบบ Printer Pool (LAN + Print Server), ระบบจองตั๋ว (Internet + 2 Server + Subsystem ตัดสินใจกระจายโหลด/กันล่ม)

9.3 กรณีศึกษา: ระบบบทเรียนคอมพิวเตอร์ช่วยสอน (CAI) ผ่านอินเทอร์เน็ต

ระบบมี 15 บทเรียน + แบบทดสอบท้ายบท 15 ชุด ผู้เรียนทำแบบทดสอบได้ ≥ 60% ถือว่าผ่าน ไปบทถัดไปได้ ต่ำกว่าต้องเรียนใหม่ ตอนนี้สาธิตการนำ OOA (แผนภาพคลาส, แผนภาพลำดับ) มาทำ OOD โดยเพิ่มรายละเอียดเชิงเทคนิคและความต้องการไม่เชิงฟังก์ชัน แล้วส่งเป็นเอกสารออกแบบ (UML Design Document) ให้โปรแกรมเมอร์

ประเด็นที่ควรเน้นเพื่อสอบ

  • ความต่างของการวิเคราะห์ (OOA = What) กับการออกแบบ (OOD = How): OOA ไม่คำนึงเครื่องคอมพิวเตอร์ มองหาปัญหา; OOD หาวิธีแก้ให้ใช้งานจริงในคอมพิวเตอร์ — ข้อนี้ออกสอบแน่นอน
  • 4 ส่วนของการออกแบบเชิงวัตถุ: Refinement, Application Architecture, System Architecture, Persistent Data Design
  • Logical vs Physical Design: Logical ยังไม่ระบุอุปกรณ์ (ใกล้ขั้นวิเคราะห์); Physical ระบุอุปกรณ์/เทคโนโลยี ได้ System Design Specification
  • แยก 2 แผนภาพหลักของหน่วยนี้: Component Diagram = ฝั่งซอฟต์แวร์ (Application Architecture); Deployment Diagram = ฝั่งฮาร์ดแวร์ (System Architecture)
  • คอมโพเนนท์ใช้แทน 4 อย่าง: Executable Program, Source Code, Database, Table — และ 1 คอมโพเนนท์มีคลาส ≥ 1
  • แผนภาพดีพลอยเมนท์ใช้ลูกบาศก์แทน Node และใช้ Stereotype << >> + Communication Path (LAN/TCP/IP/HTTP)
  • สถาปัตยกรรมระบบ 4 แบบ: รวมศูนย์, ไฟล์เซิร์ฟเวอร์, ไคลเอ็นต์เซิร์ฟเวอร์, เทียร์ — รู้ข้อดี/ข้อเสีย และโครงสร้าง 3-Tier (Client/App/DB) กับ 4-Tier (เพิ่ม Web Server)
  • Persistent Data + การแมป Class→Table, Object→Row, Attribute→Column (Method ไม่แมปเพราะไม่ใช่ข้อมูล) และ Primary Key เป็น NOT NULL เสมอ
  • หลักการแปลงความสัมพันธ์เป็นตาราง: จุดสำคัญคือ Composition บังคับ FK เป็น NOT NULL + เป็นส่วนหนึ่งของ PK (เพราะ Part อยู่ไม่ได้ถ้าไม่มี Whole) ต่างจาก Aggregation ที่ FK เป็น NULL ได้
  • System Decomposition แบบ Drill Down และ 3 ระบบย่อย (Presentation / Business / Database Logic)
  • จำว่าแผนภาพ UML ที่สร้างใหม่ในขั้นออกแบบคือ Component และ Deployment ส่วน Class/Sequence ถูกเพิ่มรายละเอียดเชิงเทคนิค

ภาพรวม

หน่วยนี้ว่าด้วย "แผนภาพปฏิสัมพันธ์" (Interaction Diagram) ซึ่งเป็นกลุ่มแผนภาพเชิงพฤติกรรม (Behavior Diagram) ในมาตรฐาน UML 2.5 ที่ใช้แสดงรูปแบบการรับ-ส่งการทำงานและการสื่อสารระหว่างคลาส/วัตถุ (class/object) ในระบบเชิงวัตถุ เนื้อหาเน้น 2 แผนภาพที่นิยมแพร่หลาย คือ แผนภาพลำดับ (Sequence Diagram) ที่ร้อยเรียงปฏิสัมพันธ์ตามลำดับเวลา และแผนภาพการสื่อสาร (Communication Diagram) ที่เน้นตำแหน่งและความเป็นศูนย์กลางของการสื่อสาร โดยทั้งสองแผนภาพมีวัตถุประสงค์เดียวกันคือสร้างความเข้าใจที่ตรงกัน ต่างกันเพียงวิธีนำเสนอ

ปฏิสัมพันธ์ (interaction) คือกระบวนการทำงานร่วมกันระหว่างองค์ประกอบตั้งแต่ 2 องค์ประกอบขึ้นไปเพื่อให้เกิดผลลัพธ์ร่วมกัน ในระบบเชิงวัตถุ การทำงานถูกแบ่งเป็นหน่วยย่อย (คลาส/วัตถุ) ที่ทำงานเสร็จสมบูรณ์ในตัวเอง และเปิด "ส่วนหน้า" ไว้ให้องค์ประกอบอื่นติดต่อผ่านการเรียกใช้ตามแบบแผนเท่านั้น (เปรียบกับห้องยาที่มีเฉพาะเคาน์เตอร์จ่ายยาให้คนภายนอกติดต่อ) การออกแบบปฏิสัมพันธ์จึงสำคัญมาก เพราะยิ่งแยกองค์ประกอบย่อยมากเท่าใด ยิ่งต้องการการสื่อสารที่มีประสิทธิภาพมากขึ้น แผนภาพปฏิสัมพันธ์ช่วยให้ผู้พัฒนาเห็นภาพการรับส่งงานได้ชัดเจน และเป็นเครื่องนำทางให้นักเขียนโปรแกรมพัฒนาได้ตรงตามที่นักออกแบบวางไว้

เนื้อหาหลัก

10.1 ความรู้พื้นฐานเกี่ยวกับแผนภาพปฏิสัมพันธ์

องค์ประกอบหลักของแผนภาพปฏิสัมพันธ์ มี 2 ส่วน คือ (1) คลาส/วัตถุ (class/object) ซึ่งเป็นองค์ประกอบที่ทำงานเสร็จในตัวเองและมีส่วนติดต่อภายนอก และ (2) เส้นแสดงปฏิสัมพันธ์ระหว่างคลาส/วัตถุ ที่มีทิศทางออกจากคลาส/วัตถุหนึ่งวิ่งเข้าหาอีกอันหนึ่ง แสดงการเรียกใช้บริการพร้อมการรับส่งข้อมูล

ประเภทของแผนภาพปฏิสัมพันธ์ (ตาม UML 2.5) มี 4 ประเภท ได้แก่ แผนภาพลำดับ (Sequence Diagram), แผนภาพการสื่อสาร (Communication Diagram / เดิมเรียก Collaboration Diagram ใน UML v.1), แผนภาพช่วงเวลา (Timing Diagram) และแผนภาพปฏิสัมพันธ์ภาพรวม (Interaction Overview Diagram) — หน่วยนี้เน้นเฉพาะ 2 ประเภทแรก

10.1.2 แผนภาพลำดับ (Sequence Diagram) — สัญลักษณ์สำคัญ

แผนภาพลำดับแสดงปฏิสัมพันธ์ "เรียงตามลำดับเวลา" สัญลักษณ์หลักได้แก่

  • เส้นชีวิต (Lifeline): เส้นประแนวดิ่ง มีกรอบสี่เหลี่ยมชื่ออยู่ด้านบน รูปแบบมาตรฐาน UML คือ ชื่อวัตถุ : ชื่อคลาส (เช่น data: Stock) สำหรับผู้ใช้ระบบ (Actor) ใช้สัญลักษณ์รูปคนแทนกรอบสี่เหลี่ยม นอกจากคลาส/วัตถุ/Actor ยังเขียนเป็นเส้นชีวิตได้แก่ Boundary Element (ส่วนติดต่อผู้ใช้ เช่น หน้าจอ), Control Element (ตรรกะที่เชื่อม Boundary กับ Entity) และ Entity Element (องค์ประกอบเชิงแนวคิดแบบ Entity ใน ER)
  • Occurrence: ช่วงเวลาบนเส้นชีวิตที่คลาส/วัตถุกำลังทำงาน วาดเป็นสี่เหลี่ยมผืนผ้าบางเกาะบนเส้นชีวิต
  • ข้อความ (Message): การเรียกให้คลาส/วัตถุอื่นทำงาน มี 3 แบบ คือ Synchronous (ลูกศรทึบ — ผู้ส่งรอผลก่อนทำต่อ), Asynchronous (ลูกศรแบบแง่ง/เปิด — ผู้ส่งทำงานอื่นต่อได้) และ Return (เส้นประลูกศรแง่ง — ข้อความตอบกลับ) ชื่อกำกับเส้นข้อความอาจเป็นชื่อข้อมูล/เอกสาร หรือชื่อเมทอดพร้อมพารามิเตอร์ เปรียบกับการเรียกเมทอดในการเขียนโปรแกรมเชิงวัตถุ

รูปแบบปฏิสัมพันธ์ในแผนภาพลำดับ ที่ควรรู้: Execution Occurrence, Destruction Occurrence (หยุดวัตถุทันที), Overlapping Execution (Occurrence ซ้อนกัน), Synchronous/Asynchronous Call, Create Message (สร้างคลาสใหม่), Lost/Found Message (ต้นทาง/ปลายทางนอกขอบเขต ใช้จุดทึบแทน) และกรอบควบคุมการไหล ได้แก่ alt (Alternative — เงื่อนไขแบบ if-else), opt (Option — ทำหรือไม่ทำ), loop (วนซ้ำ — ระบุจำนวนรอบหรือไม่ก็ได้ และมี break), par (Parallel — ทำงานคู่ขนาน)

การวาดแผนภาพลำดับ มักวาด 1 แผนภาพคู่กับ 1 ยูสเคส (use case) และสัมพันธ์กับแผนภาพคลาส เพราะคลาส/วัตถุและชื่อเมทอด/พารามิเตอร์ต้องสอดคล้องกัน ขั้นตอน: เลือกยูสเคส (รวม include/extend) → เขียนลำดับกิจกรรมและระบุคลาส/เมทอดที่เกี่ยวข้อง → นำ Actor และคลาสมาวาดเป็นเส้นชีวิตแล้วใส่เส้นข้อความทีละขั้น

10.1.3 แผนภาพการสื่อสาร (Communication Diagram)

นำเสนอปฏิสัมพันธ์เหมือนแผนภาพลำดับ แต่ "วางตำแหน่งวัตถุ/คลาสอย่างอิสระ" จึงเน้นบริเวณที่เป็นศูนย์กลางของระบบได้ดีกว่า องค์ประกอบ:

  • เฟรม (Frame): กรอบสี่เหลี่ยมแสดงขอบเขต มีชื่อแผนภาพในกรอบตัดมุมล่างขวาบริเวณมุมบนซ้าย
  • คลาส/วัตถุ: กรอบสี่เหลี่ยมมีชื่อ แต่ "ไม่มีเส้นประลงแนวดิ่ง"
  • เส้นข้อความ: ใช้ "ตัวเลขลำดับ" กำกับแทนการเรียงบนลงล่าง โดยเลขน้อยเกิดก่อนเลขมาก, ใช้ทศนิยมจัดกลุ่ม (เช่น 2 ก่อน 2.1), ใช้ตัวอักษรต่อท้ายเลขเดียวกันแสดงการทำงานคู่ขนาน (เช่น 2.3a กับ 2.3b ทำพร้อมกัน), ใช้วงเล็บ [ ] แสดงเงื่อนไข และใช้ * หรือ *|| แสดงการวนซ้ำ (*|| คือวนแบบคู่ขนานพร้อมกัน)

ประเด็นที่ควรเน้นเพื่อสอบ

  • Sequence vs Communication: ทั้งคู่แสดงปฏิสัมพันธ์เดียวกัน ต่างที่การนำเสนอ — Sequence เรียง "ตามลำดับเวลา" (เส้นชีวิตยาวลงล่าง), Communication วางตำแหน่งอิสระและใช้ "ตัวเลขลำดับ" บอกก่อนหลัง เน้นศูนย์กลางการสื่อสาร
  • จำองค์ประกอบ/สัญลักษณ์ของแผนภาพลำดับให้ได้: Lifeline, Occurrence/Execution, Message (3 แบบ) และรูปแบบเส้นชีวิตของ Actor (รูปคน) vs คลาส/วัตถุ (กรอบสี่เหลี่ยม)
  • แยกความต่างของข้อความ 3 แบบ: Synchronous (ลูกศรทึบ, รอผล) / Asynchronous (ลูกศรแง่ง, ไม่รอ) / Return (เส้นประ)
  • เข้าใจกรอบควบคุมการไหล: alt = if-else, opt = ทำหรือไม่ทำ, loop = วนซ้ำ (loop(10) วน 10 รอบ, break ออกจากลูป), par = คู่ขนาน
  • จดจำรูปแบบพิเศษ: Create Message (สร้างวัตถุใหม่), Destruction Occurrence (หยุดวัตถุ), Overlapping Execution (Occurrence ซ้อน), Lost/Found Message (นอกขอบเขต)
  • การกำกับเส้นข้อความในแผนภาพการสื่อสารด้วยตัวเลข: เลขน้อยมาก่อน, ทศนิยมใช้จัดกลุ่ม, ตัวอักษรเดียวกันที่เลขเดียวกัน = คู่ขนาน, [ ] = เงื่อนไข, * และ *|| = วนซ้ำ (แบบลำดับ/แบบขนาน)
  • ความสัมพันธ์ของแผนภาพปฏิสัมพันธ์กับแผนภาพยูสเคสและแผนภาพคลาส: มัก 1 แผนภาพลำดับต่อ 1 ยูสเคส และชื่อเมทอด/พารามิเตอร์ต้องตรงกับแผนภาพคลาส
  • UML 2.5 จัดแผนภาพปฏิสัมพันธ์เป็น Behavior Diagram และมี 4 ประเภท (Sequence, Communication, Timing, Interaction Overview)

หน่วยที่ 11: แผนภาพสถานะและแผนภาพกิจกรรม (State Machine & Activity Diagram)

ภาพรวม

หน่วยนี้นำเสนอแผนภาพเชิงพฤติกรรม (behavioral diagram) แบบพลวัต/ไดนามิก (dynamic diagram) 2 ชนิด คือ แผนภาพสถานะ (State Machine / Statechart Diagram) ที่เน้นแสดง "สถานะ" และ "การเปลี่ยนสถานะ" ของวัตถุหนึ่งวัตถุเมื่อมีเหตุการณ์มากระตุ้น กับแผนภาพกิจกรรม (Activity Diagram) ที่เน้น "กระแสการไหลของกิจกรรม" (workflow) จากกิจกรรมหนึ่งไปอีกกิจกรรมหนึ่ง คล้ายผังงาน (flowchart) แต่มีรูปแบบเส้นทางการไหลมากกว่า ทั้งสองแผนภาพช่วยให้นักวิเคราะห์และออกแบบระบบมองเห็นการเปลี่ยนแปลงและขั้นตอนการทำงานของระบบได้ชัดเจนขึ้น

แนวคิดสำคัญ

แผนภาพสถานะใช้กับวัตถุเพียง "ชนิดเดียว" เพื่อแสดงว่าวัตถุนั้นมีสถานะใดได้บ้างและเปลี่ยนสถานะอย่างไรเมื่อมีเหตุการณ์ (event) มากระตุ้น — ต่างจากแผนภาพลำดับที่แสดงการสื่อสารระหว่างหลายวัตถุ ระบบที่ตอบสนองต่อเหตุการณ์ลักษณะนี้เรียกว่า รีแอกทีฟซิสเต็ม (reactive system) อินพุตสำคัญในการเขียนแผนภาพสถานะคือ แผนภาพคลาส (class diagram) และแผนภาพลำดับ (sequence diagram) โดยทั่วไปจำนวนแผนภาพสถานะ = จำนวนเมธอดของแต่ละคลาสรวมกัน แต่เลือกเฉพาะเมธอดที่ซับซ้อนมาเขียนก็พอ ส่วนแผนภาพกิจกรรมใช้แสดงการทำงานย่อยของวัตถุในแต่ละยูสเคส โดยไม่เน้นที่ตัววัตถุแต่เน้นที่ลำดับการไหลของงาน

11.1 แผนภาพสถานะ (State Machine Diagram)

องค์ประกอบหลัก 4 อย่าง: วัตถุ (object), สถานะของวัตถุ (state), การเปลี่ยนสถานะ (transition) และเหตุการณ์ที่กระตุ้น (event)

1) เหตุการณ์ (event): ข่าวสาร (message) หรือสัญญาณ (signal) ที่วัตถุได้รับจากการเรียกใช้เมธอดในคลาส อาจทำให้เปลี่ยนสถานะ หรือคงสถานะเดิม (self transition) ก็ได้ (เช่น วัตถุตู้เอทีเอ็มมีสถานะว่าง/ไม่พร้อมให้บริการ, วัตถุบัญชีธนาคารมีสถานะปกติ/แจ้งเตือนเงินไม่พอถอน/บัญชีถูกปรับปรุง)

2) สถานะ (state): ค่าที่บอกถึงวัตถุ ณ เวลาหนึ่ง ถูกกำหนดโดยค่าของแอตทริบิวต์ (attribute) สัญลักษณ์มาตรฐาน UML คือ สี่เหลี่ยมมุมมน (round rectangle) ที่มีชื่อสถานะอยู่ภายใน ยกเว้น 2 สถานะพิเศษ ได้แก่ สถานะเริ่มต้น (initial state) = วงกลมทึบ และ สถานะสิ้นสุด (end/final state) = วงกลมทึบในวงกลมใส

  • กิจกรรมภายในสถานะ (internal activity) มี 4 ลักษณะ ทำให้กลายเป็น สถานะรวม (composite state) คือ entry/action (ทำเมื่อเข้าสถานะ), do/action (ทำระหว่างอยู่ในสถานะ), exit/action (ทำก่อนออกจากสถานะ) และ condition/action (ทำเมื่อเงื่อนไขเป็นจริง)
  • สถานะย่อย (substate) 2 ประเภท: แบบเรียงลำดับ (sequential substate) เปลี่ยนทีละสถานะตามลำดับ และ แบบขนาน (concurrent substate) สถานะย่อยหลายอันเปลี่ยนพร้อมกัน โดยต้องระบุสถานะหลัก (super state) ก่อน

3) การเปลี่ยนสถานะ (transition): ความสัมพันธ์ระหว่าง 2 สถานะ องค์ประกอบพื้นฐาน ได้แก่ สถานะต้นทาง (source state), สถานะปลายทาง (target state), ตัวกระตุ้น (event trigger), เงื่อนไข (guard condition) และการกระทำ (action) สัญลักษณ์เป็นเส้นลูกศรจากสถานะต้นทางไปปลายทาง รูปแบบการเขียนบนเส้นคือ

> เหตุการณ์/ตัวกระตุ้น (event trigger) [เงื่อนไข (guard)] / การกระทำ (action)

โดยส่วนเงื่อนไขและการกระทำต้องมีอย่างน้อยหนึ่งอย่าง รูปแบบการเปลี่ยนสถานะมี 2 แบบ: (1) เปลี่ยนจากสถานะหนึ่งไปอีกสถานะเมื่อมีตัวกระตุ้น และ (2) คงสถานะเดิม = เซลฟ์ทรานซิชัน (self transition) (เช่น "กำลังตรวจสอบ" รายการสินค้าทีละชิ้น จะวนอยู่สถานะเดิมจนกว่าจะครบ)

โครงสร้างการเขียน 2 ลักษณะ: แบบทั่วไป (ระบุเฉพาะสถานะหลัก) และแบบมีสถานะย่อย/กิจกรรมภายใน หลักการเขียน: พิจารณาจำนวนแผนภาพจากเมธอดในแผนภาพคลาส → หาวัตถุสำคัญและระบุสถานะที่เป็นไปได้ (ใช้แผนภาพลำดับช่วย) → เขียนแผนภาพสถานะของแต่ละเมธอด → เพิ่มสถานะ/กิจกรรมภายในให้สมบูรณ์ → ทำซ้ำจนครบทุกคลาส

11.2 แผนภาพกิจกรรม (Activity Diagram)

แผนภาพกิจกรรมอยู่ในกลุ่มเชิงพฤติกรรม แต่ไม่เน้นที่ตัววัตถุ — เน้น "การไหลของกิจกรรม/กระแสงาน (workflow)" คล้าย flowchart แต่มีเส้นทางการไหลหลากหลายกว่า ใช้แสดงการทำงานย่อยของวัตถุในแต่ละยูสเคส

องค์ประกอบพื้นฐาน:

  • จุดเริ่มต้นและจุดสิ้นสุด (ต้องมีทั้งคู่เสมอ)
  • กิจกรรม (activity / action): การทำงานในระบบ เช่น การคำนวณผลลัพธ์, การเปลี่ยนสถานะของระบบ, การส่งค่ากลับคืน, การส่งสัญญาณ, การเรียกเมธอดอื่น, การสร้าง/ทำลายวัตถุ
  • เส้นทางการไหลของกิจกรรม อธิบายไว้ 3 รูปแบบหลัก:

1. แบบต่อเนื่อง (sequential) — กิจกรรมไหลทีละขั้นตามลำดับ (เช่น เลือกไฟล์ → อัปโหลดไฟล์)

2. แบบมีทางเลือก/ทางแยก (branched/decision) — มีทางเลือกแยกออกตามเงื่อนไข (เปรียบกับ decision)

3. แบบทำขนานกันไป (concurrent) — งานหลายอย่างทำพร้อมกัน (เกี่ยวข้องกับ fork/join)

สวิมเลน/พาร์ทิชัน (swimlane / partition): ใช้แบ่งแสดงว่า "ใครเป็นผู้รับผิดชอบ" ในแต่ละกิจกรรม ทำให้โครงสร้างแผนภาพกิจกรรมมี 2 รูปแบบ คือ แบบทั่วไป (ไม่ระบุผู้รับผิดชอบ) และแบบระบุผู้รับผิดชอบชัดเจนด้วยสวิมเลน

คุณลักษณะที่ควรมี: เน้นการสื่อสารเชิงพฤติกรรม, เลือกแสดงเฉพาะส่วนสำคัญ, มีจุดเริ่มต้นและจุดสิ้นสุดเสมอ, อ่านจากบนลงล่าง (แนวตั้ง) หรือซ้ายไปขวา (แนวนอน) หลักการเขียน: วิเคราะห์กิจกรรมและเงื่อนไขให้ครบ เรียงลำดับให้เหมาะสม และเพิ่มเส้นทางไหลไปยังวัตถุที่ได้รับผลกระทบจากกระแสงาน

ประเด็นที่ควรเน้นเพื่อสอบ

  • State Machine vs Sequence Diagram: แผนภาพสถานะเน้นวัตถุ "ชนิดเดียว" และการเปลี่ยนสถานะ ส่วนแผนภาพลำดับเน้นการสื่อสารระหว่าง "หลายวัตถุ"
  • จำองค์ประกอบ 4 อย่างของแผนภาพสถานะ: object, state, transition, event และสัญลักษณ์ 3 ตัวสำคัญ (สี่เหลี่ยมมุมมน = state, วงกลมทึบ = initial, วงกลมทึบในวงใส = final)
  • รูปแบบการเขียนบนเส้น transition: event trigger [guard] / action (เงื่อนไขหรือการกระทำต้องมีอย่างน้อยหนึ่ง)
  • กิจกรรมภายในสถานะ 4 ลักษณะ: entry / do / exit / condition (ออกสอบบ่อย — แยกให้ได้ว่าแต่ละคำทำเมื่อใด)
  • สถานะย่อย: sequential (เรียงลำดับ) vs concurrent (ขนาน), และเข้าใจ self transition (คงสถานะเดิม)
  • อินพุตของแผนภาพสถานะ = แผนภาพคลาส (กำหนดจำนวนแผนภาพ) + แผนภาพลำดับ (เป็นแนวทางสถานะ); จำนวนแผนภาพสถานะ ≈ จำนวนเมธอด แต่เลือกเฉพาะที่ซับซ้อน
  • องค์ประกอบของแผนภาพกิจกรรม: จุดเริ่มต้น, จุดสิ้นสุด, กิจกรรม (activity), เส้นทางการไหล — ต้องมีจุดเริ่มต้นและจุดสิ้นสุดเสมอ
  • เส้นทางไหล 3 แบบ: ต่อเนื่อง (sequential), ทางแยก/มีทางเลือก (branched/decision), ขนาน (concurrent) — แบบขนานเกี่ยวข้องกับ fork/join
  • สวิมเลน (swimlane/partition): ใช้ระบุผู้รับผิดชอบกิจกรรม ทำให้โครงสร้างแผนภาพกิจกรรมแยกเป็นแบบไม่ระบุ vs ระบุผู้รับผิดชอบ
  • ความต่างของแผนภาพกิจกรรมกับ flowchart: คล้ายกันแต่แผนภาพกิจกรรมมีรูปแบบเส้นทางการไหล (คู่ขนาน/สวิมเลน) มากกว่า

หน่วยที่ 12: การทดสอบระบบ (System Testing)

ภาพรวม

หน่วยนี้ว่าด้วยการทดสอบระบบ (system testing) ซึ่งเป็นกระบวนการค้นหาข้อผิดพลาด/ข้อบกพร่องและตรวจสอบว่าระบบทำงานตรงตามข้อกำหนดความต้องการ (system requirement specification) เป็นขั้นตอนสำคัญในการประกันคุณภาพซอฟต์แวร์ (software quality assurance) เนื้อหาครอบคลุมแนวคิดและมาตรฐาน ISO/IEC/IEEE 29119, แนวคิดการทวนสอบ (verification) กับการตรวจสอบยืนยันความถูกต้อง (validation) ผ่านแบบจำลองวี (V-model), การจำแนกประเภทการทดสอบ 3 มุมมอง (ตามระดับชั้น/ตามหน้าที่/ตามวิธีการ) และเอกสารการทดสอบกับการออกแบบกรณีทดสอบ (test case)

แนวคิดสำคัญ

วัตถุประสงค์ของการทดสอบระบบมี 4 ข้อ: (1) เพื่อการป้องกัน (preventive) เช่น ตรวจไวยากรณ์ (syntax) และข้อผิดพลาดทางตรรกะ (logic error), (2) เพื่อการค้นหา (detective) จุดที่ระบบทำงานผิด (runtime error), (3) เพื่อการแก้ไข (corrective) และ (4) เพื่อให้เป็นไปตามมาตรฐานคุณภาพ การทดสอบที่ดีต้องเตรียมข้อมูลทดสอบ (test data) หรือกรณีทดสอบ (test case) ให้ครอบคลุมทุกเงื่อนไข แล้วเปรียบเทียบผลที่ได้จริง (actual output) กับผลที่คาดหวัง (expected output) — ถ้าต่างกันถือว่าโปรแกรมไม่ผ่าน (fail) ยิ่งให้ความสำคัญกับการทดสอบและตรวจพบข้อผิดพลาดในขั้นตอนแรกๆ มากเท่าใด ยิ่งลดค่าใช้จ่ายในการแก้ไข (ข้อผิดพลาดที่ย้อนกลับไปถึงขั้นเก็บความต้องการมีค่าใช้จ่ายสูงสุด)

เนื้อหาหลัก

12.1 แนวคิดของการทดสอบระบบ

มาตรฐาน ISO/IEC/IEEE 29119 (วิศวกรรมซอฟต์แวร์และระบบ–การทดสอบซอฟต์แวร์) เป็นข้อแนะนำ (guideline) มี 5 ส่วน: ส่วนที่ 1 แนวคิดและนิยาม (Concepts & Definitions), ส่วนที่ 2 กระบวนการทดสอบ (Process), ส่วนที่ 3 เอกสารการทดสอบ (Documentation), ส่วนที่ 4 เทคนิคการทดสอบ (Testing Techniques), ส่วนที่ 5 การทดสอบด้วยคำสำคัญ (Keyword-Driven Testing)

  • ส่วนที่ 2 กระบวนการทดสอบ แบ่งเป็น 3 กระบวนการ: (1) กระบวนการทดสอบระดับองค์กร (organizational test process — กำหนดนโยบาย/กลยุทธ์การทดสอบ), (2) กระบวนการจัดการการทดสอบ (test management process — วางแผน → ติดตามและควบคุม → ดำเนินการเมื่อเสร็จสิ้น) และ (3) กระบวนการทดสอบแบบไดนามิก (dynamic test process — สั่งโปรแกรมทำงานจริงแล้วดูพฤติกรรม ใช้ test data, test object และอาจมี test stub เป็นโปรแกรมแทนส่วนที่ถูกเรียกใช้แต่ยังไม่ได้ทดสอบ)

การทวนสอบ (verification) vs การตรวจสอบยืนยันความถูกต้อง (validation)

  • Verification = "Building the system right?" สร้างระบบถูกต้องตามกระบวนการ/ข้อกำหนดหรือไม่ — ทดสอบสิ่งที่เกิดขึ้นระหว่างพัฒนา (work product) เช่น unit testing, integration testing, automated testing
  • Validation = "Building the right system?" สร้างระบบได้ตรงตามความต้องการผู้ใช้/ข้อกำหนดทางธุรกิจ (business requirement) หรือไม่ — เกี่ยวข้องกับผู้ใช้โดยตรง เช่น user acceptance testing, usability testing
  • การทดสอบที่เป็นทั้งสองอย่าง: regression testing, system testing, beta testing — โดย validation จะทำหลัง verification เสร็จสมบูรณ์

แบบจำลองวี (V-model): ให้ความสำคัญกับกิจกรรมในการพัฒนาระบบและการทดสอบ "เท่าๆ กัน" โดยมีแผนการทดสอบ (test plan) เชื่อมโยงทุกขั้นตอน ฝั่งซ้ายคือลำดับการพัฒนา (บนลงล่าง) ฝั่งขวาคือการทดสอบ (ล่างขึ้นบน): unit testing → subsystem integration testing → system integration testing → system testing → acceptance testing → operation testing — ข้อดี เข้าใจง่าย เหมาะกับการพัฒนาแบบน้ำตก (waterfall); ข้อจำกัด ไม่รองรับการเปลี่ยนความต้องการบ่อยๆ

12.2 ประเภทของการทดสอบระบบ

(ก) แบ่งตามระดับชั้น/ระยะ (level/phase) มี 6 ประเภท — เป็นหัวข้อสำคัญที่สุด:

  1. การทดสอบหน่วย/โมดูล (unit / module / component testing) — ทดสอบส่วนเล็กสุด คือ คลาส/ฟังก์ชัน ผู้เขียนโปรแกรมมักทดสอบเอง ค้นหา syntax error, สูตรคำนวณผิด, ลำดับการทำงานผิด ทดสอบแต่ละหน่วยอย่างเป็นอิสระ
  2. การทดสอบแบบบูรณาการ (integration testing) — ทดสอบการทำงานร่วมกันระหว่างโมดูล มี 2 ลักษณะ: top-down (เริ่มจากโมดูลบนแล้วเพิ่มย่อย) และ bottom-up (เริ่มจากโมดูลย่อยแล้วรวมขึ้นบน)
  3. การทดสอบฟังก์ชัน (function testing) — ทดสอบหน้าที่ของระบบเทียบกับเอกสารระบุความต้องการฟังก์ชัน (system functional requirement)
  4. การทดสอบประสิทธิภาพ (performance testing) — ทดสอบส่วนที่ไม่ใช่ฟังก์ชัน (non-function) เช่น เวลาประมวลผล จำนวนผู้ใช้สูงสุด ความปลอดภัย
  5. การทดสอบเพื่อการยอมรับ (acceptance testing / UAT) — ขั้นสุดท้ายก่อนผู้ใช้ยอมรับ ผู้ใช้จริงต้องมีส่วนร่วม จบด้วยการลงนามอนุมัติ (UAT sign off); แบ่งย่อยเป็น alpha testing (จำลองสถานการณ์ ทดสอบโดยผู้ใช้ร่วมกับนักพัฒนา) และ beta testing (ใช้ข้อมูลจริงในสภาพแวดล้อมจริง แก้ปัญหาที่พบจาก alpha)
  6. การทดสอบการติดตั้ง (installation testing) — ติดตั้งจริงทั้งฮาร์ดแวร์/ซอฟต์แวร์ในสภาพแวดล้อมที่จะใช้งานจริง

(ข) แบ่งตามหน้าที่การทำงาน มี 2 ประเภท:

  • การทดสอบส่วนของฟังก์ชัน (functional testing) — "What the system does?" ตรวจ validation ของแต่ละฟังก์ชันให้ตรง business requirement เช่น unit, integration, system testing, interface & usability testing, smoke testing (ทดสอบฟังก์ชันพื้นฐานเพื่อตัดสินใจทดสอบต่อ), sanity testing (เมื่อปรับเวอร์ชันใหม่), regression testing, UAT
  • การทดสอบความพร้อมของระบบ (non-functional testing) — "How well the system works?" เน้น performance เช่น performance testing, installation, endurance/stress testing, volume testing, availability & reliability test (กู้คืน recovery ได้), security/penetration testing, spike testing, compatibility testing, network sensitivity testing, scalability testing

(ค) แบ่งตามวิธีการทดสอบ มี 3 วิธี — ออกสอบบ่อยมาก:

  • กล่องดำ (black box testing): ทดสอบเฉพาะ input และ output โดยไม่สนใจการทำงานภายใน เหมาะให้ผู้ใช้ทดสอบ; ข้อดี ผู้ทดสอบไม่ต้องรู้โค้ด, ข้อเสีย ไม่รับประกันว่ากรณีทดสอบครอบคลุมทุกเงื่อนไข (ตัวอย่าง ATM ถอน 100–30,000 บาท ทดสอบ 99 / 25,000 / 30,001)
  • กล่องขาว (white box testing / glass box testing): ทดสอบทั้ง input ส่วนประมวลผลภายใน และ output ผู้ทดสอบต้องรู้การทำงานภายในทั้งหมด ทดสอบตรรกะและการคำนวณ นิยมใช้กราฟควบคุมการทำงาน (Control Flow Graph: CFG); ผู้ทดสอบคือทีมพัฒนา; ข้อดี หาและแก้ข้อผิดพลาดได้เร็ว, ข้อเสีย ต้องใช้ผู้เชี่ยวชาญและเวลา/ค่าใช้จ่ายสูง
  • กล่องเทา (gray box testing): ผสมกล่องดำ+กล่องขาว ทดสอบ input บางส่วนของการประมวลผล และ output ผู้ทดสอบต้องรู้ว่าจะทดสอบประเด็นใด

12.3 เอกสารการทดสอบและกรณีทดสอบ

เอกสารการทดสอบ (ตาม ISO/IEC/IEEE 29119 ส่วนที่ 3 / อ้าง IEEE 829) แบ่ง 3 กลุ่มตามกระบวนการในส่วนที่ 2:

  1. กระบวนการระดับองค์กร — เอกสารนโยบายการทดสอบ (test policy), เอกสารกลยุทธ์การทดสอบ (test strategy)
  2. กระบวนการจัดการการทดสอบ — แผนการทดสอบ (test plan), รายงานสถานะการทดสอบ (test status report), รายงานเมื่อเสร็จสิ้น (test completion)
  3. กระบวนการทดสอบแบบไดนามิก — test design specification, test case specification, test procedure specification, test data requirements, test environment requirement, actual results, test result, test execution log, test incident report

กรณีทดสอบ (test case): เอกสารที่ระบุขั้นตอน เงื่อนไข ชุดข้อมูลป้อนเข้า และผลลัพธ์ที่คาดหวัง ต้องออกแบบให้ครอบคลุมทุกสถานการณ์เพื่อค้นพบข้อบกพร่องมากที่สุด เทคนิคการออกแบบกรณีทดสอบ แบ่งตาม 2 แนวทาง:

  • กล่องดำ 3 เทคนิค: (1) การแบ่งกลุ่มข้อมูลแบบสมมูล (equivalence partitioning) — แบ่ง input/output เป็นกลุ่มที่ให้ผลเหมือนกันแล้วเลือกตัวแทนกลุ่มละ 1 ค่า ทั้งกลุ่มถูกต้อง (valid) และไม่ถูกต้อง (invalid), (2) การวิเคราะห์ค่าข้อมูลขอบ (Boundary Value Analysis: BVA) — เลือกค่าที่จุดขอบและใกล้ขอบ เพราะข้อผิดพลาดมักเกิดที่ขอบเขต (เช่น 99, 100, 101), (3) การจับคู่เหตุและผลลัพธ์ด้วยกราฟ (cause-effect graphing) — ใช้ตรรกศาสตร์เชื่อม input กับ output ด้วยกราฟ
  • กล่องขาว: ใช้แผนภาพกราฟแสดงโครงสร้างการทำงานของโปรแกรมและกำหนดเงื่อนไขในการทดสอบ (criteria/test requirement)

ประเด็นที่ควรเน้นเพื่อสอบ

  • 4 วัตถุประสงค์ของการทดสอบ: ป้องกัน (preventive) → ค้นหา (detective) → แก้ไข (corrective) → เป็นไปตามมาตรฐานคุณภาพ
  • Verification vs Validation: จำประโยคหลัก — Verification = "Building the system right?" (ระหว่างพัฒนา), Validation = "Building the right system?" (เทียบความต้องการผู้ใช้); validation ทำหลัง verification
  • ระดับการทดสอบ 6 ระดับ (ตามระดับชั้น): unit → integration → function → performance → acceptance → installation (ออกสอบบ่อยมาก จำลำดับและความหมาย)
  • black-box vs white-box vs gray-box: กล่องดำดูแค่ input/output (ผู้ใช้ทดสอบ ไม่รู้โค้ด), กล่องขาวรู้ภายในทั้งหมด (ทีมพัฒนา ทดสอบตรรกะ ใช้ CFG), กล่องเทาผสมทั้งสอง — รวมข้อดี/ข้อเสียของแต่ละแบบ
  • integration testing 2 ทิศทาง: top-down (บนลงล่าง) vs bottom-up (ล่างขึ้นบน)
  • acceptance testing: ผู้ใช้จริงต้องร่วมทดสอบ, แบ่งเป็น alpha (จำลองสถานการณ์) และ beta (ข้อมูลจริง/สภาพจริง), จบด้วย UAT sign off
  • V-model: เชื่อมการพัฒนา (ซ้าย บนลงล่าง) กับการทดสอบ (ขวา ล่างขึ้นบน) ด้วย test plan ทุกขั้น — ข้อดี/ข้อจำกัด (เหมาะ waterfall, ไม่เหมาะถ้าความต้องการเปลี่ยนบ่อย)
  • มาตรฐาน ISO/IEC/IEEE 29119: 5 ส่วน และเอกสารทดสอบแบ่ง 3 กลุ่มตามกระบวนการ (องค์กร / จัดการ / ไดนามิก)
  • เทคนิคออกแบบ test case แบบกล่องดำ 3 เทคนิค: equivalence partitioning, BVA (ทดสอบค่าขอบ เช่น 99/100/101), cause-effect graphing
  • เข้าใจการตัดสินผล: เทียบ actual output กับ expected output — ต่างกันคือ fail เท่ากันคือ pass; และคำว่า test stub (โปรแกรมแทนส่วนที่ยังไม่ทดสอบในกระบวนการไดนามิก)

แนวคิดสำคัญ

เคสทูล (CASE tool) เป็นเทคโนโลยีที่เพิ่มความสามารถให้นักวิเคราะห์ระบบ (Systems Analyst: SA) ทำงานได้สะดวกและเป็นอัตโนมัติมากขึ้น โดยมีฟังก์ชันให้เลือกใช้ เช่น ตัวแก้ไขการออกแบบ (Design Editor) พจนานุกรมข้อมูล (Data Dictionary) คอมไพเลอร์ (Compiler) และดีบักเกอร์ (Debugger) จุดสำคัญที่ต้องเข้าใจคือ เคสทูลถูก จำแนกได้หลายมุมมอง: ตามขอบข่าย SDLC (Upper/Lower/Integrated CASE), ตามหน้าที่ (Functional), ตามกระบวนการทำงาน (Process) และตามการประสานกับเครื่องมืออื่น (Integration) นอกจากนี้ยังต้องเข้าใจว่าเคสทูลมีทั้งข้อดี (สื่อสารในทีม สร้างมาตรฐาน ลดเวลา) และข้อเสีย (ค่าใช้จ่ายสูง ต้องฝึกอบรม เจ้าหน้าที่อาจต่อต้าน)

เนื้อหาหลัก

13.1.1 แนวคิดเกี่ยวกับเคสทูล (นิยาม ข้อดี-ข้อเสีย คุณลักษณะ)

นิยาม: เคสทูล (CASE tools) คือ ซอฟต์แวร์ที่สนับสนุนกิจกรรมต่างๆ ของงานวิศวกรรมซอฟต์แวร์ ช่วยสร้างแผนภาพ รายงาน และให้โค้ดโปรแกรม เป็นเครื่องมือหลักของนักวิเคราะห์ระบบ

ข้อดี (สรุปประเด็นสอบ): ช่วยสร้างระบบที่มีประสิทธิภาพและใช้งานได้นาน, ตรงความต้องการผู้ใช้, ยืดหยุ่น, สร้างมาตรฐานสัญลักษณ์และไดอะแกรม, ช่วยสื่อสารในทีมพัฒนา, ตรวจสอบคุณภาพแบบจำลองอัตโนมัติ, ลดเวลาและความยากลำบาก, นำแบบจำลองกลับมาใช้ใหม่ได้

ข้อเสีย: ระบบเริ่มต้นมีค่าใช้จ่ายสูง, ต้องกำหนดความต้องการให้ครอบคลุมถูกต้อง, ต้องฝึกเจ้าหน้าที่, เชื่อมต่อกับระบบเดิมซับซ้อน, ความถูกต้องของการตรวจสอบไม่ได้แปลว่าตรงความต้องการผู้ใช้, มีค่าใช้จ่ายเครื่องมือ, เจ้าหน้าที่อาจต่อต้านการใช้เคสทูล (Staff resistance)

คุณลักษณะที่เคสทูลที่ดีควรมี 6 ข้อ: 1) มีวิธีดำเนินงานเป็นมาตรฐาน (A standard methodology) 2) มีความยืดหยุ่น (Flexibility) 3) มีการบูรณาการที่แข็งแกร่ง (Strong Integration) 4) บูรณาการกับซอฟต์แวร์ทดสอบ (Integration with testing software) 5) สนับสนุนวิศวกรรมย้อนกลับ (Support for reverse engineering) 6) มีการช่วยเหลือออนไลน์ (Online help)

13.1.2 ประเภทของเคสทูล

(1) จำแนกตามขอบข่ายของเครื่องมือสนับสนุนการพัฒนาระบบ (CASE Tool framework) อ้างอิงจาก SDLC แบ่งเป็น

  • อัปเปอร์เคส (Upper-CASE): ช่วยช่วงต้นของ SDLC ได้แก่ ขั้นวางแผน (Planning) วิเคราะห์ (Analysis) และออกแบบ (Design) เก็บข้อมูลการออกแบบ ไดอะแกรม หน้าจอ รายงาน
  • โลเวอร์เคส (Lower-CASE): ช่วยช่วงท้าย ได้แก่ การพัฒนา/นำไปใช้ (Implementation) ทดสอบ (Testing) และบำรุงรักษา (Maintenance) ใช้สร้างโปรแกรม ลดเวลาและความผิดพลาดในการเขียนโค้ด
  • อินทิเกรตเคสทูล (Integrated CASE Tools: I-CASE): รวมทั้ง 2 ระดับเข้าด้วยกัน ครอบคลุมตั้งแต่ความต้องการ (Requirement) จนถึงการทดสอบ (Testing) และการจัดทำเอกสาร (Documentation)

(2) จำแนกตามหน้าที่ (Functional Perspective) เปรียบเสมือนคลังเก็บเครื่องมือ (The Repository) เช่น เครื่องมือการวางแผน (Planning tools เช่น PERT), การแก้ไข (Editing tools), การบริหารการเปลี่ยนแปลง (Change management), การบริหารคอนฟิกูเรชัน (Configuration management), สร้างต้นแบบ (Prototyping), สนับสนุนวิธีการ (Method-support), วิเคราะห์โปรแกรม (Program analysis), ทดสอบระบบ (Testing), ดีบัก (Debugging), จัดทำเอกสาร (Documentation), ประมวลผลภาษา (Language-processing) และรีเอนจิเนียริง (Re-engineering)

(3) จำแนกตามกระบวนการทำงาน (Process Perspective) แบ่งเป็น 8 กลุ่ม: 1) เครื่องมือวิเคราะห์ความต้องการ (Software Requirement Tool) 2) ออกแบบซอฟต์แวร์ (Software Design Tools เช่น Rational Rose, EA) 3) สร้างซอฟต์แวร์ (Software Construction Tools เช่น Eclipse) 4) ทดสอบซอฟต์แวร์ (Software Testing Tools) 5) บำรุงรักษาซอฟต์แวร์ (Software Maintenance Tools) 6) บริหารจัดการโครงแบบ (Software Configuration Management) 7) บริหารงานวิศวกรรมซอฟต์แวร์ (Software Engineering Management) 8) คุณภาพซอฟต์แวร์ (Software Quality Tools)

(4) จำแนกตามการประสานกับเครื่องมืออื่น (Integration Perspective) ครอบคลุมทุกด้าน เช่น เมทะทูล (Meta Tools) ที่เป็นเครื่องมือที่ใช้สร้างเครื่องมืออื่นได้

13.1.3 ตัวอย่างเครื่องมือ (จัดกลุ่มตามหน้าที่ พร้อมตัวอย่างซอฟต์แวร์)

  1. สร้างไดอะแกรม (Diagram Tools): Flow Chart Maker, Draw.io
  2. สร้างแบบจำลองกระบวนการ (Process Modeling): EPF Composer
  3. บริหารจัดการโครงการ (Project Management): Creative Pro Office, Trac, Basecamp, Redmine

  5. วิเคราะห์ (Analysis): Accept 360, CaseComplete, Visible Analyst (ตรวจความไม่สอดคล้อง/ซ้ำซ้อน)
  6. ออกแบบ (Design): Animated Software Design (ใช้เทคนิคแบ่งรายละเอียด refinement)
  7. บริหารจัดการโครงร่าง (Configuration Management): Fossil, Git, AccuREV (จัดการ version/revision, baseline, change control)
  8. ควบคุมการเปลี่ยนแปลง (Change Control)
  9. เขียนโปรแกรม (Programming): เทียบเท่า IDE (Integrated Development Environment) เช่น Eclipse, Cscope
  10. สร้างต้นแบบ (Prototyping): Serena prototype composer, Mockup Builder
  11. พัฒนาเว็บ (Web Development): Adobe Edge Inspect, Brackets, Foundation 3

13.2.1 เคสทูลเพื่อสร้างไดอะแกรม (draw.io)

draw.io เป็น เว็บแอปพลิเคชัน (web-based application) ฟรี เรียกใช้ผ่าน web browser ทันทีโดยไม่ต้องติดตั้ง บันทึกไฟล์ได้ทั้ง Google Drive, OneDrive หรือบนเครื่อง (Device) และบันทึกอัตโนมัติระหว่างทำงาน หมวดหมู่ไอเทม (shape) ที่สำคัญ ได้แก่ General, Misc, Advanced, Basic, Arrows, Flowchart, Entity Relation, UML, และ BPMN (Business Process Model and Notation)

13.2.2 ตัวอย่างการสร้างแผนภาพยูสเคส (Use-case diagram) ด้วย draw.io

ต้องเปิด ห้องสมุดรูปทรงยูเอ็มเอล (UML shape library) ผ่าน More Shapes แล้ว Apply ขั้นตอน: เลือก Actor → สร้างกรอบระบบ (rectangle) → สร้างยูสเคส (วงรี) → โยงเส้นเชื่อม (line) ตัวอย่างคือระบบตู้จำหน่ายตั๋วอัตโนมัติ (Actor: ลูกค้า, เจ้าหน้าที่; Use case: ซื้อตั๋ว ตรวจสอบตั๋ว เพิ่มตั๋ว) และระบบบันทึกนิสัย (habit tracker) ความสัมพันธ์ในยูสเคส: ใช้ include เมื่อการกระทำหนึ่งขึ้นกับอีกการกระทำ, extend เมื่อขยายจากการกระทำอื่น และใช้ multiplicity บอกความสัมพันธ์ระหว่างอีลิเมนต์

13.2.3 ตัวอย่างการสร้างแผนภาพคลาส (Class diagram) ด้วย draw.io

แผนภาพคลาสเป็นรากฐานของการเขียนโปรแกรมเชิงวัตถุ (OOP) คลาส แทนด้วยกล่อง 3 ส่วน: ชื่อคลาส (Class name), คุณลักษณะ (Attribute เขียน name: type), การดำเนินการ (Operations เขียน method name(parameter): return type) สัญลักษณ์การมองเห็น/เข้าถึง: public (+), protected (#), private (-), derived (/), package (~) ความสัมพันธ์ (Associations): 1) สองทาง (Bi-directional, เส้นทึบไม่มีลูกศร) 2) ทางเดียว (One way, ลูกศรโปร่ง) 3) ขึ้นต่อกัน (Dependency, เส้นประ) 4) เป็นส่วนหนึ่ง (Aggregation, เพชรโปร่ง) 5) ส่วนประกอบ (Composition, เพชรทึบ) 6) มัลติพลิซิตี้ (Multiplicity เช่น 0..*, 1..1, 1..5) ใน draw.io เลือกได้ 2 แบบ: Class (3 ส่วนครบ) และ Class 2 (เฉพาะชื่อคลาสกับคุณลักษณะ)

ภาพรวม

หน่วยนี้เป็นกรณีศึกษาที่นำหลักการวิเคราะห์และออกแบบเชิงวัตถุ (OOAD) มาประยุกต์กับ "ระบบจัดการและติดตามการแก้ไขระบบ" (Issue Tracking System: ITS) ซึ่งเป็นระบบที่ใช้ในงานการบำรุงรักษาซอฟต์แวร์หลังส่งมอบ เพื่อเก็บรวบรวมและติดตามสถานะของการร้องขอแก้ไขระบบ (issue) เนื้อหาแบ่งเป็น 2 ตอน คือ ตอนที่ 14.1 แนวคิดของระบบ (การบำรุงรักษา ลักษณะการร้องขอ กระบวนการ บทบาทผู้ใช้ สถานะ และผังงาน) และตอนที่ 14.2 การวิเคราะห์และออกแบบเชิงวัตถุ (เก็บความต้องการ → แบบจำลองโดเมน → ยูสเคส → แผนภาพกิจกรรม/คลาส/ลำดับ)

แนวคิดสำคัญ

หัวใจของการควบคุมคุณภาพซอฟต์แวร์คือการบำรุงรักษาระบบที่มีเสถียรภาพ เพราะซอฟต์แวร์ใดๆ ที่ส่งมอบแล้วย่อมมีการร้องขอแก้ไขเสมอ ระบบ ITS จึงถูกนำมาใช้เป็น ศูนย์กลางจัดเก็บข้อมูล (Central Repository) ของทุกการร้องขอ ช่วยติดตามสถานะ และทำให้บุคคลในแต่ละบทบาททำงานร่วมกันได้ ประเด็นที่ต้องเข้าใจให้ลึกคือ การร้องขอ (issue) แต่ละรายการจะไหลผ่าน กระบวนการ 6 ขั้นตอน ตามบทบาทผู้ใช้ (6 บทบาท) และมี สถานะ (status) เปลี่ยนไปตามผังงาน (workflow) ทั้งที่เกี่ยวข้องและไม่เกี่ยวข้องกับสภาพแวดล้อม (environment) จากนั้นจึงนำข้อมูลทั้งหมดไปวิเคราะห์/ออกแบบเชิงวัตถุด้วยสัญลักษณ์ UML

14.1.1 การบำรุงรักษาระบบ (System Maintenance)

นิยาม: กระบวนการเฝ้าสังเกต ตรวจสอบ ประเมิน และแก้ไขระบบอย่างต่อเนื่องหลังนำขึ้นใช้งาน เพื่อให้ทำงานมีประสิทธิภาพและตรงความต้องการที่เปลี่ยนไปตามเวลา ประเภทการบำรุงรักษา 4 ประเภท (พร้อมสัดส่วนตามภาพที่ 14.1):

  1. Perfective (เพื่อความสมบูรณ์) — 50% เพิ่มฟีเจอร์/ฟังก์ชันให้ใช้ง่ายหรือเร็วขึ้น (มากที่สุด)
  2. Adaptive (เพื่อปรับเปลี่ยน) — 25% ปรับตามการเปลี่ยนกฎเกณฑ์ ข้อบังคับ หรือสภาพแวดล้อม
  3. Corrective (เพื่อความถูกต้อง) — 21% แก้จุดบกพร่อง (bug) ที่เกิดจากการออกแบบ/ระบุความต้องการไม่ชัด
  4. Preventive (เพื่อป้องกัน) — 4% ลดโอกาสเกิดข้อผิดพลาดในอนาคต/เพิ่มขีดความสามารถรองรับผู้ใช้

ความจำเป็นในการมีระบบ ITS (3 เหตุผลหลัก): 1) เป็นศูนย์กลางระบุและบันทึกทุกการร้องขอ (Central Repository) 2) ช่วยติดตามและระบุสถานะปัจจุบันของการร้องขอ 3) ช่วยให้บุคคลแต่ละบทบาทสื่อสารและทำงานร่วมกันได้

14.1.2 ลักษณะของการร้องขอและกระบวนการ

ลักษณะของการร้องขอ (มาจากหลายช่องทาง): รายงานจุดบกพร่อง (Bug report) จาก continuous test, จาก functional test, ปัจจัยภายนอก (ข้อมูลนำเข้าจากระบบอื่นเปลี่ยน), ฝ่ายพัฒนาธุรกิจขององค์กร, และผลตอบรับจากลูกค้า (โทรศัพท์/อีเมล)

กระบวนการจัดการและติดตามการแก้ไขระบบ 6 ขั้นตอน (ภาพที่ 14.2):

  1. การร้องขอ (ผู้รายงานสร้าง/ส่งคำร้องเข้าระบบ — ใช้ TestLink สร้าง Test Plan/Test Case ได้)
  2. ประเมินประเภทและความสำคัญ (bug หรือ feature — ปกติแก้ bug ก่อนเพิ่ม feature)
  3. ประเมินความเป็นไปได้และผลกระทบ (impact)
  4. พัฒนาระบบ (ใช้ version control เช่น TortoiseSVN, Subversion; ทำ code review/peer review ด้วย Review Board)
  5. ทดสอบระบบ
  6. ดีพลอยระบบ (deploy)

สภาพแวดล้อม (environment) สำหรับระบบซับซ้อน แบ่งเป็น 3: สำหรับการพัฒนาระบบ (Development), สำหรับการทดสอบระบบ (Test — ใกล้เคียงสภาพจริง), และสำหรับการใช้งานจริง (Production) การดีพลอยขึ้น production ควรกำหนดเวลาที่กระทบผู้ใช้น้อยที่สุด (เช่น ระบบซื้อขายหุ้นทำหลังตลาดปิด)

ตัวอย่างซอฟต์แวร์ ITS 3 ตัว (สำคัญสอบ):

  • เรดไมน์ (Redmine): โอเพนซอร์ส (open source) ฟรี พัฒนาบน Ruby on Rails ใช้ฐานข้อมูล MySQL/PostgreSQL/SQLite เป็นเว็บแอป มีปลั๊กอินมาก (เช่น Agile plugin แสดงเป็นบอร์ด) แต่แอปบนมือถือยังน้อย
  • จีระ (JIRA): เชิงพาณิชย์ของ Atlassian ไม่ใช่โอเพนซอร์ส เด่นเรื่องบริหารโปรเจ็กต์แบบ Agile สร้าง workflow ด้วยการลากวาง (drag and drop) แบ่งงานเป็นสปรินต์ (sprint) ได้ ตัวอย่างสถานะ: OPEN → IN PROGRESS → RESOLVED → CLOSED → REOPENED
  • เทรลโล (Trello): ผู้พัฒนาเดียวกับ JIRA โปรเจ็กต์ = 1 บอร์ด (board), การร้องขอ = การ์ด (card), สถานะ = รายการ (list); ลากการ์ดข้ามลิสต์เพื่อเปลี่ยนสถานะ

14.1.3 ผังงานของระบบ (บทบาท สถานะ ผังงาน)

บทบาทผู้ใช้งาน 6 บทบาท แบ่งเป็น 3 ทีม:

  • ทีมจัดการการร้องขอ: ผู้รายงาน (Reporter — รวบรวม/เพิ่มการร้องขอ ตรวจสอบและปิด close), ผู้จัดการการร้องขอ (Issue Manager — ประเมินความสำคัญ สร้างโปรเจ็กต์ เพิ่มผู้ใช้ กำหนดบทบาท)
  • ทีมพัฒนาระบบ: หัวหน้านักพัฒนา (Head of Developer — วิเคราะห์ผลกระทบ กระจายงาน), นักพัฒนา (Developer — พัฒนาและระบุ due date)
  • ทีมทดสอบระบบ: หัวหน้านักทดสอบ (Test Manager — วางแผน/กระจายงานทดสอบ), นักทดสอบ (Tester — เขียน Test Case และทดสอบ)

สถานะของการร้องขอ แบ่งเป็น 2 กลุ่ม:

  1. ไม่เกี่ยวข้องกับสภาพแวดล้อม (7 สถานะ): Open, Not Bug, Duplicate, Cancelled, Proceed Dev., Resolved and Closed, Suspended
  2. เกี่ยวข้องกับสภาพแวดล้อม (12 สถานะ ตาม 3 environment):

- Development: Developing, Fixed/Testable, Dev Testing, Dev Passed, Dev Failed

- Test: Developed, Testing, Test Passed, Test Failed

- Production: Deployed, Hot Fix, Resolved

ผังงาน (workflow): เมื่อการร้องขอถูกเปิด (Open) จะถูกส่งต่อตามบทบาท เปลี่ยนสถานะและส่งต่อจนเสร็จและถูกปิด (Closed) จุดที่ออกสอบ: ความต่างของ Not Bug (ระบุผิดว่าเป็น bug), Cancelled (ระบุประเภทถูกแต่ไม่ต้องทำ), Suspended (เริ่มทำแล้วแต่ถูกระงับกลางคัน) และการลดทอนได้ — บทบาทที่ลดได้คือ 3 หัวหน้า/ผู้จัดการ (เหลือ Reporter/Developer/Tester) และสภาพแวดล้อมลดเหลือ 2 (รวม Dev กับ Test)

14.2.1 การเก็บรวบรวมและวิเคราะห์ความต้องการ

ความต้องการเก็บได้จากการสัมภาษณ์และศึกษาระบบเดิม แล้วบันทึกใน เอกสารความต้องการด้านซอฟต์แวร์ (Software Requirement Specification: SRS / Software Requirement Document) อ้างอิงมาตรฐาน IEEE โครงสร้างเอกสาร: 1) บทนำ (Purpose, Scope, Definitions/Acronyms, References, Overview) 2) คำบรรยายโดยรวม (Product Perspective, Product Functions, User Characteristics, Constraints, Assumptions) 3) ข้อกำหนดความต้องการ (System Overview, Functional Requirements, Non-Functional Requirements, ความต้องการเครื่องมือพัฒนา) 4) ภาคผนวก 5) ดัชนี ระบบ ITS ในกรณีศึกษาใช้บำรุงรักษาระบบซื้อขายออนไลน์ (OSS) มีการระบุ จุดประสงค์หลัก และ กฎเกณฑ์ทางธุรกิจ (Business Rule) เช่น แอดมินสร้างยูสเซอร์, ทุกการร้องขอต้องมีสถานะและอยู่ใต้โปรเจ็กต์, ยูสเซอร์ 1 คนมีหลายบทบาทได้

14.2.2 การวิเคราะห์ระบบเชิงวัตถุ

(1) สร้างแบบจำลองโดเมน (Domain Model): เริ่มจากหา conceptual class โดยขีดเส้นใต้คำนาม/นามวลี (noun phrase) จากคำบรรยายระบบ → เลือกคำที่เป็นแนวคิดหลัก (ตัดคำที่เป็น attribute หรือค่า เช่น Bug/Feature) → กำหนด attribute → หา association ระหว่างคลาส คลาสหลักที่ได้: Project, Issue, Project Member, Reporter, Developer, Tester (โดย Reporter/Developer/Tester เป็น generalization ของ Project Member และ Project–Project Member เป็น composition)

(2) แผนภาพยูสเคส (Use-case Diagram):

  • แอ็คเตอร์: เริ่มจาก User (พื้นฐาน), Admin, และที่ไม่ใช่แอดมิน (Reporter, Developer, Tester, Issue Manager, Head of Developer, Test Manager)
  • ความสัมพันธ์: แอ็คเตอร์ User มีความสัมพันธ์แบบ generalization กับแอ็คเตอร์อื่นทั้งหมด; ระบบเล็กใช้ 3 แอ็คเตอร์หลัก ระบบใหญ่เพิ่มแอ็คเตอร์ฝ่ายบริหาร
  • คำอธิบายยูสเคส (Use Case Description): เขียนรายละเอียด ID, Brief description, Primary/Secondary actors, Preconditions, Main flow, Postconditions, Alternative Flows ตัวอย่างหลักคือ NA006 (Create Issue) และ NA009 (Assign Issue)

14.2.3 การออกแบบระบบเชิงวัตถุ

นำผลการวิเคราะห์มาออกแบบ โดยลำดับการสร้างแผนภาพ:

  1. แผนภาพกิจกรรม (Activity Diagram): ใช้แบบ swim lanes เพื่อแสดงว่าขั้นตอนงานใดอยู่ในความรับผิดชอบของใคร/ออบเจ็กต์ใด (ทำสำหรับ NA006, NA009)
  2. แผนภาพคลาส (Class Diagram): วิเคราะห์หาคลาสพร้อม attribute และ method ของระบบ
  3. แผนภาพลำดับ (Sequence Diagram): เป็น แผนภาพปฏิสัมพันธ์ (interaction diagram) ที่เลือกใช้ในกรณีศึกษานี้ แสดงการส่งแมสเสจ (message) ระหว่างคลาสเพื่อให้งานของยูสเคสสำเร็จ (ทำสำหรับ NA006, NA009)

เมื่อการวิเคราะห์/ออกแบบเสร็จ ระบบจะถูกพัฒนา (เขียนซอร์สโค้ด) แล้วเข้าสู่ implement และ deploy ซึ่งต่อเนื่องไปยังหน่วยที่ 15

ภาพรวม

หน่วยนี้เป็นภาคต่อจากกรณีศึกษาในหน่วยที่ 14 โดยกล่าวถึง 2 ส่วนสำคัญก่อนและหลังการพัฒนาระบบ คือ (1) การออกแบบส่วนต่อประสานผู้ใช้ (User Interface Design: UI) เน้นการทำต้นแบบ (prototype) หน้าจอของโปรแกรมประยุกต์บนเว็บ และ (2) สถาปัตยกรรม (architecture) และการดีพลอย (deployment) ระบบขึ้นใช้งานจริง เนื้อหาแบ่งเป็นตอนที่ 15.1 การออกแบบส่วนต่อประสานผู้ใช้ และตอนที่ 15.2 สถาปัตยกรรมและการดีพลอยระบบ พร้อมตัวอย่างจากระบบจัดการและติดตามการแก้ไขระบบตลอดทั้งหน่วย

แนวคิดสำคัญ

ส่วนต่อประสานผู้ใช้ที่ดีต้องออกแบบจากการ "รู้จักและเข้าใจผู้ใช้" (ทักษะ ประสบการณ์ ความคาดหวัง) เพราะผู้ใช้มักตัดสินโปรแกรมจาก UI มากกว่าฟังก์ชันการทำงาน การออกแบบ UI ที่ดีเกิดจากการผสมผสาน 3 ศาสตร์: การออกแบบเชิงภาพ (Visual Design) + การออกแบบปฏิสัมพันธ์เชิงโต้ตอบ (Interactive Design) + สถาปัตยกรรมข้อมูล (Information Architecture) อีกแนวคิดหลักคือ กระบวนการออกแบบ UI เป็นแบบวนซ้ำ (iterative) โดยใช้ "ต้นแบบ" เป็นสื่อกลางสื่อสารกับผู้ใช้และรับผลตอบรับ ส่วนด้านสถาปัตยกรรม เน้น สถาปัตยกรรมทางกายภาพ (physical architecture) ที่อธิบายทั้งซอฟต์แวร์และฮาร์ดแวร์ผ่านแผนภาพดีพลอย (deployment diagram) และวางแผนสภาพแวดล้อมเพื่อการดีพลอยอย่างเป็นระบบ

เนื้อหาหลัก

15.1.1 หลักการและกระบวนการของการออกแบบ UI

นิยาม: การออกแบบส่วนต่อประสานผู้ใช้ (user interface design) คือ กระบวนการออกแบบส่วนต่อประสานของซอฟต์แวร์ที่เน้นภาพลักษณ์ให้ดึงดูดและง่ายต่อการใช้งาน (เป็นได้ทั้งกราฟิก เสียง และท่าทาง)

ข้อจำกัดของมนุษย์ (อ้างอิง Ian Sommerville) 4 ข้อ: 1) จดจำระยะสั้นได้จำกัด (เฉลี่ย ~7 รายการ) 2) คนทำผิดพลาดเป็นเรื่องปกติ 3) แต่ละคนมีความสามารถทางกายภาพต่างกัน 4) แต่ละคนพึงพอใจการโต้ตอบต่างกัน (รูปภาพ vs ข้อความ)

แนวคิด 3 ส่วนในการออกแบบ UI บนเว็บ: Visual Design (สี แบบอักษร ภาพ), Interactive Design (ออกแบบจากพฤติกรรมเชิงบวกของผู้ใช้), Information Architecture (จัดโครงสร้างข้อมูล/เมนูให้ค้นหาง่าย)

หลักการพื้นฐานของ UI (Ian Sommerville) 6 ข้อ (สำคัญสอบ):

  1. ความคุ้นเคยของผู้ใช้ (User familiarity) — ใช้คำ/ส่วนประกอบที่เป็นมาตรฐานและคุ้นเคย
  2. ความคงที่สม่ำเสมอ (Consistency) — ใช้คำสั่ง/เมนูเหมือนกันทั้งระบบ
  3. การควบคุมความรู้สึกแปลกใจ (Minimal surprise) — เลือกส่วนประกอบให้คาดเดาได้ (เช่น checkbox = เลือกหลายรายการ, radio button = เลือกได้รายการเดียว)
  4. ความสามารถในการกู้คืน (Recoverability) — มีปุ่มย้อนกลับ/ยกเลิก/ยืนยัน และกู้ข้อมูลได้เมื่อระบบขัดข้อง
  5. คำแนะนำแก่ผู้ใช้ (User guidance) — มีคำแนะนำเมื่อพบข้อผิดพลาด เช่น คู่มือออนไลน์
  6. ความหลากหลายของผู้ใช้ (User diversity) — รองรับผู้ใช้หลากหลาย เช่น ปุ่มขยายหน้าจอสำหรับผู้มีปัญหาสายตา

กระบวนการออกแบบ UI 8 ขั้นตอน (ภาพที่ 15.1, แบบวนซ้ำ): 1) วิเคราะห์/เข้าใจกิจกรรมผู้ใช้ → 2) ทำต้นแบบกระดาษ (paper-based) → 3) ประเมินต้นแบบกับผู้ใช้ → 4) แก้ต้นแบบกระดาษ → 5) ออกแบบต้นแบบฉบับร่างที่ 1 (design prototype) → 6) ประเมินและทำต้นแบบไดนามิก → 7) ได้ executable prototype → 8) พัฒนา UI จริงตามต้นแบบ

ส่วนประกอบ UI บนเว็บ 4 หมวดหมู่ (ออกสอบบ่อย):

  1. กลุ่มควบคุมอินพุต (Input controls): checkbox, radio button, dropdown list, list box, button, dropdown button, toggle, text field, date/time picker
  2. กลุ่มการนำทาง (Navigation components): search field, breadcrumb (แทบแสดงลำดับการเชื่อมโยง), pagination (ทำเลขหน้า), tag, slider, icon, image carousel
  3. กลุ่มแสดงข้อมูล (Informational components): notification, progress bar, tool tip, message box, modal window/pop-up
  4. กลุ่มคอนเทนเนอร์ (Containers): accordion (พื้นที่เปิดปิด)

การเลือกสีและตัวอักษร:

  • แสงและเงา: สร้างความลึก เช่น ปุ่มเข้มขึ้นเมื่อถูกกด (วัตถุ 3 มิติ)
  • สี — กฎ 60-30-10: 60% สีหลัก (กลาง สบายตา), 30% สีรอง (ตรงข้ามสีหลัก), 10% สีเน้น (สด/เด่น ใช้กับปุ่มสำคัญ เช่น ดาวน์โหลด ซื้อ)
  • ตัวอักษร 4 กลุ่ม: นำทาง, พาดหัว, รองพาดหัว, เนื้อหา; ตัวอักษรไม่มีหัว (Sans Serif) สำหรับหัวข้อ/นำทาง, มีหัว (Serif) สำหรับเนื้อหายาว; ไม่ควรใช้เกิน 2 แบบต่อโปรแกรม

ข้อควรคำนึง (Do): เรียบง่ายดีที่สุด, คงที่สม่ำเสมอ, เลย์เอาท์สื่อจุดประสงค์, สี/ตัวอักษรคุ้นตา, ลายเส้นชัดเจน, ใช้ค่าเริ่มต้น (default) ลดภาระผู้ใช้

ข้อควรหลีกเลี่ยง (Don't): ไม่ออกแบบตามความต้องการผู้ใช้, ไม่เรียนรู้กลุ่มเป้าหมาย, ใช้ลูกเล่นมากเกินไป, ไม่สำรวจเพิ่มเติม, ลงรายละเอียดมากเกินไปในช่วงแรก

15.1.2 การจัดทำต้นแบบ (Prototype) — 3 ประเภท

  1. โครงร่าง (Wireframe): ต้นแบบระดับเริ่มต้น แสดงภาพรวมและเลย์เอาท์คร่าวๆ ยังไม่ลงรายละเอียดกราฟิก เป็นได้ทั้งแบบคงที่ (วาดบนกระดาษ) และโต้ตอบได้ ใช้สื่อสารให้เห็นภาพรวมตรงกัน
  2. แบบจำลอง (Mockup): ลงรายละเอียดภาพลักษณ์มากขึ้น (สี ตัวอักษร ขนาด ภาพ ข้อความ) อยู่กึ่งกลางระหว่าง wireframe กับ clickable prototype ใช้นำเสนอผลลัพธ์หน้าตาและรับคำติชม
  3. แบบจำลองผลลัพธ์แบบโต้ตอบได้ (Clickable Prototyping): ต้นแบบสุดท้ายที่ใกล้เคียงระบบจริงมากที่สุด กดปุ่ม/ใส่อินพุต/โต้ตอบได้เหมือนใช้งานจริง

ตัวอย่างจากกรณีศึกษา (ITS): ยกตัวอย่าง 4 หน้าจอที่เชื่อมโยงกัน — Login, หน้าหลัก (My issue), สร้างการร้องขอ (Create), แก้ไขข้อมูลส่วนตัว (Profile) ใช้เครื่องมือ draw.io ช่วยสร้าง mockup; clickable prototype มีการตรวจสอบรูปแบบข้อมูล เช่น แจ้งเตือนอีเมลขาด "@" และแสดงสัญลักษณ์ OK สีเขียวเมื่อกรอกถูกต้อง

15.2.1 สถาปัตยกรรม (Architecture)

3 มุมมองเชิงสถาปัตยกรรมในงานไอที: ตรรกะ (logical — ความสัมพันธ์/การขึ้นต่อกันของคลาส), การดำเนินงาน (implementation — ใช้แผนภาพองค์ประกอบ), กายภาพ (physical — ซอฟต์แวร์ + ฮาร์ดแวร์) หน่วยนี้เน้น สถาปัตยกรรมทางกายภาพ (physical architecture) ซึ่งใช้แผนภาพองค์ประกอบ (component diagram) และแผนภาพดีพลอย (deployment diagram) อธิบายสภาพแวดล้อมการทำงานจริง

สถาปัตยกรรมทางกายภาพของ ITS (ภาพที่ 15.21): ใช้สัญลักษณ์ <<deploys>> แสดงว่าองค์ประกอบซอฟต์แวร์ (HTML/JSP form, Controller, Model) ทำงานบน Web server ซึ่งมี Security และ DB interface; เชื่อม Database Server (Oracle DB) ผ่าน LAN; เครื่องผู้ใช้ (Work station) ใช้ web browser เชื่อมผ่าน HTTP/HTTPS

องค์ประกอบของสถาปัตยกรรมทางกายภาพ 2 ส่วน (สำคัญสอบ):

  • ซอฟต์แวร์คอมโพเนนท์ (Software Component): การจัดเก็บข้อมูล (Data Storage), ตรรกะการเข้าถึงข้อมูล (Data Access Logic เช่น ODBC, OLEDB), ตรรกะของแอปพลิเคชัน (Application Logic เช่น Java), ตรรกะของการนำเสนอ (Presentation Logic เช่น dropdown, pop-up)
  • ฮาร์ดแวร์คอมโพเนนท์ (Hardware Component): เครื่องคอมพิวเตอร์ลูกข่าย (Client computer), เครื่องเซิร์ฟเวอร์ (Servers/Storage Server), อุปกรณ์เชื่อมต่อเครือข่าย (Connecting Network เช่น Router, Firewall, Switch, Hub)

15.2.2 การนำระบบขึ้นใช้งาน (Deployment)

ข้อควรปฏิบัติในการดีพลอย (8 ข้อ): 1) ทำเช็คลิสต์ดีพลอย (deployment checklist) 2) เลือกเครื่องมือเหมาะสม 3) ใช้ระบบ CI (รวมงานต่อเนื่อง + ทดสอบ) 4) ใช้ CD (ส่งมอบต่อเนื่อง เตรียมแพ็คเกจ) 5) ดีพลอยแบบอัตโนมัติ (ลดความเสี่ยง) 6) พิจารณาการดีพลอยแบบต่อเนื่อง (ต้องผ่านทดสอบก่อนขึ้น production เสมอ) 7) ตรวจสอบประสิทธิภาพเสมอ (เซิร์ฟเวอร์ error rate log database) 8) เตรียมพร้อมทำย้อนกลับ (rollback) เสมอ — เครื่องมือเช่น Jenkins, TeamCity, Octopus Deploy, Automic

สภาพแวดล้อมเพื่อการดีพลอย 4 แบบ (อย่างน้อย — สำคัญสอบ):

  1. สภาพแวดล้อมพัฒนา (Develop environment)
  2. สภาพแวดล้อมทดสอบ (Testing environment)
  3. สภาพแวดล้อมจัดเตรียม (Staging environment)
  4. สภาพแวดล้อมผลผลิต (Production environment)

ขั้นตอนดีพลอย (ภาพที่ 15.22 — Build Pipeline + Release Pipeline): นักพัฒนา build/test ที่ Local → ดึงและ merge ซอร์สโค้ดล่าสุดจาก repository → CI build แต่ละชิ้นแล้ว run unit test → CD ดีพลอยไปสภาพแวดล้อมพัฒนา → trigger รัน automated test (เทสเคสย่อยเฉพาะ feature) → ถ้าผ่านส่งต่อสภาพแวดล้อมจัดเตรียม รัน regression test + acceptance test (QA) → เมื่อทุกอย่างผ่านและฝ่ายธุรกิจอนุมัติ จึงดีพลอยขึ้น production

ผลลัพธ์การดีพลอยตามกรณีศึกษา ITS (ภาพที่ 15.23): แผนภาพดีพลอยระดับสเปก (specification level) แสดงชิ้นส่วน (deployed artifact) เช่น executable files, libraries, database schema, configuration files วางอยู่ในโนด (node) และเชื่อมผ่านเส้นทางสื่อสาร (communication path) ตัวอย่าง: เว็บแอปจาวารันบน JSP Server (Tomcat 7) ใช้ Catalina Servlet; artifact = ISSUE_APP.war, web-tools-lib.jar, user_service.jar กำหนดค่าใน web.xml; แอปเซิร์ฟเวอร์ = Sun Fire X4150; เชื่อม Database Server (Sun SPARC + Oracle 10g, schema Users/Issues/Role) ด้วย TCP/IP

สัญลักษณ์ความสัมพันธ์ในแผนภาพคลาส (ออกสอบบ่อย)

ความสัมพันธ์สัญลักษณ์ / ความหมาย
Associationเส้นตรงเชื่อมคลาส ระบุ multiplicity ปลายเส้น
Aggregationข้าวหลามตัด "โปร่ง" (ว่าง) ที่ฝั่งส่วนรวม — ส่วนย่อยอยู่เองได้
Compositionข้าวหลามตัด "ทึบ" (ดำ) ที่ฝั่งส่วนรวม — ลบส่วนรวมแล้วส่วนย่อยหายตาม
Generalization (Inheritance)สามเหลี่ยม "โปร่ง" ชี้ไปคลาสแม่ (is a kind of)
Dependencyเส้นประหัวลูกศรเปิด (พึ่งพาชั่วคราว)
Multiplicity1 (หนึ่งเท่านั้น), 0..1 (มีหรือไม่มี), 0..* (ศูนย์ถึงหลาย), 1..* (อย่างน้อยหนึ่ง)

คู่ที่มักสับสน (ต้องแยกให้ออก)

คู่จุดต่างที่ต้องจำ
OOA กับ OODOOA เน้น "ทำอะไร" (What/วิเคราะห์ปัญหา); OOD เน้น "ทำอย่างไร" (How/ออกแบบทางแก้)
เชิงโครงสร้าง กับ เชิงวัตถุเชิงโครงสร้างเน้นฟังก์ชัน/ขั้นตอน (แยกข้อมูลกับฟังก์ชัน); เชิงวัตถุรวมข้อมูล+พฤติกรรมในวัตถุ
วัตถุ (Object) กับ คลาส (Class)คลาสคือแม่แบบ; วัตถุคือสิ่งที่สร้างจริงจากคลาส (instance)
Aggregation กับ CompositionAggregation แยกออกได้ (โปร่ง); Composition แยกไม่ได้ ส่วนย่อยตายตามส่วนรวม (ทึบ)
include กับ extendinclude = เรียกใช้ทุกครั้งเพื่อ reuse (ลูกศรชี้ยูสเคสที่ถูกรวม); extend = เรียกบางกรณี (ลูกศรชี้ยูสเคสหลัก)
Sequence กับ CommunicationSequence เน้นลำดับ "เวลา" (บนลงล่าง); Communication เน้น "ความเชื่อมโยง" กำกับด้วยเลขลำดับ
State กับ Activity DiagramState แสดงสถานะของวัตถุและการเปลี่ยนสถานะ; Activity แสดงการไหลของกิจกรรม/งาน (มี fork/join, swimlane)
Component กับ DeploymentComponent = ส่วนประกอบ "ซอฟต์แวร์"; Deployment = การติดตั้งบน "ฮาร์ดแวร์" (Node)
Verification กับ ValidationVerification = "สร้างถูกวิธีไหม"; Validation = "สร้างถูกระบบตามที่ต้องการไหม"
Black-box กับ White-boxBlack-box ทดสอบโดยไม่รู้โครงสร้างภายใน (เน้นอินพุต-เอาต์พุต); White-box รู้โครงสร้าง/โค้ดภายใน
Alpha กับ Beta TestingAlpha ทดสอบโดยทีม/ในสภาพจำลอง; Beta ทดสอบโดยผู้ใช้จริงในสภาพแวดล้อมจริง

รวมศัพท์เชิงวัตถุ (OO) และ UML ที่พบบ่อยตลอดทั้งชุดวิชา ใช้ท่องทบทวนก่อนสอบ (เรียงตามตัวอักษร)

ตัวย่อ / ศัพท์ความหมาย
Abstraction (นามธรรม)การดึงเฉพาะสาระสำคัญของสิ่งหนึ่งมาเป็นคลาส
Activity Diagramแผนภาพกิจกรรม แสดงการไหลของงาน (workflow) มี fork/join และ swimlane
Actor (แอ็กเตอร์)ผู้หรือสิ่งที่โต้ตอบกับระบบในแผนภาพยูสเคส (สัญลักษณ์รูปกระดูกคน)
Aggregationความสัมพันธ์ "เป็นส่วนหนึ่งของ" แบบแยกออกได้ (สัญลักษณ์ข้าวหลามตัดโปร่ง)
Associationความสัมพันธ์ทั่วไประหว่างคลาส "เกี่ยวข้องกัน" (เส้นตรง + multiplicity)
Attribute (แอตทริบิวต์)คุณลักษณะ/ข้อมูลของคลาส
BVABoundary Value Analysis — การวิเคราะห์ค่าขอบเขตในการออกแบบกรณีทดสอบ
CASE ToolComputer-Aided Software Engineering — เครื่องมือช่วยงานวิศวกรรมซอฟต์แวร์ (แบ่ง Upper/Lower/Integrated)
Class (คลาส)พิมพ์เขียว/แม่แบบของวัตถุ กำหนดแอตทริบิวต์และเมท็อด
Class Diagramแผนภาพคลาส แสดงคลาสและความสัมพันธ์ระหว่างคลาส (เชิงโครงสร้าง)
Classificationนามธรรมแบบจัดหมวดวัตถุเข้าเป็นคลาส "is a member of" (สำคัญที่สุด)
Communication Diagramแผนภาพการสื่อสาร เน้นความเชื่อมโยงของวัตถุ กำกับลำดับด้วยเลข
Component Diagramแผนภาพคอมโพเนนต์ แสดงส่วนประกอบ "ซอฟต์แวร์"
Compositionความสัมพันธ์ "เป็นส่วนประกอบ" แบบแยกไม่ได้ (ข้าวหลามตัดทึบ; ลบส่วนรวมแล้วส่วนย่อยหายด้วย)
Dependencyความสัมพันธ์แบบพึ่งพา (เส้นประหัวลูกศร)
Deployment Diagramแผนภาพดีพลอยเมนต์ แสดงการติดตั้งบน "ฮาร์ดแวร์" (Node เป็นลูกบาศก์)
Encapsulation (การห่อหุ้ม)รวมข้อมูล+พฤติกรรมไว้ด้วยกันและซ่อนรายละเอียดภายใน
extend (<<extend>>)ยูสเคสที่ทำงาน "บางกรณี" (ทางเลือก) ลูกศรชี้ไปยังยูสเคสหลัก (base)
Generalizationความสัมพันธ์แบบสืบทอด "is a kind of" (สามเหลี่ยมโปร่ง ชี้คลาสแม่)
include (<<include>>)ยูสเคสที่ถูกเรียกใช้ "ทุกครั้ง" เพื่อใช้ซ้ำ (reuse) ลูกศรชี้ไปยูสเคสที่ถูกรวม
Inheritance (การสืบทอด)คลาสลูกรับคุณสมบัติจากคลาสแม่
Instance / Instantiationการสร้างวัตถุจริงจากคลาส (วัตถุ = instance ของคลาส)
Interaction Diagramแผนภาพปฏิสัมพันธ์ (รวม Sequence และ Communication)
JAD / RADJoint/Rapid Application Development — เทคนิคเก็บความต้องการ/พัฒนาแบบเร่งรัด
Lifelineเส้นอายุของวัตถุในแผนภาพลำดับ (Sequence)
Message (ข้อความ)การส่งคำสั่ง/เรียกเมท็อดระหว่างวัตถุ (Synchronous/Asynchronous/Return)
Method / Operationเมท็อด/โอเปอเรชัน — พฤติกรรม/ฟังก์ชันการทำงานของคลาส
Multiplicity (พหุคูณ)จำนวนความสัมพันธ์ เช่น 1, 0..1, 0..*, 1..*, 2..4
Object (วัตถุ)สิ่งที่มีทั้งข้อมูล (attribute) และพฤติกรรม (method) เป็น instance ของคลาส

OOA / OOD / OOP / OOTวิเคราะห์ / ออกแบบ / เขียนโปรแกรม / ทดสอบ เชิงวัตถุ
Polymorphism (พหุสัณฐาน)ชื่อเมท็อดเดียวกันทำงานต่างกันได้ตามวัตถุ
Problem Domainโดเมนปัญหา — ขอบเขตของปัญหาที่ระบบจะแก้
Sequence Diagramแผนภาพลำดับ เน้น "ลำดับเวลา" ของข้อความ (lifeline + activation)

Stakeholderผู้มีส่วนได้ส่วนเสียกับระบบ (กว้างกว่า User)
State Machine Diagramแผนภาพสถานะ แสดงสถานะของวัตถุและการเปลี่ยนสถานะ (transition/event)
