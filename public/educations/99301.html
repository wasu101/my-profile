ภาพรวมชุดวิชา 99301 การบริการผ่านเว็บ

ชุดวิชานี้เกี่ยวกับอะไร

ชุดวิชา 99301 "การบริการผ่านเว็บ" (Web Services) ว่าด้วยการออกแบบและพัฒนาโปรแกรมประยุกต์ที่ให้บริการบนเว็บ ซึ่งทำให้ระบบที่อยู่ต่างแพลตฟอร์ม ต่างภาษา และต่างองค์กร สามารถสื่อสารและทำงานร่วมกันได้ (Interoperability) โดยอาศัยภาษากลางคือ เอ็กซ์เอ็มแอล (XML) และมาตรฐานเปิดชุดหนึ่ง หัวใจของวิชาคือเทคโนโลยีหลัก 3 ตัวที่ทำงานประสานกัน ได้แก่ เอสโอเอพี (SOAP), ดับเบิ้ลยูเอสดีแอล (WSDL) และยูดีดีไอ (UDDI) บนพื้นฐานของสถาปัตยกรรมเชิงบริการ (SOA)

ภาพใหญ่: เทคโนโลยีหลักทำงานร่วมกันอย่างไร

ให้จำเป็น "ชั้น" ของมาตรฐานที่ซ้อนกัน โดยแต่ละตัวมีหน้าที่ชัดเจน

  • XML คือ ภาษากลางที่ใช้บรรยายโครงสร้างข้อมูล เป็นรากฐานที่ SOAP, WSDL และ UDDI ต่างสร้างอยู่บนมัน
  • SOAP คือ วิธี "ห่อ" และรับส่งข้อความระหว่างบริการ (เปรียบเหมือนซองจดหมาย) ส่งผ่านโพรโทคอลขนส่งเช่น HTTP
  • WSDL คือ "คู่มือ" ที่อธิบายว่าบริการทำอะไรได้บ้างและเรียกใช้อย่างไร (อินพุต/เอาต์พุต/ที่อยู่)
  • UDDI คือ "ทะเบียน/สมุดหน้าเหลือง" ที่ผู้ให้บริการนำบริการไปประกาศ และผู้ใช้เข้ามาค้นหา

ลำดับการทำงาน (วงจรของเว็บเซอร์วิส) ที่ต้องจำ คือ ผู้ให้บริการสร้างบริการและเขียน WSDL แล้วนำไปประกาศ (Publish) ไว้ที่ UDDI จากนั้นผู้ขอใช้บริการค้นหา (Find) ใน UDDI จนได้ WSDL กลับมา ศึกษา WSDL เพื่อรู้วิธีเรียกใช้ แล้วส่งข้อความ SOAP ไปเรียกใช้บริการโดยตรง (Bind/Invoke) สุดท้ายผู้ให้บริการประมวลผลและส่งผลลัพธ์กลับเป็น SOAP

องค์ประกอบ 3 ฝ่ายของเว็บเซอร์วิส

  1. ผู้ให้บริการ (Service Provider) สร้าง ติดตั้ง และประกาศบริการ
  2. ผู้ขอใช้บริการ (Service Requestor) ค้นหาและเรียกใช้บริการ
  3. ตัวแทน/ทะเบียนบริการ (Service Registry / Broker) เป็นตัวกลางรับลงทะเบียนและให้ค้นหา (ใช้ UDDI + WSDL)

  • กลุ่มที่ 2 ภาษาและมาตรฐานหลัก (หน่วย 4–8): ภาษามาร์กอัปและสคริปต์ (หน่วย 4), พื้นฐาน XML (หน่วย 5), SOAP (หน่วย 6), WSDL (หน่วย 7), UDDI (หน่วย 8) — กลุ่มนี้คือ "หัวใจเชิงเทคนิค" ของวิชา
  • กลุ่มที่ 3 การพัฒนาและการดูแลระบบ (หน่วย 9–12): เทคโนโลยีฐานข้อมูล (หน่วย 9), การพัฒนาและใช้งาน (หน่วย 10), การรักษาความปลอดภัย (หน่วย 11), นโยบายและการจัดการ (หน่วย 12)

วิธีใช้เอกสารนี้ให้ได้ผลก่อนสอบ

อ่านภาพรวมนี้ให้เห็น "ภาพใหญ่" ก่อน แล้วค่อยลงรายละเอียดทีละหน่วย แต่ละหน่วยจัดเป็น 5 ส่วนเหมือนกันหมด คือ ภาพรวม → แนวคิดสำคัญ → เนื้อหาหลัก → คำศัพท์และตัวย่อ → ประเด็นที่ควรเน้นเพื่อสอบ เมื่ออ่านครบแล้วให้ย้อนมาทบทวนที่ "อภิธานศัพท์รวม" และ "ชีตทบทวนด่วนก่อนสอบ" ท้ายเล่ม ซึ่งรวบรวมตัวย่อและข้อเท็จจริงที่ออกสอบบ่อยไว้ให้ท่องซ้ำ

หน่วยที่ 1: ความรู้เบื้องต้นเกี่ยวกับการบริการผ่านเว็บ

ภาพรวม

หน่วยนี้เป็นพื้นฐานของทั้งชุดวิชา ปูความเข้าใจว่า "การบริการผ่านเว็บ" หรือ เว็บเซอร์วิส (Web Service) คืออะไร เกิดขึ้นมาได้อย่างไร มีองค์ประกอบและเทคโนโลยีหลักอะไรบ้าง และเชื่อมโยงกับแนวคิดการประมวลผลแบบเครือข่ายและสถาปัตยกรรมเชิงบริการ (SOA) อย่างไร เนื้อหาแบ่งเป็น 3 ตอน คือ (1.1) ความหมาย องค์ประกอบ เทคโนโลยี (1.2) การประมวลผลแบบเครือข่ายและ SOA และ (1.3) โพรโทคอล ภาษา และเว็บเทคโนโลยีที่เกี่ยวข้อง

แนวคิดสำคัญ

  • เว็บเซอร์วิส คือ โปรแกรมประยุกต์ (แอปพลิเคชัน) ที่ออกแบบมาเพื่อให้บริการบนเว็บ โดยเป็นซอฟต์แวร์คอมโพเนนต์ (Software Component) ที่รับอินพุตเข้ามาประมวลผลแล้วส่งผลลัพธ์กลับออกไป สามารถถูกเรียกใช้จากแอปพลิเคชันอื่นผ่านเครือข่ายได้
  • จุดเด่นคือทำให้ โปรแกรมที่อยู่ต่างแพลตฟอร์ม/ต่างภาษา สื่อสารกันได้ โดยใช้ภาษากลางคือ เอ็กซ์เอ็มแอล (XML) ส่งผ่านโพรโทคอลมาตรฐาน
  • การบริการผ่านเว็บเป็นรูปแบบหนึ่งของ สถาปัตยกรรมเชิงบริการ (SOA) และเป็นพัฒนาการต่อยอดจากการประมวลผลแบบกระจาย (Distributed Computing) และเว็บแอปพลิเคชัน

เนื้อหาหลัก

1.1 ความหมาย องค์ประกอบ เทคโนโลยีและประโยชน์

ความหมาย: โดยทั่วไปเว็บแสดงผลด้วยภาษา เอชทีเอ็มแอล (HTML) ซึ่งโต้ตอบกับผู้ใช้ได้จำกัด จึงเกิดแนวคิดสร้างแอปพลิเคชันที่ทำงานในรูปของ "บริการ" และใช้ภาษา เอ็กซ์เอ็มแอล (XML) อธิบายโครงสร้างข้อมูลที่รับส่งระหว่างแอปพลิเคชัน โดยอาศัย โพรโทคอลเอชทีทีพี (HTTP) ขนส่งข้อมูล สรุปว่า เว็บเซอร์วิส = ฟังก์ชันที่ฝังไว้ที่เว็บเซิร์ฟเวอร์ให้ไคลเอ็นต์หรือแอปพลิเคชันอื่นเรียกใช้ได้

ยุคของการพัฒนาเว็บ มี 3 ยุค (ออกสอบบ่อย):

  1. ยุคสแตติกเว็บ (Static Web) — นำเสนอข้อมูลคงที่ ไม่โต้ตอบ ใช้ HTML, JavaScript, VBScript, Java Applet
  2. ยุคไดนามิกเว็บ (Dynamic Web) — โต้ตอบกับผู้ใช้ ใช้สคริปต์ฝั่งเซิร์ฟเวอร์ เช่น CGI, ASP, PHP, JSP
  3. ยุคเว็บเซอร์วิส (Web Services) — เรียกใช้บริการบนเว็บข้ามกันได้ ใช้ .NET, J2EE, nuSOAP, Axis, Glue

องค์ประกอบของเว็บเซอร์วิส มี 3 ส่วน (สำคัญมาก):

  1. ผู้ให้บริการ (Service Provider) — พัฒนา ติดตั้ง และประกาศ/ลงทะเบียนเซอร์วิส
  2. ผู้ขอใช้บริการ (Service Requestor) — ค้นหาและเรียกใช้ (bind) บริการ
  3. ตัวแทนของผู้ให้บริการ (Service Registry / Service Broker) — ตัวกลางรับลงทะเบียน ใช้ UDDI บอกรายละเอียดบริษัท และ WSDL อธิบายวิธีเชื่อมต่อ

เทคโนโลยีของเว็บเซอร์วิส มี 4 ส่วน: แอปพลิเคชัน (Application), เอสโอเอพี (SOAP), ยูดีดีไอ (UDDI), และดับเบิ้ลยูเอสดีแอล (WSDL) — เทคโนโลยีพื้นฐานทั้งหมดคือ XML, HTTP, SOAP, WSDL, UDDI

ขั้นตอนการทำงานของเว็บเซอร์วิส: ผู้ให้บริการสร้าง WSDL ประกาศไว้ที่ UDDI → ผู้ใช้สอบถาม UDDI → ได้ WSDL กลับมา → ศึกษา WSDL แล้วส่งข้อความเรียกใช้บริการโดยตรง → โปรแกรมประมวลผลและส่งผลลัพธ์กลับ

1.2 การประมวลผลแบบเครือข่ายและสถาปัตยกรรมเชิงบริการ

พัฒนาการของการประมวลผลแบบเครือข่าย 3 ระยะ:

  1. การประมวลผลที่ศูนย์กลาง (Centralized Processing) ค.ศ. 1950-1980 — ประมวลผลที่เครื่องหลักเครื่องเดียว (เมนเฟรม) พีซีเป็นเพียงดัมพ์เทอร์มินัล (dump terminal)
  2. การประมวลผลแบบไคลเอ็นต์-เซิร์ฟเวอร์ (Client-Server) ค.ศ. 1980-1990 — ไคลเอ็นต์ร้องขอบริการ เซิร์ฟเวอร์ประมวลผลและให้บริการ
  3. การประมวลผลแบบกระจาย (Distributed Processing) ตั้งแต่ ค.ศ. 1990 — กระจายภาระไปยังหลายเครื่องแล้วรวมผลลัพธ์ ใช้การประมวลผลแบบขนาน (Parallel Processing)

สถาปัตยกรรมเชิงบริการ (SOA): เป็นหลักการออกแบบซอฟต์แวร์ให้แอปพลิเคชันทำงานร่วมกันได้โดยไม่ขึ้นกับแพลตฟอร์ม/ภาษา/เทคโนโลยี เว็บเซอร์วิสเป็น "วิธีการหนึ่ง" ในการพัฒนาตามหลัก SOA (ไม่ใช่สิ่งเดียวกัน) แนวคิดคือเปลี่ยนจาก Silo-Oriented Architecture (แต่ละระบบเป็นอิสระ เชื่อมต่อยาก ค่าใช้จ่ายสูง) มาเป็นระบบ 4 ชั้น:

  1. Resource Layer — โครงสร้างพื้นฐานไอที (ฐานข้อมูล Oracle, SAP)
  2. Service Layer — ส่วนประกอบเซอร์วิสที่นำกลับมาใช้ใหม่ได้
  3. Process Layer — กระบวนการทางธุรกิจ
  4. Access Layer — การเรียกใช้ผ่านเว็บไซต์/มือถือ

ประโยชน์ของ SOA 4 ข้อ: เชื่อมโยงระบบไอทีที่ต่างกันได้, ปรับเปลี่ยนได้ง่าย, ลดค่าบำรุงรักษาและคุ้มค่าการลงทุน, ฝ่ายธุรกิจและฝ่ายไอทีทำงานสอดคล้องกันมากขึ้น

1.3 โพรโทคอล ภาษา และเว็บเทคโนโลยีที่เกี่ยวข้อง

เอสโอเอพี (SOAP): ระเบียบวิธีการสื่อสารระหว่างเว็บเซอร์วิส โดยใช้ข้อมูลรูปแบบ XML ทำให้สื่อสารข้ามแพลตฟอร์ม/ภาษาได้ และไม่ขึ้นกับ Transport Protocol ใด โครงสร้าง SOAP มี 3 ส่วน (ออกสอบ):

  • SOAP Envelope — เนื้อหาทั้งหมดของเอกสาร (เปรียบเหมือนซองจดหมาย)
  • SOAP Header — ส่วนอธิบายเอกสาร (กำหนดหรือไม่ก็ได้)
  • SOAP Body — ส่วนอธิบายวิธีเรียกใช้บริการและผลลัพธ์

ยูดีดีไอ (UDDI): ไดเร็กทอรีที่เก็บรวบรวมเว็บเซอร์วิสที่ลงทะเบียนไว้ (เก็บในรูปแบบ WSDL) เปรียบเสมือน สมุดโทรศัพท์หน้าเหลือง บทบาทหน้าที่ของ UDDI ได้แก่ การลงทะเบียน (Public/Private Registry), การประกาศบริการ, การสร้างและปรับปรุงบริการ, การลบบริการ (ต้องใช้ UUID), และการค้นหาบริการ (Information Browsing / Information Drilldown)

ดับเบิ้ลยูเอสดีแอล (WSDL): ภาษา XML ที่อธิบายคุณลักษณะการเรียกใช้บริการและวิธีติดต่อกับเว็บเซอร์วิส เปรียบเสมือน "คู่มือ" ให้ระบบเรียนรู้วิธีเรียกใช้งาน ทำให้การเรียกใช้เป็นอัตโนมัติ

เว็บเทคโนโลยีที่เกี่ยวข้อง: เทคโนโลยีเว็บเซิร์ฟเวอร์ เช่น อาปาเช่ (Apache บนลีนุกซ์) และ ไอไอเอส (IIS ของไมโครซอฟต์) รวมถึงเทคโนโลยีสังคมออนไลน์ เช่น สังคมเครือข่าย บล็อก (Blog) Web 2.0 และแนวโน้ม Web 3.0

ประเด็นที่ควรเน้นเพื่อสอบ

  • นิยามเว็บเซอร์วิส และเหตุผลที่ทำให้โปรแกรมต่างแพลตฟอร์ม/ต่างภาษาสื่อสารกันได้ (เพราะใช้ XML และมาตรฐานเปิด)
  • ยุคการพัฒนาเว็บ 3 ยุค พร้อมเทคโนโลยีประจำยุค
  • องค์ประกอบ 3 ส่วน ของเว็บเซอร์วิส และหน้าที่ของแต่ละส่วน (Provider, Requestor, Registry/Broker)
  • เทคโนโลยีพื้นฐาน 4 ส่วน (Application, SOAP, UDDI, WSDL) และเทคโนโลยีพื้นฐานทั้งหมด (XML, HTTP, SOAP, WSDL, UDDI)
  • โครงสร้าง SOAP 3 ส่วน (Envelope, Header, Body) และการเปรียบเทียบกับการส่งจดหมาย
  • บทบาทหน้าที่ของ UDDI และการเปรียบ UDDI = สมุดหน้าเหลือง
  • หน้าที่ของ WSDL = คู่มืออธิบายวิธีเรียกใช้บริการ
  • พัฒนาการการประมวลผล 3 ระยะ พร้อมช่วงปี (Centralized 1950-1980, Client-Server 1980-1990, Distributed ตั้งแต่ 1990)
  • SOA vs เว็บเซอร์วิส — SOA เป็นแนวคิด/หลักการออกแบบ ส่วนเว็บเซอร์วิสเป็นวิธีการพัฒนาตามหลัก SOA
  • SOA 4 ชั้น และการเปลี่ยนจาก Silo-Oriented เป็น Service-Oriented พร้อมประโยชน์ 4 ข้อ
  • ขั้นตอนการทำงานของเว็บเซอร์วิส (Publish ที่ UDDI → ค้นหา → ได้ WSDL → เรียกใช้)

หน่วยที่ 2: โครงสร้างพื้นฐานของการบริการผ่านเว็บ

ภาพรวม

หน่วยนี้ต่อยอดจากหน่วยที่ 1 โดยเจาะลึกว่าการบริการผ่านเว็บ "เกิดขึ้นมาเพราะอะไร" (ที่มาของแนวคิด) "มีโครงสร้างและสถาปัตยกรรมอย่างไร" และ "มีมาตรฐานและองค์กรใดกำกับดูแลบ้าง" เนื้อหาแบ่งเป็น 3 ตอน คือ (2.1) แนวคิดพื้นฐานและวิวัฒนาการ (2.2) โครงสร้างสถาปัตยกรรม และ (2.3) มาตรฐานการบริการผ่านเว็บ ถือเป็นหน่วยที่เน้นความเข้าใจเชิงโครงสร้างและการจำองค์กร/มาตรฐาน

แนวคิดสำคัญ

  • การบริการผ่านเว็บเกิดจากความต้องการแก้ 2 ปัญหาหลัก คือ ปัญหาการพัฒนาระบบงานในองค์กร และปัญหาการประสานการทำงานร่วมกัน (Interoperability) ในการแลกเปลี่ยนข้อมูลระหว่างองค์กร
  • คุณลักษณะของเว็บเซอร์วิสได้รับอิทธิพลจากแนวคิด การเขียนโปรแกรมเชิงอ็อบเจกต์ (OOP) เช่น การซ่อนรายละเอียด และความเป็นอิสระต่อกันของคอมโพเนนต์ (Loosely Coupled)
  • การทำงานยึดตาม ตัวแบบการบริการผ่านเว็บ (Web Service Model) ตามหลัก SOA ที่มีผู้เล่น 3 ฝ่าย และมีมาตรฐานเปิดกำกับ โดยมีองค์กรหลักที่ดูแลมาตรฐาน

เนื้อหาหลัก

2.1 แนวคิดพื้นฐานและวิวัฒนาการของการบริการผ่านเว็บ

ที่มาของแนวคิด — 2 ปัญหา (สำคัญมาก):

  1. ปัญหาการพัฒนาระบบงานในองค์กร — ต้องพัฒนาหลายระบบที่ใช้ร่วมกันไม่ได้ ลงทุนสูง บำรุงรักษายาก
  2. ปัญหาการประสานการทำงานร่วมกัน (Interoperability) — แต่ละองค์กรใช้ภาษา/แอปพลิเคชัน/แพลตฟอร์ม/ระบบปฏิบัติการต่างกัน ทำให้แลกเปลี่ยนข้อมูลยาก

แนวทางแก้ปัญหา:

  • แก้ปัญหาที่ 1 ด้วย ซอฟต์แวร์ในรูปของบริการ (SaaS) — ใช้ซอฟต์แวร์เป็นบริการผ่านเครือข่าย จ่ายตามการใช้งาน (Pay per use) ไม่ต้องลงทุนซื้อขาด ต้นกำเนิดมาจาก ASP (Application Service Provision)
  • แก้ปัญหาที่ 2 ด้วย มาตรฐานกลาง คือ การบริการผ่านเว็บ ที่ใช้ XML เป็นภาษากลาง

พัฒนาการ: การบริการผ่านเว็บพัฒนามาจาก (1) เทคโนโลยีการประมวลผลแบบกระจาย ที่มีมิดเดิลแวร์หลายแบบ เช่น RMI, CORBA, DCOM (แต่ไม่มีตัวใดเด่นจริงและต่างแพลตฟอร์มกัน) และ (2) เว็บแอปพลิเคชัน

2.2 โครงสร้างสถาปัตยกรรมการบริการผ่านเว็บ

คุณลักษณะของการบริการผ่านเว็บ 6 ข้อ (มาจากแนวคิด OOP):

  1. การซ่อนรายละเอียด (Encapsulated)
  2. การแก้ไขโดยไม่ส่งผลกระทบเป็นลูกโซ่
  3. การรับรู้พารามิเตอร์ได้เอง (input/output parameter)
  4. ความเป็นมาตรฐาน (รากฐานจาก XML)
  5. มีคำอธิบายในตัวเอง (Self-Defining)
  6. สนับสนุนการค้นหาและเรียกใช้แบบไดนามิก (Dynamic Discovery and Invocation) หรือ จัสต์อินไทม์ (JIT)

ตัวแบบการบริการผ่านเว็บ (Web Service Model) มีองค์ประกอบ 3 ฝ่าย:

  1. ผู้ขอใช้บริการ (Service Requester) — ค้นหาบริการจาก UDDI หรือติดต่อผู้ให้บริการโดยตรง
  2. ผู้ให้บริการ (Service Provider) — เปิดบริการรองรับการขอใช้
  3. ตัวแทนของผู้ให้บริการ (Service Broker / Service Registry) — ตัวกลางรับลงทะเบียน ใช้ WSDL บอกรายละเอียด

ขั้นตอนการทำงาน 3 ขั้น: (1) ผู้ให้บริการประกาศ (Publish) บริการด้วย WSDL ไปลงทะเบียน (2) ตัวแทนรวบรวมไว้เป็นศูนย์กลางสืบค้น (มาตรฐานคือ UDDI ริเริ่มโดย Ariba, Microsoft, IBM) (3) ผู้ใช้ค้นหา พบแล้วติดต่อผู้ให้บริการโดยตรง

เหตุการณ์ (Event) ในสถาปัตยกรรม: การสร้างบริการ (Create) → การประกาศ (Publish) → การค้นหา (Find) → การอ้างอิง (Reference) → การโยง (Bind) → การเรียกใช้ (Invoke)

คอมโพเนนต์ 4 ตัวตามมาตรฐานสถาปัตยกรรม: (1) XML — ภาษามาร์กอัปส่งข้อมูลข้ามเครือข่าย (2) SOAP — โพรโทคอลส่งผ่านข้อมูล (3) WSDL — เอกสารอธิบายการทำงาน (4) UDDI — ฐานข้อมูลทะเบียนบริการ

เลเยอร์ของการบริการผ่านเว็บ 2 เลเยอร์:

  1. ลิสเทนเนอร์เลเยอร์ (Listener Layer) — เลเยอร์บนสุด ติดต่อกับไคลเอ็นต์ แปลคำร้องขอให้โพรไวเดอร์เข้าใจ
  2. บิสซิเนสเลเยอร์ (Business Layer) — เขียนตรรกะการทำงาน แบ่งย่อยเป็น Business Logic Layer และ Business Facade Layer

2.3 มาตรฐานการบริการผ่านเว็บ

แหล่งที่มาของมาตรฐาน 3 ประเภท (ออกสอบ):

  1. De facto Standards — มาตรฐานจากการใช้งานจริง ได้รับการยอมรับโดยไม่ต้องมีองค์กรรับรอง
  2. De jure Standards — มาตรฐานที่เป็นบทบัญญัติลายลักษณ์อักษร กำหนดโดยองค์กรที่เป็นนิติบุคคล (เช่น IEEE, EIA)
  3. Specifications/Consortium recommendations — ข้อกำหนด/ข้อเสนอแนะของกลุ่มบริษัทหรือสมาคม (เช่น IBM + Microsoft + Ariba จัดทำ UDDI)

องค์กรหลักที่กำกับมาตรฐานการบริการผ่านเว็บ 4 องค์กร (สำคัญมาก):

  1. โอเอซิส (OASIS) — Organization for the Advancement of Structured Information Standards องค์กรไม่หวังผลกำไร พัฒนามาตรฐานการบริการผ่านเว็บมากที่สุดในโลก ก่อตั้งปี พ.ศ. 2536 ภายใต้ชื่อ SGML Open และเปลี่ยนชื่อเป็น OASIS ปี พ.ศ. 2541
  2. ดับบลิวสามซี (W3C) — World Wide Web Consortium องค์กรไม่หวังผลกำไรระดับนานาชาติ พัฒนาเทคโนโลยีเว็บ ก่อตั้งปี พ.ศ. 2537 โดย Tim Berners-Lee มีเครือข่าย 3 ภูมิภาค (MIT, INRIA, Keio University) มาตรฐานสำคัญ เช่น HTML 4.0, XML 1.0, XHTML, CSS, DOM
  3. ดับบลิวเอส-ไอ (WS-I) — Web Services Interoperability ก่อตั้งปี พ.ศ. 2545 โดยกลุ่มบริษัทใหญ่ (Microsoft, IBM, Accenture, BEA, Fujitsu, HP, Intel, Oracle, SAP) มุ่งส่งเสริม interoperability ข้ามแพลตฟอร์ม จัดทำ Basic Profile ต่อมากลายเป็นภาคีสมาชิกของ OASIS
  4. ไออีทีเอฟ (IETF) — Internet Engineering Task Force องค์กรพัฒนาและส่งเสริมมาตรฐานอินเทอร์เน็ต

องค์กรมาตรฐานอื่นที่เกี่ยวข้อง: ANSI, IEEE, ISO, EIA, ITU, IEC

มาตรฐานพื้นฐาน 4 ตัวขององค์ประกอบการบริการผ่านเว็บ (มาตรฐานเปิด):

  1. XML — พื้นฐานของภาษากลางในการแลกเปลี่ยนข้อมูล
  2. SOAP — คอมโพเนนต์กำหนดรูปแบบการรับส่งข้อความ
  3. WSDL — ภาษาอธิบายข้อกำหนดของบริการ (มี WSDL Definition Document)
  4. UDDI — มาตรฐานทะเบียนชุดบริการให้ค้นหาได้

  • ตัวแบบการบริการผ่านเว็บ (3 ฝ่าย) และขั้นตอนการทำงาน 3 ขั้น
  • เหตุการณ์ (Event) 6 ขั้น: Create → Publish → Find → Reference → Bind → Invoke
  • คอมโพเนนต์ 4 ตัว (XML, SOAP, WSDL, UDDI) ตามมาตรฐานสถาปัตยกรรม
  • เลเยอร์ 2 ชั้น (Listener Layer, Business Layer)
  • แหล่งที่มาของมาตรฐาน 3 ประเภท (De facto, De jure, Consortium)
  • องค์กรมาตรฐาน 4 องค์กร (OASIS, W3C, WS-I, IETF) พร้อมจุดเด่นและปีก่อตั้ง — OASIS พัฒนามาตรฐานมากที่สุด, W3C ก่อตั้งโดย Tim Berners-Lee
  • มาตรฐานพื้นฐาน 4 ตัว (XML, SOAP, WSDL, UDDI) ที่เป็นมาตรฐานเปิด

หน่วยที่ 3: แบบจำลองเครือข่ายและโพรโทคอลกับการบริการผ่านเว็บ

ภาพรวม

หน่วยนี้ปูพื้นฐานด้านเครือข่ายซึ่งเป็นรากฐานที่ทำให้การบริการผ่านเว็บทำงานได้ โดยอธิบาย "แบบจำลองเครือข่าย" 2 แบบที่เป็นมาตรฐานการสื่อสาร และ "โพรโทคอล" ต่างๆ ที่ใช้รับส่งข้อมูลบนอินเทอร์เน็ตและบนเว็บเซอร์วิส เนื้อหาแบ่งเป็น 2 ตอน คือ (3.1) แบบจำลองเครือข่าย และ (3.2) โพรโทคอลกับการบริการผ่านเว็บ ถือเป็นหน่วยที่เน้นการจำลำดับชั้น จำนวนเลเยอร์ หมายเลขพอร์ต และหน้าที่ของโพรโทคอลแต่ละตัว

แนวคิดสำคัญ

  • แบบจำลองเครือข่าย คือมาตรฐานกฎระเบียบที่ทำให้คอมพิวเตอร์ต่างแพลตฟอร์ม/ต่างผู้ผลิตสื่อสารกันได้ โดยแบ่งการทำงานเป็นลำดับชั้น (layer) และกำหนดหน้าที่แต่ละชั้นชัดเจน
  • โพรโทคอล คือกฎเกณฑ์/ข้อกำหนดที่ควบคุมการเชื่อมต่อสื่อสารระหว่างคอมพิวเตอร์ต้นทางและปลายทาง หากไม่มีโพรโทคอลการสื่อสารระหว่างคอมพิวเตอร์จะเกิดขึ้นไม่ได้
  • การบริการผ่านเว็บอาศัยทั้งโพรโทคอลพื้นฐานของอินเทอร์เน็ต (HTTP, FTP, SMTP ฯลฯ) และโพรโทคอล/ภาษาเฉพาะของเว็บเซอร์วิส (SOAP, WSDL, UDDI)

เนื้อหาหลัก

3.1 แบบจำลองเครือข่าย

แบบจำลองโอเอสไอ (OSI Model):

พัฒนาโดยองค์การ ไอเอสโอ (ISO) เป็นระบบเครือข่ายแบบเปิด ทำให้อุปกรณ์ติดต่อกันได้โดยไม่ขึ้นกับผู้ผลิต แบ่งเป็น 7 ลำดับชั้น (เลเยอร์) — ต้องจำลำดับและหน้าที่ (ออกสอบแน่):

ชั้นชื่อเลเยอร์หน้าที่
7แอปพลิเคชัน (Application)ติดต่อกับผู้ใช้ ใช้งานได้ทันทีโดยไม่ต้องเขียนโปรแกรมเพิ่ม
6นำเสนอ (Presentation)กำหนดรูปแบบ/ความหมายข้อมูล เข้ารหัส-ถอดรหัส (ASCII, Unicode)
5ส่วนงาน (Session)ควบคุมการเชื่อมต่อต้นทาง-ปลายทาง สร้าง/จัดการเซสชัน
4ขนส่ง (Transport)แบ่งข้อมูลเป็น เซกเมนต์ (Segment) รับประกันการส่ง มี error recovery
3เครือข่าย (Network)แบ่งข้อมูลเป็น แพ็กเก็ต (Packet) กำหนดเลขที่อยู่
2เชื่อมโยงข้อมูล (Data Link)จัดข้อมูลเป็น เฟรม (Frame) ใส่ MAC Address และ Trailer
1กายภาพ (Physical)ควบคุมการรับส่งระดับฮาร์ดแวร์ แปลงเป็น บิต (bit) สัญญาณไฟฟ้า

หลักการส่งผ่านข้อมูล: ใช้สถาปัตยกรรมลำดับชั้น แต่ละชั้นติดต่อเฉพาะชั้นที่อยู่ติดกัน (เช่น ชั้น 6 ติดต่อชั้น 5 และ 7 เท่านั้น) เมื่อส่งข้อมูลจะเริ่มจากชั้น 7 ลงมาชั้น 1 โดยแต่ละชั้นจะใส่ ส่วนหัว (Header) เช่น AH (ชั้น 7), PH (ชั้น 6), SH (ชั้น 5) เพิ่มเข้าไป (เรียกว่าการห่อหุ้มข้อมูล)

แบบจำลองอินเทอร์เน็ต (ทีซีพี/ไอพี — TCP/IP Model):

เป็นชุดโพรโทคอลที่พัฒนาเพื่อแลกเปลี่ยนข้อมูลในยุคเริ่มแรก จุดเด่นคือแก้ปัญหาเครือข่ายได้เองโดยอัตโนมัติเมื่อเส้นทางถูกตัดขาด แบ่งเป็น 4 ลำดับชั้น:

ชั้นชื่อเลเยอร์หน้าที่
4แอปพลิเคชัน (Application)ติดต่อกับผู้ใช้ ให้บริการต่างๆ บนอินเทอร์เน็ต
3ทรานสปอร์ต (Transport)รับส่งข้อมูลระหว่างปลายด้านส่งและด้านรับ
2อินเทอร์เน็ต (Internet)ค้นหา/เลือกเส้นทาง แบ่งข้อมูลเป็นแพ็กเก็ต (packet-switching, connectionless)
1โฮสต์ทูเน็ตเวิร์ก (Host to Network / Link)ควบคุมการรับส่งระดับฮาร์ดแวร์

โพรโทคอลในชั้นทรานสปอร์ตของ TCP/IP มี 2 ชนิด (ออกสอบ — เปรียบเทียบ):

  • ทีซีพี (TCP) — Transmission Control Protocol แบบ connection-oriented ส่งแบบ Byte stream ที่ไว้ใจได้ ไม่มีข้อผิดพลาด มีการควบคุมการไหลของข้อมูล
  • ยูดีพี (UDP) — User Datagram Protocol แบบ connectionless ไม่ยืนยันการรับส่ง แต่รวดเร็ว เหมาะกับ client/server แบบถาม/ตอบ และการส่งภาพเคลื่อนไหว/เสียง

บริการในชั้นแอปพลิเคชันของ TCP/IP: เทลเน็ต (Telnet — เข้าใช้เครื่องระยะไกล), อีเมล (e-mail), สนทนาบนเครือข่าย (Live messaging เช่น Skype, MSN), ถ่ายโอนแฟ้มข้อมูล (FTP — มี Download/Upload)

เปรียบเทียบ OSI กับ TCP/IP (สำคัญ):

  • OSI มี 7 ชั้น (ละเอียดกว่า), TCP/IP มี 4 ชั้น — ทั้งสองมีวัตถุประสงค์คล้ายกันและแต่ละชั้นทำงานอิสระ
  • OSI แยกส่วนบริการกับโพรโทคอลชัดเจน แต่ TCP/IP ไม่แยกชัดเจน (ให้บริการได้แค่ IP แพ็กเกจ + รับ IP แพ็กเกจ)
  • OSI ออกแบบดี/สมบูรณ์กว่า แต่กลับไม่นิยม เพราะถูกสร้างหลัง TCP/IP ที่องค์กรลงทุนไปแล้ว และ OSI ซับซ้อน อ้างอิงเทคโนโลยีมากเกินไป; TCP/IP ยืดหยุ่นกว่าจึงถูกใช้งานต่อเนื่อง

3.2 โพรโทคอลกับการบริการผ่านเว็บ

แบ่งโพรโทคอลตามหน้าที่เป็น 2 ส่วน คือ โพรโทคอลสำหรับใช้งานอินเทอร์เน็ต และโพรโทคอลสำหรับการบริการผ่านเว็บ

1) โพรโทคอลสำหรับใช้งานอินเทอร์เน็ต (ต้องจำชื่อ + หน้าที่ + หมายเลขพอร์ต):

  • เอชทีทีพี (HTTP) — HyperText Transfer Protocol รับส่งข้อมูล (ข้อความ ภาพ เสียง) ระหว่างเบราว์เซอร์กับเว็บเซิร์ฟเวอร์ ผ่าน พอร์ต 80 ทำงานแบบไคลเอ็นต์-เซิร์ฟเวอร์ (request/response) พัฒนาโดย W3C และ IETF (RFC 2616, HTTP/1.1)
  • เอฟทีพี (FTP) — File Transfer Protocol ถ่ายโอนไฟล์ทุกฟอร์แมต ใช้ 2 ช่องทาง: พอร์ต 20 (รับส่งข้อมูล) และพอร์ต 21 (ส่งคำสั่งควบคุม) มี 3 วิธีส่ง: สตรีม (stream), บล็อค (block), บีบอัด (compression)
  • เอสเอ็มทีพี (SMTP) — Simple Mail Transfer Protocol ส่งอีเมลระหว่างโฮสต์กับโฮสต์ (เมลเซิร์ฟเวอร์) ผ่าน พอร์ต 25 ใช้ตัวแทนถ่ายโอนจดหมาย (MTA) ส่งได้เฉพาะข้อความ (text)
  • พีโอพี (POP) — Post Office Protocol ดึงอีเมลจากเมลเซิร์ฟเวอร์มาอ่านที่เครื่องผู้รับ ผ่าน พอร์ต 110 รุ่นนิยมคือ POP3 (ข้อจำกัด: ส่งรหัสผ่านเป็นข้อความ ไม่ปลอดภัย)
  • ไอแม็พ (IMAP) — Internet Mail Access Protocol เกิดหลัง POP เพื่อแก้ข้อจำกัด อนุญาตให้จัดการอีเมล (ลบ/สร้างโฟลเดอร์/ย้าย) บนเซิร์ฟเวอร์จากเครื่องใดก็ได้ ทำงานซับซ้อนกว่า POP3

2) โพรโทคอลสำหรับการบริการผ่านเว็บ (เว็บเซอร์วิสทำงานร่วมกับภาษา XML):

  • เอสโอเอพี (SOAP) — Simple Object Access Protocol โพรโทคอลพื้นฐานที่สำคัญที่สุดของเว็บเซอร์วิส กำหนดรูปแบบการติดต่อด้วยไวยากรณ์ XML แลกเปลี่ยนข้อมูลในสภาพแวดล้อมแบบกระจาย (decentralized) ยืดหยุ่น ใช้ทรานสปอร์ตโพรโทคอลใดก็ได้ (HTTP, SMTP, FTP) ขั้นตอน: ผู้ขอใช้สร้าง SOAP message → ส่งไปผู้ให้บริการ → แปลและประมวลผล → ตอบกลับ → ผู้ขอแปลและใช้ผลลัพธ์
  • ดับเบิลยูเอสดีแอล (WSDL) — Web Services Description Language ภาษา XML ที่อธิบายคุณลักษณะการให้บริการ (รองรับ distributed system แบบ IDL) เป็นเสมือนการเชื่อมต่อระหว่างผู้ร้องขอกับผู้ให้บริการ อยู่ในการดูแลของ W3C เวอร์ชันที่ใช้คือ WSDL 1.1
  • ยูดีดีไอ (UDDI) — Universal Description, Discovery and Integration ข้อกำหนดที่รองรับการจัดการข้อมูลเว็บเซอร์วิส เปรียบเสมือน เว็บไดเร็กทอรี / สมุดหน้าเหลือง / นายหน้าจัดหาบริการ (Service Broker) เป็นตัวกลางระหว่างผู้ขอกับผู้ให้บริการ ลงทะเบียนได้ 2 แบบ: Public registry (ศูนย์กลางสาธารณะ) และ Private registry (เฉพาะกลุ่ม)

ประโยชน์ของ UDDI: ค้นหาบริษัท/ธุรกิจที่ต้องการได้, ระบุข้อมูลบริการเก็บไว้, เข้าถึงกลุ่มลูกค้า/เพิ่มพันธมิตร, ดำเนินธุรกิจอัตโนมัติ, ลดต้นทุนการพัฒนาระบบ

ประเด็นที่ควรเน้นเพื่อสอบ

  • OSI มี 7 ชั้น — จำชื่อ ลำดับ และหน้าที่ (โดยเฉพาะชั้น 1 กายภาพควบคุมฮาร์ดแวร์/บิต, ชั้น 4 ขนส่ง=เซกเมนต์)
  • หน่วยข้อมูลแต่ละชั้น: บิต (Physical) → เฟรม (Data Link) → แพ็กเก็ต (Network) → เซกเมนต์ (Transport)
  • TCP/IP มี 4 ชั้น — จำชื่อและการเทียบกับ OSI (แอปพลิเคชันของ TCP/IP 1 ชั้น = หลายชั้นบนของ OSI)
  • TCP vs UDP — connection-oriented/ไว้ใจได้ vs connectionless/รวดเร็ว
  • เหตุผลที่ OSI ดีกว่าแต่ไม่นิยม ขณะที่ TCP/IP นิยมกว่า (มาก่อน + ยืดหยุ่น)
  • หมายเลขพอร์ตของโพรโทคอล (ออกสอบบ่อยมาก): HTTP=80, FTP=20/21, SMTP=25, POP=110
  • หน้าที่ของโพรโทคอลอีเมล 3 ตัว: SMTP=ส่ง, POP=ดึงมาอ่านที่เครื่อง, IMAP=จัดการบนเซิร์ฟเวอร์
  • 3 วิธีส่งข้อมูลของ FTP: stream, block, compression
  • SOAP เป็นโพรโทคอลหลักของเว็บเซอร์วิส ใช้ XML และใช้ทรานสปอร์ตโพรโทคอลใดก็ได้
  • UDDI = สมุดหน้าเหลือง/Service Broker ลงทะเบียน 2 แบบ (Public/Private)
  • WSDL อยู่ในการดูแลของ W3C เวอร์ชัน WSDL 1.1
  • การแบ่งโพรโทคอลเป็น 2 กลุ่ม: สำหรับอินเทอร์เน็ต (HTTP/FTP/SMTP/POP/IMAP) และสำหรับเว็บเซอร์วิส (SOAP/WSDL/UDDI)

หน่วยที่ 4: ภาษามาร์กอัปและภาษาสคริปต์ (Markup and Script Languages)

ภาพรวม

หน่วยนี้ปูพื้นฐานเครื่องมือสำคัญในการสร้างเว็บและการบริการผ่านเว็บ แบ่งเป็น 2 ตอนหลักคือ ภาษามาร์กอัป (Markup Language) ที่ใช้กำหนดรูปแบบการนำเสนอและจัดเก็บข้อมูลบนเว็บ และภาษาสคริปต์ (Script Language) ที่ทำให้เว็บเพจโต้ตอบกับผู้ใช้และส่งผ่านข้อมูลได้ เนื้อหาเน้นหลักการ โครงสร้างไวยากรณ์ ข้อดี-ข้อจำกัด และการเลือกใช้แต่ละภาษาให้เหมาะกับงาน โดยมาร์กอัปที่ต้องรู้ได้แก่ HTML, XHTML และ XML ส่วนสคริปต์แบ่งตามฝั่งที่ประมวลผล (ไคลเอ็นต์/เซิร์ฟเวอร์)

แนวคิดสำคัญ

  • เครื่องมือสร้างเว็บมี 3 กลุ่ม: (1) ภาษามาร์กอัปและภาษาสคริปต์ ซึ่งผู้ใช้ต้องมีความรู้การเขียนโปรแกรม (2) โปรแกรมสำเร็จรูป เช่น ดรีมวีฟเวอร์ (Dreamweaver) เว็บเมทริกซ์ (WebMatrix) ที่ช่วยผู้ขาดความรู้โปรแกรม (3) ระบบจัดการเนื้อหาเว็บ หรือ CMS (Content Management System) เช่น Mambo, Joomla, PHP-Nuke, Thainoke ซึ่งเป็นเว็บแอปพลิเคชันที่มีโปรแกรมสร้างหน้าเว็บด้วยสคริปต์ + โปรแกรมฐานข้อมูล
  • หากต้องการสร้าง "การบริการผ่านเว็บ" (Web Service) อย่างจริงจัง การใช้ CMS/โปรแกรมสำเร็จรูปอาจไม่พอ ต้องเขียนมาร์กอัปและสคริปต์เองได้
  • ประเภทเว็บเพจ: สแตติก (Static Web Page) มีแต่เนื้อหา สร้างง่าย เรียกเร็ว แต่โต้ตอบผู้ใช้ไม่ได้ vs ไดนามิก (Dynamic Web Page) สร้างจากมาร์กอัป + สคริปต์ ยืดหยุ่นและโต้ตอบผู้ใช้ได้

เนื้อหาหลัก

4.1 ภาษามาร์กอัป (Markup Language)

ภาษามาร์กอัปเป็นกลุ่มภาษาพื้นฐานที่ใช้สร้างเว็บเพจ มีโครงสร้างการเขียนชัดเจน อาศัยไฟล์เท็กซ์ (text file) ช่วยระบุส่วนเนื้อหาและจัดรูปแบบการนำเสนอ

วิวัฒนาการ: GML (Generalized Markup Language ของ IBM) → SGML (Standard Generalized Markup Language ภาษาแม่ที่ทรงพลังแต่ซับซ้อนเกินจึงไม่นิยม) → HTML และ XML ที่ดึงข้อดีของ SGML มาลดความซับซ้อน

ขั้นตอนสร้างเว็บด้วยมาร์กอัป: (1) สร้างไฟล์มาร์กอัป (เช่นใน Notepad) (2) บันทึกนามสกุล .html, .xml หรือ .xhtml (3) เปิดด้วยเว็บเบราว์เซอร์

เอชทีเอ็มแอล (HTML – HyperText Markup Language)

  • ภาษามาร์กอัปมาตรฐานที่นิยมที่สุด พัฒนาโดย Tim Berners-Lee ปัจจุบันควบคุมมาตรฐานโดย W3C (World Wide Web Consortium) มี 3 สถาบันหลัก MIT (สหรัฐ), INRIA (ยุโรป), Keio (ญี่ปุ่น)
  • คำสั่งอยู่ในเครื่องหมายวงเล็บสามเหลี่ยม < > เรียกว่า "แท็ก" (Tag) ใช้ตัวพิมพ์ใหญ่หรือเล็กได้ผลเหมือนกัน (<html> = <HTML>)
  • พัฒนาตั้งแต่เวอร์ชัน 1.0 ถึง 4 (นิยม) และเวอร์ชัน 5 (HTML5) เปิดตัว 22 ม.ค. 2551 จุดเด่น HTML5: การจัดเก็บอย่างมีระเบียบ (Semantic Markup), แท็กวิดีโอเฉพาะ, เก็บไฟล์ออฟไลน์ระดับฐานข้อมูล, กราฟิกไม่ต้องลงปลั๊กอินเพิ่ม, จัดการ input types
  • องค์ประกอบของแท็ก: แท็ก (Tag) = ชุดคำสั่ง, แอตทริบิวต์ (Attribute) = ส่วนขยาย/ระบุคุณสมบัติเพิ่มเติมใน Tag เปิด คั่นด้วยช่องว่างเมื่อมีหลายตัว
  • ประเภทแท็ก: แท็กเดี่ยว (เปิดแล้วไม่ต้องปิด) เช่น <BR>, <P> และ แท็กคู่ (ต้องมีปิดด้วย slash /) สัญลักษณ์ < > ... </> เช่น <B>ตัวหนา</B>, <Head>...</Head>
  • โครงสร้างมาตรฐาน HTML:

<HTML> ... </HTML> (บอกว่าเป็นเอกสาร HTML)

<HEAD> ... </HEAD> (กำหนดส่วนหัว มักมี TITLE)

<TITLE>...</TITLE> (ชื่อเว็บเพจ แสดงบน Title Bar)

<BODY> ... </BODY> (เนื้อหาที่แสดงในเบราว์เซอร์)

ตัวอย่างแอตทริบิวต์: <hr noshade size="15" width="30%">

  • ข้อจำกัด HTML: ไม่แสดงข้อความ error (ต้องดูผลลัพธ์เอง), เบราว์เซอร์แสดงเฉพาะคำสั่งที่ไม่ผิด, ขาดความยืดหยุ่นและไม่เป็นมาตรฐาน (แท็กใช้ตัวใหญ่/เล็กก็ได้)
  • ตรวจสอบความถูกต้องได้ที่ web page validation (w3schools)

เอ็กซ์เอชทีเอ็มแอล (XHTML – eXtensible HyperText Markup Language)

  • พัฒนาโดยรวมความสามารถของ HTML + โครงสร้างไวยากรณ์ของ XML เพื่อให้สมบูรณ์และทำงานได้ทุกแพลตฟอร์ม มุ่งทดแทน HTML 4.01
  • ความเคร่งครัดที่เพิ่มจาก HTML: ต้องเปิด-ปิดแท็กเป็นคู่สอดคล้องจากนอกสู่ใน ห้ามซ้อนเหลื่อม, ทุกแท็กต้องเป็นตัวพิมพ์เล็ก, มี <html>...</html> เพียงชุดเดียว, ตรวจสอบ error ได้ (เพราะมีพื้นฐาน XML), ใช้ CSS ได้
  • 2 ส่วนสำคัญที่เพิ่ม: (1) ส่วนประกาศประเภทเอกสาร/ด็อกไทป์ (DOCTYPE) ระบุชนิดและเวอร์ชัน (2) การประกาศเนมสเปซ (Namespace) ประกาศรายชื่อแท็กและแอตทริบิวต์ที่ใช้
  • ตัวอย่างโค้ด:

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN"

"http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">

<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en" lang="en">

<head>...<title>XHTML 1.0 Strict Example</title></head>

<body>...</body>

</html>

เอ็กซ์เอ็มแอล (XML – eXtensible Markup Language)

  • นำมาใช้ครั้งแรกปี ค.ศ. 1996 เน้นการจัดการ "โครงสร้างและการจัดเก็บข้อมูล" เป็นมาตรฐานกลางส่งผ่านข้อมูลข้ามแพลตฟอร์ม พัฒนาจาก SGML ควบคุมมาตรฐานโดย W3C
  • เป็นภาษาเมตา (Meta Language) คือข้อมูลที่อธิบายตัวข้อมูลเอง และเป็นพื้นฐานให้ภาษาอื่น เช่น RDF/XML, RSS, MathML, XHTML, SVG, cXML
  • ตัวพิมพ์ใหญ่/เล็กมีความหมายต่างกัน (<name> ≠ <NAME>) และต้องตรวจไวยากรณ์ก่อนจึงแสดงได้
  • ลักษณะสำคัญ: เก็บข้อมูลในรูปแอตทริบิวต์, ผู้ใช้กำหนด Tag เองได้, ใช้ได้หลายแพลตฟอร์ม/ฐานข้อมูล, แลกเปลี่ยนข้อมูลระหว่างระบบต่างกันได้
  • โครงสร้างแบบลำดับขั้นคล้ายต้นไม้: รูทอิลิเมนต์ (root element) → ไชด์อิลิเมนต์ (child element) → อ้างอิง เอ็นทิตี้ (entity) ภายนอกได้
  • ตัวแปลภาษาเรียกว่า XML Parser ทำหน้าที่แปลและตรวจสอบชนิด/ความถูกต้องผ่าน DTD
  • โครงสร้าง XML มี 3 ส่วน: (1) ส่วนประกาศ/โปรล็อก (Prolog) ประกอบด้วยการประกาศ XML เช่น <?xml version="1.0"?> และหมายเหตุ/คอมเมนต์ <!-- ... --> (2) ส่วนประกอบเอกสาร/รูทอิลิเมนต์ (Document Element) เช่น <CATALOG> ที่มีอิลิเมนต์ย่อย <TITLE> <ARTIST> <COUNTRY> <COMPANY> <PRICE> <YEAR> (3) ส่วนนิยามเอกสาร/ดีทีดี (DTD – Document Type Definition) อธิบายว่ามีอิลิเมนต์อะไร ลำดับ และชนิดข้อมูล ใช้ตรวจความเป็นเอกสาร Valid
  • รูปแบบที่ดีของเอกสาร XML มี 2 ระดับ: Well-formed XML Document (รูปแบบถูกต้องตามไวยากรณ์) และ Valid XML Document (well-formed + มี DTD และทุกส่วนตรงตาม DTD)
  • DTD ใช้งานได้ 2 แบบ: สร้างไว้ในเอกสาร XML (ขึ้นต้น <!DOCTYPE ชื่ออิลิเมนต์>) หรือสร้างแยกไฟล์แล้วอ้างถึง (นิยมเมื่อมีผู้พัฒนาหลายกลุ่ม)

4.2 ภาษาสคริปต์ (Script Language)

ภาษาสคริปต์คือชุดคำสั่งที่ทำให้เว็บเพจโต้ตอบกับผู้ใช้ได้ ควบคุมการทำงาน/แสดงผลของแอปพลิเคชัน มักทำงานร่วมกับมาร์กอัป เป็นพื้นฐานสำคัญของการบริการผ่านเว็บ

ประเภทตามฝั่งประมวลผล:

  • ไคลเอ็นต์ไซต์สคริปต์ (Client-Side Script): แปลและประมวลผลด้วยเว็บเบราว์เซอร์ มักเขียนรวมในเอกสาร HTML เช่น JavaScript, VBScript, JScript
  • เซิร์ฟเวอร์ไซต์สคริปต์ (Server-Side Script): แปลและประมวลผลโดยเว็บเซิร์ฟเวอร์ แล้วส่งผลลัพธ์เป็น HTML กลับมา เช่น PHP, ASP.NET, Java Servlet, JSP

ประเภท Dynamic Web Page แบ่ง 3 แบบ: Server-Side Dynamic, Client-Side Dynamic, และแบบรวมทั้งสอง

ตัวอย่างภาษา Server-Side:

  • PHP (Hypertext Preprocessor): โอเพ่นซอร์ส ใช้ฟรี โครงสร้างมาจาก C/Java/Perl เรียนง่าย ทำงานหลายแพลตฟอร์ม นิยมใช้คู่กับ MySQL บนเซิร์ฟเวอร์ Apache/IIS ไฟล์นามสกุล .php สนับสนุนการเขียนเชิงวัตถุ และอ่าน/เขียน XML ได้
  • ASP.NET: ของไมโครซอฟท์ (พัฒนาต่อจาก ASP) ใช้ดอตเน็ตเฟรมเวิร์ก (.NET Framework) มี Class Library และ Common Language Runtime (CLR) เลือกเขียนได้หลายภาษา (C#, VB.NET, JScript)

ตัวอย่างภาษา Client-Side:

  • JavaScript: ไวยากรณ์คล้าย C ใช้ร่วม HTML ประมวลผลฝั่งไคลเอ็นต์ เดิมชื่อ LiveScript พัฒนาโดย Netscape ทำงานได้ดีหลายเบราว์เซอร์จึงนิยมที่สุด คำสั่งอยู่ระหว่าง <SCRIPT>...</SCRIPT> ข้อจำกัด: ติดต่อฐานข้อมูลเซิร์ฟเวอร์ไม่ได้, สร้างส่วนแสดงผลเองไม่ได้ (ต้องพึ่ง HTML)
  • VBScript: ภาษาย่อยของ Visual Basic ของไมโครซอฟท์ เป็นสคริปต์โดยปริยายของ ASP ทำงานดีที่สุดกับ Internet Explorer

ประเด็นที่ควรเน้นเพื่อสอบ

  • แยกแยะ แท็ก (Tag) vs แอตทริบิวต์ (Attribute) และ แท็กเดี่ยว vs แท็กคู่ ให้ได้พร้อมตัวอย่าง
  • จำโครงสร้างมาตรฐาน HTML 3 แท็กหลัก: <HTML>, <HEAD> (ซึ่งมี <TITLE>), <BODY>
  • เปรียบเทียบ HTML vs XHTML vs XML: HTML เน้นแสดงผล/ตัวพิมพ์ใหญ่เล็กไม่ต่างกัน; XHTML เคร่งครัด (ตัวพิมพ์เล็ก เปิดปิดเป็นคู่ มี DOCTYPE + Namespace); XML เน้นโครงสร้างข้อมูล/ตัวพิมพ์ใหญ่เล็กต่างกัน/กำหนด Tag เองได้
  • แยก Well-formed vs Valid ของเอกสาร XML (Valid = well-formed + ตรงตาม DTD)
  • จำ 3 ส่วนของโครงสร้าง XML: โปรล็อก / รูทอิลิเมนต์ / DTD และหน้าที่ของ XML Parser
  • ความหมายและบทบาทของ DTD + วิธีใช้ 2 แบบ (ภายใน/ภายนอกเอกสาร)
  • วิวัฒนาการ GML → SGML → HTML/XML และเหตุที่ SGML ไม่นิยม (ซับซ้อนเกินไป)
  • ความแตกต่าง Client-Side vs Server-Side Script: Client-Side ผู้ใช้เห็นโค้ดในเครื่องตน เบราว์เซอร์ประมวลผล; Server-Side ผู้ใช้เห็นแต่ HTML ทำงานได้ทุกเบราว์เซอร์และติดต่อฐานข้อมูลคล่องกว่า
  • จำตัวอย่างภาษาแต่ละฝั่ง: Server-Side (PHP, ASP.NET, JSP, Java Servlet) / Client-Side (JavaScript, VBScript, JScript) และเหตุที่ JavaScript นิยมสุด
  • ประเภทเว็บเพจ Static vs Dynamic (ข้อดี-ข้อจำกัด) และเครื่องมือสร้างเว็บ 3 กลุ่ม (มาร์กอัป+สคริปต์ / โปรแกรมสำเร็จรูป / CMS)

หน่วยที่ 5: ความรู้พื้นฐานของภาษาเอ็กซ์เอ็มแอล (XML)

ภาพรวม

หน่วยนี้เจาะลึกภาษา XML (eXtensible Markup Language) ซึ่งเป็นภาษามาร์กอัปมาตรฐานของ W3C สำหรับจัดส่ง/ประมวลผลข้อมูลบนอินเทอร์เน็ต แบ่งเป็น 3 ตอน: (5.1) หลักการและโครงสร้างภาษา XML รวมถึงกฎ Well-formed (5.2) การนิยามชนิดเอกสารด้วย DTD, เอ็นทิตี้ (Entity) และ XML Schema (5.3) การจัดรูปแบบการแสดงผลด้วย CSS และ XSL จุดเด่นของ XML คือแยกข้อมูลออกจากการแสดงผล อธิบายตัวเองได้ และทำงานข้ามแพลตฟอร์ม จึงเป็นพื้นฐานของการบริการผ่านเว็บ (Web Service) ที่ใช้ SOAP

แนวคิดสำคัญ

  • XML มีรากฐานจาก SGML (Standard Generalized Markup Language) ที่ทรงพลังแต่ซับซ้อนเกิน XML จึงเป็น "เซตย่อย (subset) ของ SGML" ที่ลดความซับซ้อนลง เน้นแลกเปลี่ยนข้อมูลผ่านอินเทอร์เน็ต
  • เทียบกับ HTML: HTML กำหนดแค่ "การแสดงผล" (เช่นแสดงชื่อในแท็ก <p>) ส่วน XML แสดง "รายละเอียดความหมายของข้อมูล" (เช่น <firstname>Somsri</firstname>, <lastname>Jaidee</lastname>) จึงสั้นกระชับและสื่อความหมายกว่า
  • XML แยกส่วนระหว่างผู้ใช้กับข้อมูล ทำงานร่วมข้อมูลภายนอกได้ดี และยืดหยุ่นกว่า HTML

เนื้อหาหลัก

5.1 หลักการและโครงสร้างภาษา XML

เป้าหมายหลักของ XML 10 ข้อ (กำหนดโดย W3C) สรุปสาระ: ใช้งานตรงบนอินเทอร์เน็ต, สนับสนุนโปรแกรมประยุกต์หลากหลาย, เข้ากันได้กับ SGML, เขียนโปรแกรมประมวลผลได้ง่าย, มีคุณลักษณะทางเลือก (option) น้อยที่สุด, อ่านเข้าใจง่ายชัดเจน, นำสู่มาตรฐานได้เร็ว, รูปแบบกะทัดรัดไม่คลุมเครือ, สร้างเอกสารง่าย, มาร์กอัปไม่รวบรัดเกินไป (เน้นความชัดเจนมากกว่าลดขั้นตอน)

การประมวลผล XML ใช้ตัวแปลชื่อ XML Parser / XML Processor ทำหน้าที่อ่าน (read) สร้าง (create) แก้ไข (update) และเข้าถึง (manipulate) เอกสาร แบ่งตาม:

  • การตรวจสอบคุณสมบัติ: Validating Document (ตรวจทั้ง well-formed และ valid ทำงานช้ากว่า) vs Non-validating Document (ตรวจแค่ well-formed เร็วกว่า)
  • การสำรวจเนื้อหา (API): DOM (Document Object Model) ใช้ง่ายแต่กินหน่วยความจำมาก vs SAX (Simple API for XML) ใช้ยากแต่เร็วและกินหน่วยความจำน้อย

โปรแกรมประยุกต์ที่ใช้ร่วมกับ XML (vocabulary XML): XSL (Extensible Style Sheet Language), XML Schema, XLink (สร้างลิงก์หลายที่หมาย), XPointer (สร้างลิงก์ยืดหยุ่นสูง)

ความสามารถของ XML: จัดเก็บฐานข้อมูล (field ใน record), กำหนดโครงสร้างเอกสารแบบลำดับขั้น, สื่อสารระหว่างโปรแกรมบนเว็บด้วย SOAP, จัดเก็บกราฟิกเวกเตอร์ (XML Graphic Vector)

XML กับการบริการผ่านเว็บ: Web Service ใช้ XML เป็นภาษากลางสื่อสารระหว่างคอมพิวเตอร์ ผ่านโพรโทคอล HTTP (หรือ SMTP, FTP)

โครงสร้างเอกสาร XML มี 2 ส่วน:

  1. โปรล็อก (Prolog): ส่วนแรก ประกอบด้วย การประกาศ XML (XML declaration) เช่น <?xml version="1.0"?> (ควรประกาศเสมอ อาจมี encoding declaration เช่น UTF-8) และ หมายเหตุ (comment) เริ่ม <!-- ปิด -->
  2. อิลิเมนต์ข้อมูล (Document Element): เก็บเนื้อหาจริงทั้งหมด ประกอบด้วยอิลิเมนต์ย่อย (subelements) และเอ็นทิตี้ภายนอก

กฎทั่วไปการสร้างอิลิเมนต์ใหม่ (สำคัญต่อ Well-formed):

1) ชื่ออิลิเมนต์เริ่มด้วยตัวอักษรอังกฤษ/ขีดล่าง _/ตัวเลข/จุด ./ยัติภังค์ - (เช่น NAME, _2ndFood, St_Address)

2) ห้ามขึ้นต้นด้วย "xml" (สงวนไว้)

3) ตัวพิมพ์ใหญ่/เล็กต่างกัน → <title> ≠ <TITLE>

4) ต้องมี รูทอิลิเมนต์ (root element) เพียงหนึ่งเดียว เช่น <CATALOG>

5) ทุกอิลิเมนต์ต้องมีแท็กเปิด (start tag) และแท็กปิด (end tag) ชื่อเดียวกัน

6) ต้องวางตำแหน่งถูกต้อง ห้ามซ้อนเหลื่อม (overlap)

7) แท็กว่างเขียนได้ 2 แบบ: <name></name> หรือ <name/>

คุณลักษณะข้อมูลภายในอิลิเมนต์: อิลิเมนต์แบบตัวอักษร (character data), อิลิเมนต์แบบอ้างอิงตัวอักษร (character reference), อิลิเมนต์แบบซ้อน (nested element)

แอตทริบิวต์ (Attribute): ใช้เก็บค่าเพิ่มเติมในอิลิเมนต์ เช่น <ARTIST name="...">, <PRICE discount="9.90" cost="20.45">50.45</PRICE>

การแสดงผล: แบบไม่มี Style sheet (เปิดด้วย Windows Explorer เห็นโครงสร้างพับ/ยืดด้วย "-" "+") และแบบใช้ CSS (เก็บไฟล์ .css แยก จัดสีพื้นหลัง/ตัวอักษรให้สวยงาม)

5.2 ภาษา XML กับการนิยามชนิดเอกสาร (DTD / Entity / Schema)

การนิยามชนิดเอกสาร (DTD – Document Type Declaration/Definition): อธิบายโครงสร้างที่อนุญาตให้ตัวประมวลผลตรวจสอบว่าเอกสาร valid หรือไม่ อยู่ในส่วนโปรล็อก เมื่อประกาศแล้ว DTD เป็นกฎข้อบังคับทันที

วิธีประกาศ DTD:

  • External DTD subset (ภายนอกเอกสาร): เก็บไฟล์ .dtd แยก เรียกผ่าน DOCTYPE ใช้คำสั่ง SYSTEM (อ้างเป็น URL) หรือ PUBLIC (relative path)
  • Internal DTD subset (ภายในเอกสาร): เก็บใน XML เดียวกัน โปรแกรมประมวลผลตัวภายในก่อนแล้วค่อยโหลดภายนอก

การประกาศอิลิเมนต์ใน DTD กำหนดชนิด/เนื้อหา (content):

1) อิลิเมนต์ว่าง (empty element) ใช้คำสำคัญ EMPTY

2) อิลิเมนต์อื่นๆ (any element) ใช้ ANY เก็บได้ทุกชนิด

3) อิลิเมนต์จัดเก็บโมเดลเนื้อหา (content model) มีแค่อิลิเมนต์ย่อยที่ประกาศไว้

4) อิลิเมนต์ผสม (mixed content) ใช้ #PCDATA (Parsed Character Data – ข้อความที่ใช้ทำมาร์กอัป)

ตัวอย่างประกาศ:

<!DOCTYPE STUDENT [

<!ELEMENT STUDENT (NAME,PROVINCE,GENDER,AGE)>

<!ELEMENT NAME (#PCDATA)>

...

]>

การประกาศแอตทริบิวต์ (attribute-list declaration) ใช้ <!ATTLIST ...>

  • ชนิดแอตทริบิวต์ (10 ชนิด): CDATA, ENTITY, ENTITIES, ID (ค่าไม่ซ้ำ), IDREF, IDREFS, NMTOKEN, NMTOKENS, NOTATION, Enumerated
  • ค่าเริ่มต้น (Default): #REQUIRED (ต้องใส่ค่า), #IMPLIED (ใส่หรือไม่ก็ได้), #FIXED (ค่าตายตัว), default (ระบุค่าเริ่มต้น)
  • ตัวอย่าง: <!ATTLIST STUDENT GENDER CDATA (Female|Male) "Female">

เอ็นทิตี้ (Entity): เครื่องมือรวมข้อมูลที่ใช้บ่อยให้เป็น "บล็อกข้อความ" เพื่อแทรกซ้ำได้ ประกาศใน DTD แบ่ง 3 มิติ:

  • General entity (ใช้ได้ทั้งใน DTD/ไฟล์นอก) vs Parameter entity (ใช้เฉพาะใน DTD, ใช้เครื่องหมาย %)
  • Internal entity (ภายในเอกสาร) vs External entity (ไฟล์แยกภายนอก)
  • Parsed entity (แทนที่ด้วยข้อความ replacement text เรียกด้วย &name;) vs Unparsed entity (ต้องมี NOTATION คู่กันเสมอ ใช้คำสำคัญ NDATA)
  • คำสำคัญในการอ้างอิงภายนอก: SYSTEM (ภายในเอกสาร), PUBLIC (สาธารณะ), identifier (URI/URL)
  • ตัวอย่าง: <!ENTITY R "Romance"> แล้วเรียกใช้ <genre>&R;</genre>

ข้อจำกัดของ DTD: สนับสนุนชนิดข้อมูลแค่ 10 ชนิด และ "ไม่อยู่ในรูปไวยากรณ์ XML" จึงใช้ XML Parser แปล DTD ไม่ได้ → จึงไม่นิยมมากนัก

เอ็กซ์เอ็มแอลสคีมา (XML Schema): สร้างด้วย XML-Data เพื่อแก้ข้อจำกัด DTD

  • ข้อต่างจาก DTD: Schema "เขียนด้วยโค้ด XML เอง" (well-formed ตรวจสอบได้) ใช้แท็ก <schema> เป็นรูทอิลิเมนต์ อ้างเนมสเปซ http://www.w3.org/2001/XMLSchema กำหนด "ชนิดข้อมูล (data type)", ช่วงค่า, และค่าแอตทริบิวต์ได้ และไม่มีส่วน DTD ในตัว
  • ชนิดข้อมูลพื้นฐาน: string, boolean, float (32 บิต), double (64 บิต), decimal, timeDuration, binary, uniReference (URI), ID, IDREF, ENTITY, NOTATION, QName
  • ชนิดข้อมูลที่สืบทอด: กำหนดจากชนิดที่มีอยู่ เช่น age สืบจาก integer กำหนด minInclusive=0, maxInclusive=100
  • การนิยาม 2 แบบ: การนิยามอย่างง่าย (Simple definition) ใช้ <simpleType> (มีแอตทริบิวต์ name, base, abstract, derivedBy) พร้อมข้อบังคับ (facets) เช่น enumeration, minLength, maxLength, pattern, whiteSpace; และ การนิยามชนิดข้อมูลซับซ้อน (Complex type definition) ใช้ <complexType> (เพิ่มแอตทริบิวต์ content = textOnly/Mixed/eltOnly/empty) ใช้แท็ก <sequence> กำหนดลำดับ และ <choice> กำหนดตัวเลือก
  • ตัวอย่าง:

<simpleType name="firstname" base="string">

<minlength value="5"/>

<maxlength value="15"/>

</simpleType>

5.3 การจัดรูปแบบการแสดงเอกสาร XML (CSS / XSL)

เนื่องจาก XML เน้นโครงสร้างข้อมูล ไม่ได้เน้นความสวยงาม จึงต้องใช้สไตล์ชีต (Style sheet) มาช่วยจัดตำแหน่ง รูปแบบตัวอักษร รูปภาพ สีสัน

  • CSS (Cascading Style Sheet): สร้างได้ 2 แบบ คือ ภายใน (Internal) แทรกในเอกสารด้วย <style>...</style> (ขาดความยืดหยุ่น) และ ภายนอก (External) เก็บไฟล์ .css แยก เรียกใช้ผ่านแท็ก <link> (anchor tag <a href="...">) ในส่วน <head> โดยใช้คำสั่ง rel (relationship) – แบบภายนอกแก้ไขง่ายและใช้ซ้ำได้
  • XSL (Extensible Style Sheet Language / เอ็กซ์เอสแอล): สไตล์ชีตที่ถูกประยุกต์/เพิ่มความสามารถให้มากและยืดหยุ่นกว่า CSS ใช้จัดรูปแบบเอกสารให้อยู่ในรูปแบบเดียวกัน

ประเด็นที่ควรเน้นเพื่อสอบ

  • จำ โครงสร้าง XML 2 ส่วน: โปรล็อก (XML declaration + comment) และอิลิเมนต์ข้อมูล (รูท + ย่อย)
  • ท่อง กฎ Well-formed 7 ข้อ โดยเฉพาะ: รูทอิลิเมนต์เดียว, ทุกแท็กเปิด-ปิดเป็นคู่, ตัวพิมพ์ใหญ่/เล็กต่างกัน, ห้ามซ้อนเหลื่อม, ห้ามขึ้นต้นชื่อด้วย xml
  • แยก Well-formed vs Valid และ Validating vs Non-validating parser
  • เปรียบเทียบ DOM vs SAX (ใช้ง่าย/กินหน่วยความจำ vs เร็ว/ประหยัด)
  • ความหมายและวิธีประกาศ DTD 2 แบบ (Internal vs External) และข้อจำกัด 2 ข้อของ DTD (10 ชนิดข้อมูล + ไม่อยู่ในไวยากรณ์ XML)
  • เขียน/อ่านการประกาศอิลิเมนต์ (<!ELEMENT>) และแอตทริบิวต์ (<!ATTLIST>) ได้ พร้อมรู้ค่า #REQUIRED #IMPLIED #FIXED และ #PCDATA
  • จำการแบ่งประเภท เอ็นทิตี้ (Entity) 3 มิติ: General/Parameter, Internal/External, Parsed/Unparsed และวิธีอ้างอิง &name; กับเครื่องหมาย %
  • เปรียบเทียบ DTD vs XML Schema: Schema เขียนด้วยโค้ด XML, กำหนด data type/ช่วงค่าได้, แก้ข้อจำกัด DTD
  • รู้จัก simpleType vs complexType และแท็ก <sequence>, <choice> ใน Schema
  • แยก CSS (Internal vs External) vs XSL และเหตุผลที่ต้องใช้สไตล์ชีตกับ XML (XML แยกข้อมูลออกจากการแสดงผล)
  • เข้าใจความต่าง HTML vs XML (แสดงผล vs สื่อความหมายข้อมูล) และบทบาท XML ในเว็บเซอร์วิส (ภาษากลาง + SOAP)

ภาพรวม

หน่วยนี้อธิบาย SOAP (Simple Object Access Protocol) ซึ่งเป็นโพรโทคอลมาตรฐานสำหรับการสื่อสารระหว่างเว็บเซอร์วิส (Web Service) แบ่งเป็น 2 ตอน: (6.1) ความรู้พื้นฐาน – ความเป็นมา ความหมาย ขั้นตอนรับ-ส่งข้อความ ข้อดี-ข้อจำกัด และองค์ประกอบ/โครงสร้างของข้อความ SOAP (6.2) การพัฒนาและใช้งาน SOAP ด้วยภาษา PHP และ Java จุดเด่นของ SOAP คือเป็นโพรโทคอลกลางที่ไม่มีใครเป็นเจ้าของ ทำงานข้ามแพลตฟอร์ม/ระบบปฏิบัติการ/ภาษาโปรแกรมได้ โดยใช้ XML เป็นรูปแบบข้อความและส่งผ่าน HTTP

แนวคิดสำคัญ

  • ปัญหาที่ทำให้เกิด SOAP: แต่ละองค์กรใช้แพลตฟอร์ม (platform), แบบจำลองทางวัตถุ (object model) และภาษาโปรแกรมต่างกัน ทำให้สื่อสารระยะไกล (RPC – Remote Procedure Call) ลำบาก และ HTTP เพียงอย่างเดียวรองรับไม่ได้ จึงต้องมีโพรโทคอลกลางที่เข้าถึงแอปพลิเคชันบนรีโมตคอมพิวเตอร์ได้ง่าย ไม่ซับซ้อน อธิบายชัดเจน ยืดหยุ่น
  • SOAP ถูกกำหนดโดยการรวมตัวของหลายบริษัท (Microsoft, IBM, Lotus, UserLand, Developer Mentor + อีกกว่า 30 บริษัท) จัดตั้งเป็น W3C XML Protocol Workgroup กลายเป็น Services-Oriented Architecture Protocol
  • SOAP ทำให้โปรแกรมต่างภาษา (เช่น Java กับ Cobol/C++/Perl) ที่ปกติคุยกันไม่ได้ สามารถเชื่อมโยง (integrate) กันผ่าน XML ได้ โดยมี WSDL อธิบายบริการ และ UDDI เป็นทะเบียน
  • SOAP ไคลเอ็นต์ต้องแปลภาษาท้องถิ่น (local เช่น Java) ให้เป็นภาษามาตรฐาน (standard language) แล้วแปลงเป็นเอกสาร SOAP ก่อนส่งออกเครือข่าย

เนื้อหาหลัก

6.1 ความรู้พื้นฐานเกี่ยวกับ SOAP

ความหมาย: SOAP (Simple Object Access Protocol) คือโพรโทคอล/ระเบียบวิธีสื่อสารระหว่างเว็บเซอร์วิส ใช้ติดต่อระหว่างคอมพิวเตอร์ต่างที่ในสภาพแวดล้อมแบบกระจาย ในรูปแบบ "คำร้องขอ-คำตอบรับ" (Request-Response) ด้วยข้อความ SOAP (SOAP message) จุดเด่นคือเป็นกลาง ทำงานกับโพรโทคอลอื่นได้หลายชนิด และไม่มีใครเป็นเจ้าของ

ขั้นตอนการรับ-ส่งข้อความ SOAP:

1) ไคลเอ็นต์สร้างข้อความเพื่อสื่อสาร/บอกความต้องการ

2) ส่ง SOAP request message ขอรับบริการกับเซิร์ฟเวอร์ผ่านเครือข่าย

3) ข้อความส่งผ่านด้วยโพรโทคอล HTTP พอร์ต 80

4) เซิร์ฟเวอร์อ่านข้อความและสร้างข้อความตอบกลับ

5) เซิร์ฟเวอร์ส่ง SOAP response message กลับไปยังไคลเอ็นต์

ผู้ดำเนินการ (handler): การรับ-ส่ง SOAP ต้องผ่าน handler ที่อยู่ในคลาสเชื่อมต่อกับเว็บเซอร์วิส ทำ pre-processing และ post-processing บนข้อความ XML วงจรชีวิตของ handler มี 4 ขั้น:

1) การสร้าง (Instantiated) ด้วยเมธอด init()

2) การร้องขอ (Request) ด้วยเมธอด handleRequest()

3) การส่งกลับ (Return) ด้วย handleResponse() หรือแสดงข้อผิดพลาด handleFault()

4) การทำลาย (Destroy) ด้วย destroy()

ข้อดีของ SOAP:

1) รูปแบบเป็นที่ยอมรับ (Microsoft, Sun Microsystems รับรอง) ใช้ได้กับ C#, VB.NET, Perl, Java, Python ฯลฯ

2) ไม่ต้องปรับเปลี่ยนรูปแบบโปรแกรม (ทำงานบนพื้นฐานเว็บอยู่แล้ว)

3) รองรับแพลตฟอร์มต่างกัน (XML ส่งเป็น text format ที่อธิบายตัวเองได้)

4) ทำงานทะลุไฟร์วอล (firewall) ได้ เพราะอยู่บน HTTP

5) ส่งข้อมูลปริมาณมากได้ (ใช้ XML)

6) มีความมั่นคงปลอดภัย เข้ารหัสด้วยมาตรฐาน HTTP รองรับหลายโพรโทคอล (FTP, HTTP)

ข้อจำกัดของ SOAP:

1) ส่งข้อมูลไม่ได้ทันที ต้องจัดทำ SOAP packet (request + message) ก่อน ผู้พัฒนาต้องเขียนโปรแกรมแยก/แปลข้อความเพิ่มเอง

2) เอกสารใช้หน่วยความจำมาก (XML + DTD/Schema กินพื้นที่และเวลาประมวลผล อาจเกิด Overhead)

3) ไม่ตรวจสอบเอกสารหลังแปล (พบ error ได้เฉพาะตอนรันไทม์)

4) ต้องตกลงระหว่างองค์กรบนมาตรฐาน XML Schema เสมอ

องค์ประกอบและโครงสร้างข้อความ SOAP

SOAP ไม่กำหนดรูปแบบวัตถุหรือการเชื่อมต่อภาษา มีแค่ "เฟรมเวิร์ก (framework)" สำหรับข้อความ XML ข้อมูลพื้นฐานของ SOAP มี 4 ส่วน: โครงสร้างด้วยข้อความ XML, การเรียก procedure ระยะไกลด้วย XML, การเชื่อมต่อไปยัง HTTP, และความสะดวกในการส่งข้อผิดพลาดกลับ

ข้อความ SOAP (SOAP Message Structure) อยู่ในรูปเอกสาร XML แบ่งเป็น 3 ส่วนหลัก:

  1. SOAP Envelope (ส่วนซองจดหมาย): อิลิเมนต์หลัก (root) ของข้อความ ทำหน้าที่เหมือนซองจดหมาย เก็บข้อมูลจำเป็น เช่น รุ่น (Version), วิธีเข้ารหัส (Encoding) โดยมี Header และ Body เป็นอิลิเมนต์ลูก

<env:Envelope> ... </env:Envelope>

encoding เป็นทางเลือก (option) ชี้ไปยัง XML Schema เพื่อกำหนดโครงสร้างให้ถูกต้อง

  1. SOAP Header (ส่วนหัว): เหมือนหัวจดหมาย เป็นอิลิเมนต์ทางเลือก (มีหรือไม่มีก็ได้) เก็บข้อมูลเสริม เช่น Transaction, การเข้ารหัส, การระบุรหัสข้อความ (MessageId ให้ไม่ซ้ำ) และ correlation (จัดการด้วย Message Correlation Handler)
  1. SOAP Body (ส่วนเนื้อหา): เหมือนเนื้อหาจดหมาย เก็บข้อมูลที่ส่งให้ผู้รับ/เรียกใช้เว็บเซอร์วิสและผลลัพธ์ ภายในประกอบด้วย:

- อิลิเมนต์ลูก (Child Element) เรียก Body Block เป็นข้อมูลชนิดต่างๆ (เช่น fGross ชนิด float = 3500, iGrade ชนิด integer = 1)

- SOAP Fault เป็น Body Block ชนิดหนึ่ง คำสั่งมาตรฐานสำหรับแสดงข้อผิดพลาดกลับไปยังผู้ส่ง ประกอบด้วย: <faultcode> (รหัสระบุ), <faultstring> (ข้อความที่มนุษย์อ่านได้), <faultactor> (ใครทำให้ผิดพลาด), <detail> (รายละเอียดผิดพลาดใน body element) — รหัส SOAP Fault มาตรฐาน: VersionMismatch, MustUnderstand, Client, Server

ลำดับโครงสร้างในเอกสาร (ตัวอย่าง): บรรทัด 1 = รุ่น XML, Envelope บรรทัด 2-18, Header บรรทัด 3-14, Body บรรทัด 15-17

ขั้นตอนการส่งข้อความ SOAP (5 ลำดับ):

  • ลำดับ 1: ไคลเอ็นต์สร้างข้อความ SOAP เรียกเว็บเซอร์วิส ระบุเซิร์ฟเวอร์ปลายทาง
  • ลำดับ 2: SOAP ไคลเอ็นต์ส่ง SOAP request ผ่านเครือข่าย (ตอบรับด้วย HTTP)
  • ลำดับ 3: เซิร์ฟเวอร์ปลายทางรับ request ที่ SOAP เซิร์ฟเวอร์ ประมวลผล และสร้าง SOAP response
  • ลำดับ 4: SOAP เซิร์ฟเวอร์ส่ง response กลับไปเซิร์ฟเวอร์ต้นทาง
  • ลำดับ 5: SOAP ไคลเอ็นต์รับ response แปลงข้อความให้แอปพลิเคชันรับทราบผล

6.2 การพัฒนาและใช้งาน SOAP

ขั้นตอนพัฒนาเว็บเซอร์วิสโดยทั่วไป:

  • ผู้พัฒนาสร้างเว็บเซอร์วิส (Implementation/create)
  • ติดตั้งเว็บเซอร์วิส (Deploy)
  • ลงทะเบียนเพื่อเปิดบริการ (UDDI Registry)
  • ผู้ใช้สืบค้นบริการจาก UDDI (Finding/Consume) อาจมีค่าบริการ

ส่วนประกอบสำคัญ: แอปพลิเคชัน (โปรแกรมให้บริการ มี SOAP เป็นโพรโทคอลสื่อสาร) และ WSDL (Web Services Description Language) = เอกสาร XML ที่อธิบายว่าเซอร์วิสให้บริการอะไรและติดต่ออย่างไร

ภาษาที่มีโมดูล/เครื่องมือ (toolkit) ช่วยจัดการ SOAP: Java, C#, C++, C, Perl, PHP, Python (เช่น Microsoft .NET for C, Apache SOAP for Java, SOAP::Lite for Perl)

6.2.1 การพัฒนาด้วย PHP

PHP เหมาะกับเว็บแอปพลิเคชัน เพราะง่าย เรียนรู้เร็ว ใช้ฟรี และสนับสนุนเว็บเซอร์วิสในตัวด้วย SOAP extension ไม่ต้องดาวน์โหลดเพิ่ม

  • ติดตั้ง SOAP extension: เพิ่มคำสั่งในไฟล์ php.ini (Windows: c:/windows/php.ini) แล้ว restart Apache (Linux เปิดด้วย -enable-soap)
  • Runtime Configuration: ฟังก์ชันรันไทม์อยู่ใน php.ini เช่น soap.wsdl_cache_enabled, soap.wsdl_cache_dir, soap.wsdl_cache_ttl
  • สร้าง SOAP server จากคลาส PHP: สร้าง wrapper file โหลดคลาสเข้ามาทำงานกับ SOAP server
  • หมายเหตุ: PHP 4 ต้องใช้ NuSOAP library แต่ PHP 5 ขึ้นไปมี SOAP ในตัว (ชนกับ NuSOAP)
  • ฟังก์ชัน SOAP สำคัญ: is_soap_fault() (ตรวจว่ามี fault), use_soap_error_handler()
  • SoapClient (มีตั้งแต่ PHP 5.0.1): __call(), __doRequest(), __getLastRequest(), __getFunctions(), __getTypes(), __soapCall() (ส่งพารามิเตอร์หลายตัวแบบอาร์เรย์), __setLocation(), __setSoapHeaders() — สร้างด้วย SoapClient(wsdl, options)
  • SoapServer: addFunction(), addSoapHeader(), getFunctions(), handle() (จัดการ request), setClass(), setObject(), fault() — สร้างด้วย SoapServer(wsdl, options)
  • คำเตือนความปลอดภัย: SOAP_FUNCTIONS_ALL เพิ่มทุกฟังก์ชัน PHP ให้ไคลเอ็นต์เรียกได้ อาจเป็นภัยคุกคาม (เช่นไคลเอ็นต์อาจเรียกอ่านไฟล์รหัสผ่านได้)

6.2.2 การพัฒนาด้วย Java

Java เขียนคล้าย C/C++ มีเครื่องมือ/โมดูล/เมธอดเรียกใช้ได้ทันที (import คลาสก่อนทำงาน) ตัวอย่างใช้ Java Servlet 2 ตัว (ส่งและรับข้อความ SOAP) บน JavaServer (ดาวน์โหลดได้จาก Tomcat)

  • ขั้นตอน: สร้างลอจิกธุรกิจ (business logic) → ติดตั้งคลาสจาวาที่ SOAP server (เช่น WASP deployment tool, Java XML pack, Java Web Services Developer's Pack) → สร้างคลาสเข้าถึงไคลเอ็นต์ (ใช้ proxy object แปลง call เป็น XML ตอนร้องขอ และแปลง response กลับเป็น Java object) → พัฒนาไคลเอ็นต์แอปพลิเคชัน
  • ฝั่งเซิร์ฟเวอร์: import library → เมธอด init() สร้างออบเจ็กต์ connection (SOAP connection) → เมธอด doGet() สร้าง MessageFactory + SOAP message, สร้าง envelope (header + body), ส่งข้อความ และตรวจสอบข้อความตอบกลับ
  • ฝั่งไคลเอ็นต์: เมธอด init() สร้าง MessageFactory → เมธอด onMessage() รับข้อความ (getSOAPPart()) แยกส่วนซองและเนื้อหา ใส่คำตอบ และส่งค่ากลับ (getValue())

ประเด็นที่ควรเน้นเพื่อสอบ

  • จำ โครงสร้างข้อความ SOAP 3 ส่วนหลัก: Envelope (ซอง/อิลิเมนต์หลัก), Header (ส่วนหัว/ทางเลือก), Body (เนื้อหา) พร้อมการเปรียบเทียบกับ "ซองจดหมาย-หัวจดหมาย-เนื้อหาจดหมาย" — ออกสอบบ่อย
  • SOAP Fault อยู่ใน Body และคำสั่งย่อย 4 ตัว (faultcode, faultstring, faultactor, detail) + รหัส VersionMismatch/MustUnderstand/Client/Server
  • ความหมาย SOAP และเหตุผลที่ต้องมี (แพลตฟอร์ม/ภาษา/object model ต่างกัน + HTTP ไม่รองรับ RPC ระยะไกล)
  • จำ ขั้นตอนรับ-ส่งข้อความ SOAP 5 ลำดับ (Request-Response ผ่าน HTTP พอร์ต 80)
  • ข้อดี 6 ข้อ / ข้อจำกัด 4 ข้อ ของ SOAP โดยเฉพาะ: ทำงานทะลุไฟร์วอลได้ (เพราะอยู่บน HTTP), ข้ามแพลตฟอร์ม (เพราะ XML อธิบายตัวเองได้) vs ใช้หน่วยความจำมาก/ตรวจ error ได้แค่รันไทม์
  • บทบาทของ WSDL (อธิบายบริการ) และ UDDI (ทะเบียน/สืบค้น) ในเว็บเซอร์วิส
  • วงจรชีวิตของ handler 4 ขั้น และเมธอด init()/handleRequest()/handleResponse()/destroy()
  • SOAP ใช้ XML เป็นรูปแบบข้อความ และส่งผ่าน HTTP เป็นหลัก (ได้ SMTP/FTP ด้วย)
  • การพัฒนาด้วย PHP: SOAP extension ใน php.ini, ฟังก์ชัน SoapClient vs SoapServer, ข้อควรระวัง SOAP_FUNCTIONS_ALL
  • การพัฒนาด้วย Java: ใช้ Java Servlet + proxy object, ขั้นตอน create → deploy → generate client → develop client

หน่วยที่ 7: โพรโทคอลดับเบิ้ลยูเอสดีแอล (WSDL) ในการบริการผ่านเว็บ

ภาพรวม

หน่วยนี้อธิบายเอกสาร WSDL (Web Services Description Language) ซึ่งเป็นเอกสารเอ็กซ์เอ็มแอล (XML) ที่ทำหน้าที่ "อธิบายรายละเอียดการให้บริการ" (Service Description) ของการบริการผ่านเว็บ เปรียบเสมือนแบบฟอร์มสั่งซื้อสินค้าที่บอกว่าผู้ให้บริการมีบริการอะไร เรียกใช้ได้อย่างไร และเรียกใช้จากที่ใด เมื่อผู้ร้องขอบริการได้รับเอกสารนี้ก็จะนำไปพัฒนาแอปพลิเคชันเพื่อเรียกใช้บริการได้ เนื้อหาครอบคลุมความเป็นมา องค์ประกอบพื้นฐาน และโครงสร้าง/องค์ประกอบของ WSDL ทั้งเวอร์ชัน 1.1 และ 2.0 ที่รับรองโดย W3C

แนวคิดสำคัญ

  • WSDL ทำงานภายใต้สถาปัตยกรรมเชิงบริการ (Service-Oriented Architecture: SOA) ซึ่งเทียบได้กับระบบไปรษณีย์: ผู้ร้องขอบริการ (Service Requestor) คือผู้ส่งจดหมาย, ผู้ให้บริการ (Service Provider) คือผู้รับ, โพรโทคอล SOAP คือซองจดหมาย, และ WSDL คือแบบฟอร์มสั่งซื้อสินค้า
  • WSDL ตอบคำถามสำคัญ 5 ข้อ (5W): Where (เรียกใช้จากที่ใด), What (มีบริการอะไร), How (เรียกใช้อย่างไร), Why (ต่างจากบริการอื่นอย่างไร), Who (รายละเอียดผู้ให้บริการ)
  • ข้อ 1-3 (Where/What/How) เป็น รายละเอียดที่เป็นฟังก์ชันนอล (Functional Description) ขาดไม่ได้ ส่วนข้อ 4-5 (Why/Who) เป็น รายละเอียดที่ไม่เป็นฟังก์ชันนอล (Non-functional Description) ซึ่งอาจไม่มีก็ได้
  • เอกสาร WSDL หนึ่งฉบับสามารถมีรายละเอียดได้มากกว่า 1 บริการ

เนื้อหาหลัก

7.1.1 ความเป็นมาและบทบาทของ WSDL

  • ลำดับเวลา: ปี ค.ศ. 1970 ARPANET พัฒนา IDL (Interface Definition Language) สำหรับ RPC (Remote Procedure Call) ต่อมาบริษัท Xerox นำไปใช้เชิงพาณิชย์ปี 1981 จากนั้นพัฒนาเป็น XML-RPC; กลุ่ม OMG (Object Management Group) คิดค้น CORBA โดยใช้ OMG-IDL เป็นรายละเอียดการให้บริการ — ทั้ง RPC และ CORBA ไม่ขึ้นกับแพลตฟอร์มแต่ไม่ได้รับการรับรองจาก W3C
  • กำเนิด WSDL: กันยายน ค.ศ. 2000 บริษัท IBM, Microsoft และ Ariba ร่วมกันพัฒนา WSDL 1.0 โดยรวมแนวคิดจาก NASSL (ของ IBM) และ SDL (ของ Microsoft)
  • เวอร์ชันที่ W3C รับรอง: 15 มี.ค. 2001 รับรอง WSDL 1.1 (ใช้กับ SOAP 1.1); 26 มิ.ย. 2007 รับรอง WSDL 2.0 (ใช้กับ SOAP 1.2) ซึ่งจริงๆ คือ WSDL 1.2 ที่เปลี่ยนแปลงมากจนเปลี่ยนเป็นเวอร์ชัน 2.0
  • WSDL 1.0 และ 1.1 มีโครงสร้างเหมือนกันทุกประการ

7.1.2 องค์ประกอบพื้นฐานของ WSDL

มีพื้นฐานสำคัญ 3 ส่วน:

  1. ภาษาเอ็กซ์เอ็มแอล (XML Language): เป็นโครงสร้างพื้นฐานที่สำคัญที่สุด เพราะตัวเอกสารและอีก 2 ส่วนล้วนเขียนจากภาษานี้ มีองค์ประกอบ เช่น แท็ก (Tag: แท็กเปิด/แท็กปิด/แท็กว่าง), อิลิเมนต์ (Element), แอตทริบิวต์ (Attribute), ใช้ยูนิโค้ด (Unicode) และเป็น Case Sensitive ต้องมีการประกาศค่า XML (<?xml version="1.0" encoding="UTF-8"?>) ในบรรทัดแรกเสมอ
  2. สกีมาเอ็กซ์เอ็มแอลหรือเอ็กซ์เอสดี (XML Schema Definition: XSD): เอกสารที่นิยามโครงสร้าง เช่น กำหนดอิลิเมนต์/แอตทริบิวต์ที่มีได้ ลำดับ ชนิดข้อมูล และค่าเริ่มต้น; เอกสาร XML อ้างอิงสกีมาผ่านยูอาร์ไอ (URI) และสกีมาอ้างอิงสกีมาอื่นได้ (คุณสมบัติ Extensible)
  3. เอ็กซ์เอ็มแอลเนมสเปซ (XML Namespaces: xmlns): ป้องกันการตั้งชื่ออิลิเมนต์ซ้ำซ้อน/ลดความกำกวม โดยใช้คำนำหน้า (Prefix) รูปแบบ <prefix:tagName> และประกาศด้วย xmlns:prefix="uri"
  • ตารางสกีมา/เนมสเปซสำคัญของ WSDL 1.1 ได้แก่ wsdl (สกีมาพื้นฐาน/Default Namespace), soap, http, mime, soapenc (Encoding), soapenv (Envelope), xsi (Instance), xsd, tns (This Namespace)

7.1.3 WSDL 1.1 — โครงสร้างและองค์ประกอบ

  • อิลิเมนต์ราก (Root Element) คือ definitions ซึ่งภายในมีชายน์อิลิเมนต์สำคัญ
  • องค์ประกอบแบ่งเป็น 2 ส่วนหลัก:

- ส่วนแอ็บสแทรกต์ (Abstract) — นิยาม what และ how:

- types (ไทป์): กำหนดชนิดข้อมูลที่รับส่งในเมสเซจ มี 2 แบบคือธรรมดา (simpleType) และซับซ้อน (complexType) สร้างจากชนิดข้อมูลพื้นฐาน เช่น string, decimal, float, double — เป็นหน่วยเล็กที่สุด

- message (เมสเซจ): กำหนดลักษณะข้อความ มีอิลิเมนต์ลูก part ที่ระบุตัวแปร (parameter) และอ้างอิงชนิดข้อมูลไปยัง types

- operation (โอเปอร์เรชัน): กำหนดบริการและรูปแบบรับส่งข้อความ มีลูก 3 อิลิเมนต์: input (ขาเข้า), output (ขาออก), fault (เมื่อผิดพลาด) — รูปแบบรับส่งมี 4 ประเภท: Request-response (นิยมที่สุด: ขอ→ตอบ), One-way (ส่งอย่างเดียว มี input เท่านั้น), Notification (ผู้ให้บริการส่งก่อน มี output เท่านั้น), Solicit-response (ตรงข้าม Request-response: ผู้ให้บริการเริ่มก่อน)

- portType (พอร์ตไทป์): กำหนดอินเตอร์เฟส (Interface) รวบรวมว่ามี operation อะไรให้เรียกใช้บ้าง

- ส่วนคอนกรีต (Concrete) — ระบุ where:

- binding (ไบน์ดิง): คล้าย portType แต่เพิ่มรูปแบบ/รายละเอียดของโพรโทคอล (เช่น SOAP บน HTTP รูปแบบ rpc); มีจำนวน operation เท่ากับใน portType และชื่อตรงกัน

- port (พอร์ต): กำหนดว่าเรียกใช้บริการได้จากที่ใด มีอิลิเมนต์ address ที่มีแอตทริบิวต์ location แสดงยูอาร์แอล (URL)

- service (เซอร์วิส): รวบรวมอิลิเมนต์ port ที่อยู่ภายใต้บริการเดียวกัน (โดยทั่วไปมี 1 อิลิเมนต์)

  • รูปจำลองข้อมูล (Information Model): อิลิเมนต์เล็กสุดเป็นส่วนประกอบของอิลิเมนต์ที่ใหญ่กว่า (types → message → operation → portType และ binding → port → service)

7.1.4 WSDL 2.0

  • พัฒนาต่อจาก 1.1 ไปเป็น 1.2 โดยคงส่วน Abstract และ Concrete ไว้ แต่ยกเลิก/เพิ่ม/เปลี่ยนชื่ออิลิเมนต์หลักบางตัว จึงเปลี่ยนเป็นเวอร์ชัน 2.0
  • โครงสร้างหลักของ 2.0 ประกอบด้วยสกีมา เช่น WSDL, WSDLi, WSDLx, WRPC, WSOAP, WHTTP, XS, XSI
  • เนมสเปซของ WSDL 2.0 ใช้ยูอาร์ไอ http://www.w3.org/ns/wsdl/ (ต่างจาก 1.1)

  • องค์ประกอบ 6 อิลิเมนต์หลักของ WSDL 1.1: types, message, portType, binding, service และอิลิเมนต์รากคือ definitions (จำการแบ่ง Abstract: types/message/operation/portType vs Concrete: binding/port/service)
  • องค์ประกอบพื้นฐาน 3 ส่วน: XML, XSD (สกีมา), xmlns (เนมสเปซ) — และรู้หน้าที่ของแต่ละส่วน
  • ความสัมพันธ์เวอร์ชัน: WSDL 1.1 ↔ SOAP 1.1, WSDL 2.0 ↔ SOAP 1.2 (ออกสอบบ่อย)
  • 5W ของ WSDL: Where/What/How (Functional) vs Why/Who (Non-functional)
  • รูปแบบการรับส่งข้อความ 4 ประเภท: Request-response, One-way, Notification, Solicit-response และจำว่าแต่ละแบบมี input/output อะไรบ้าง
  • หน้าที่ของแต่ละอิลิเมนต์: types (กำหนดชนิดข้อมูล), message+part (กำหนดข้อความ/ตัวแปร), operation (กำหนดบริการ+รับส่ง), portType (อินเตอร์เฟส), binding (โพรโทคอล), port (URL/ที่อยู่), service (รวม port)
  • ปี/ผู้พัฒนา: WSDL 1.0 (2000, IBM+Microsoft+Ariba), W3C รับรอง 1.1 (2001) และ 2.0 (2007)
  • เปรียบเทียบ SOA กับระบบไปรษณีย์: SOAP=ซอง, WSDL=แบบฟอร์ม, Requestor=ผู้ส่ง, Provider=ผู้รับ
  • จุดสำคัญของภาษา XML ใน WSDL: Case Sensitive, ใช้ Unicode, ต้องประกาศ XML Declaration บรรทัดแรก

หน่วยที่ 8: ยูดีดีไอ (UDDI)

ภาพรวม

หน่วยนี้อธิบายยูดีดีไอ (Universal Description, Discovery and Integration: UDDI) ซึ่งเป็น "บริการไดเร็กทอรี" (Directory Service) ที่เป็นการบริการผ่านเว็บประเภทหนึ่ง ทำหน้าที่สำคัญ 2 ประการคือ การประกาศบริการ (Service Publication) และการสืบค้นบริการ (Service Discovery) เปรียบได้กับสำนักทะเบียนบริการ (Service Registry) หรือสมุดหน้าเหลือง เนื้อหาครอบคลุมโครงสร้างเอกสาร XML ของ UDDI องค์ประกอบ 5 อิลิเมนต์ เอพีไอ (API) สำหรับเรียกใช้บริการ ความสัมพันธ์ระหว่าง WSDL กับ UDDI และการพัฒนาด้วยภาษาพีเอชพี จาวา และดอตเน็ต

แนวคิดสำคัญ

  • SOA มีพื้นฐาน 3 บทบาทและ 3 ความสัมพันธ์: ผู้ให้บริการ (Service Provider), ผู้ขอใช้บริการ (Service Requester), สำนักทะเบียนบริการ (Service Registry) เชื่อมกันด้วย Publish (ผู้ให้↔ทะเบียน), Inquiry/Discovery (ผู้ขอใช้↔ทะเบียน), Bind/Invoke (ผู้ให้↔ผู้ขอใช้)
  • UDDI อยู่ลำดับชั้นบนสุด: ต้องรู้จัก UDDI ก่อน → ค้นหาจนได้เอกสาร WSDL → จึงเรียกใช้บริการด้วย SOAP (ผ่าน HTTP หรือ SMTP)
  • UDDI เองก็เป็นการบริการผ่านเว็บ จึงต้องอ่าน WSDL ของ UDDI เพื่อเรียกใช้ แต่เพื่อความสะดวกจึงมี "เอพีไอ" ให้ใช้แทน (1 เอพีไอ = 1 บริการของ UDDI)

เนื้อหาหลัก

8.1.1 บทบาทของ UDDI กับการบริการผ่านเว็บ

  • ที่มา: เริ่มพัฒนา 6 ก.ย. ค.ศ. 2000 โดยองค์กร uddi.org (3 เวอร์ชันแรก) ต่อมาเสนอให้สมาคม OASIS (Organization for the Advancement of Structured Information Standards) รับรอง — เม.ย. 2003 รับรองเวอร์ชัน 2.0, ก.พ. 2005 ประกาศเวอร์ชัน 3.0; มีบริษัทชั้นนำกว่า 300 แห่งสนับสนุน (Microsoft, IBM, SAP, Sun, Oracle, Intel, HP) — หน่วยนี้เน้นเวอร์ชัน 2.0
  • หน้าที่ 2 ประการ: (1) การประกาศบริการ — ผู้ให้บริการประกาศ/แก้ไข/ลบบริการ, (2) การสืบค้นบริการ — ผู้ขอใช้ค้นหาบริการที่ประกาศไว้
  • ประเภท UDDI 2 ชนิด: (1) สาธารณะ (Public) หรือ UBR (UDDI Business Registry) ให้บริการโดยบริษัทต่างๆ ที่ส่งข้อมูลอัพเดตซึ่งกันและกัน แต่ขอบเขตกว้างและความน่าเชื่อถือต่ำ, (2) ส่วนตัว (Private) บนอินเทอร์เน็ต/อินทราเน็ต ใช้เฉพาะกลุ่ม น่าเชื่อถือสูง (เช่น ในองค์กร, e-Market, กลุ่มการเงิน)
  • กลไกการสืบค้น 2 รูปแบบ: แบบตั้งใจ (Focused — ระบุเป้าหมายชัด) และแบบไม่ตั้งใจ (Unfocused — ได้ข้อมูลประกอบเพิ่ม)
  • ประเภทการสืบค้นบริการผ่านเว็บ 3 ประเภท (เรียงจากง่าย/ซับซ้อนน้อย → ยืดหยุ่น/ซับซ้อนมาก):

1. สืบค้นโดยตรง (Direct): ได้ WSDL จากผู้ให้บริการตรงๆ (ผ่าน Disk, Email, FTP, HTTP) — ต้องรู้จักกันก่อน เปรียบเหมือนถามเบอร์โทรโดยตรง

2. สืบค้นโดย WS-Inspection: ผ่านเอกสาร XML ที่รวบรวมลิงก์ของ WSDL ไว้ — เปรียบเหมือนนามบัตร

3. สืบค้นผ่านบริการไดเร็กทอรี (Directory Service): ผ่านตัวกลาง (Third-Party) — เปรียบเหมือนสมุดหน้าเหลือง = UDDI นั่นเอง

8.1.2 โครงสร้างและองค์ประกอบ UDDI

UDDI เป็นเอกสาร XML ประกอบด้วย 5 อิลิเมนต์สำคัญ แบ่งเป็น 3 ส่วน:

  • ส่วนที่ 1: publisherAssertion — กำหนดความสัมพันธ์ระหว่างธุรกิจ (businessEntity) มีอิลิเมนต์ fromKey/toKey (รหัสของ businessEntity); ความสัมพันธ์มี 3 ประเภท: parent-child (แม่-ลูก), peer-peer (กลุ่มเดียวกัน), identity (บริษัทเดียวกัน)
  • ส่วนที่ 2 (แสดงว่ามีบริการอะไร):

- businessEntity — อิลิเมนต์นอกสุด อธิบายรายละเอียดธุรกิจ อ้างอิงด้วย businessKey (ไม่ซ้ำ); ข้อมูลภายในแบ่งเป็น 3 กลุ่ม:

- ไวท์เพจ (White pages): ข้อมูลติดต่อ — ชื่อบริษัท ที่อยู่ เบอร์โทร อีเมล หมายเลข DUNS (อิลิเมนต์ contacts, identifierBag)

- เยลโล่เพจ (Yellow pages): ประเภทบริการที่ให้ (อิลิเมนต์ categoryBag)

- กรีนเพจ (Green pages): วิธีการใช้งานบริการ (อิลิเมนต์ businessService)

- businessService — อธิบายรายละเอียดการให้บริการ อ้างอิงด้วย serviceKey (ไม่ซ้ำ) และมี businessKey อ้างถึง businessEntity

- bindingTemplate — รวบรวมข้อมูลเทคนิคในการเรียกใช้ เช่น URL, วิธีไบน์ดิง อ้างอิงด้วย bindingKey; อิลิเมนต์ accessPoint กำหนดวิธีไบน์ดิง 3 แบบ: SOAP, HTTP, SMTP

  • ส่วนที่ 3: tModel — กำหนดรายละเอียดทางเทคนิค/อินเตอร์เฟส เช่น ที่อยู่ของ WSDL; อ้างอิงด้วย tModelKey; มี overviewDoc (มี overviewURL ชี้ไปยัง WSDL) — สามารถถูกอ้างอิงร่วมกันจากหลาย businessEntity ได้
  • การจัดกลุ่มธุรกิจ 3 ประเภท: NAICS (ตามอุตสาหกรรม), UNSPSC (ตามสินค้า/บริการ), ISO 3166 (ตามภูมิศาสตร์); อิลิเมนต์ identifierBag/categoryBag มี keyedReference (แอตทริบิวต์: keyName, keyValue, tModelKey ที่เป็นรหัส uuid)

8.2.1 เอพีไอของ UDDI

  • API (Application Programming Interface): ส่วนต่อประสานให้โปรแกรมเรียกใช้กันได้ แบ่งเป็น 2 แบบ: ขึ้นกับภาษา (language-dependent) ซึ่งเอพีไอของ UDDI เป็นแบบนี้ และ ไม่ขึ้นกับภาษา (language-independent)
  • เอพีไอของ UDDI รองรับการเรียกแบบ SOAP Binding ด้วย HTTP Post เท่านั้น แบ่งเป็น 2 กลุ่ม (ตาม OASIS):

- กลุ่มประกาศบริการ (Service Publication) — ต้องลงทะเบียน/ตรวจสอบตัวตนก่อนทุกครั้ง: get_authToken (ขอโทเคน), discard_authToken (ยกเลิกโทเคน), add_publisherAssertions, delete_publisherAssertions, get_publisherAssertions, save_business, delete_business, save_service, delete_service, save_binding, delete_binding, save_tModel, delete_tModel

- กลุ่มสืบค้นบริการ (Service Discovery) — ไม่ต้องลงทะเบียน/ไม่มี authInfo: ใช้รูปแบบ find_ (ค้นหาทั่วไป) และ get_ (เรียกดูรายละเอียด): find_business/get_businessDetail, find_service/get_serviceDetail, find_binding/get_bindingDetail, find_tModel/get_tModelDetail

  • ตัวอย่างเช่น find_business มีแอตทริบิวต์ maxRows (จำกัดจำนวนผลลัพธ์), อิลิเมนต์ findQualifiers (กำหนดการเรียงลำดับ), name (ใช้ "%" ค้นแบบ Wildcard ได้); ข้อดีของเอพีไอคือสะดวกกว่าการนำ WSDL ของ UDDI ไปพัฒนาเอง (เพราะ WSDL รวมทุกบริการไว้ในเอกสารเดียว)

8.2.2 ความสัมพันธ์ระหว่าง WSDL และ UDDI

เมื่อผู้ให้บริการประกาศบริการ รายละเอียดใน UDDI จะสอดคล้องกับ WSDL โดยมีความสัมพันธ์:

  • อิลิเมนต์ service ของ WSDL ↔ businessService ของ UDDI
  • อิลิเมนต์ port ของ WSDL ↔ bindingTemplate ของ UDDI
  • อิลิเมนต์ binding ของ WSDL ↔ tModel ของ UDDI
  • ที่อยู่ของเอกสาร WSDL อ้างอิงได้จากอิลิเมนต์ overviewURL ของ UDDI (แอตทริบิวต์ keyValue="wsdlSpec" บ่งบอกว่าเป็นเอกสาร WSDL)
  • UDDI ไม่ได้รองรับเฉพาะ WSDL แต่รองรับเอกสารอื่นได้ด้วย เช่น RDF, RosettaNet, Text

8.3 การพัฒนาและใช้งาน UDDI ด้วย 3 ภาษา

  • พีเอชพี (PHP): เครื่องมือเป็นฟรีแวร์ เรียกเอพีไอว่า "คลาส/ไลบรารี" คลาสยอดนิยม: phpUDDI และ pear (รองรับ UDDI 2.0 สมบูรณ์)
  • จาวา (Java): นิยมในองค์กรขนาดกลาง-ใหญ่ เพราะใช้ได้ทุกแพลตฟอร์มที่มี Java Runtime; จาวาเอพีไอ: UDDI4J (ออกแบบให้แต่ละอิลิเมนต์เป็นคลาสแยก), JAXR (สำหรับ XML เช่น UDDI/WSDL/ebXML), JAX-RPC
  • ดอตเน็ต (.NET): Microsoft พัฒนา UDDI SDK (เรียกเอพีไอว่า "คลาสไลบรารี") รองรับ UDDI 2.0; มี 5 เนมสเปซ: Microsoft.Uddi, Microsoft.Uddi.Businesses, Microsoft.Uddi.Extensions, Microsoft.Uddi.Services, Microsoft.Uddi.TModels

ประเด็นที่ควรเน้นเพื่อสอบ

  • 5 อิลิเมนต์หลักของ UDDI: publisherAssertion, businessEntity, businessService, bindingTemplate, tModel และการแบ่งเป็น 3 ส่วน (ออกสอบแน่)
  • White / Yellow / Green pages: จำให้ได้ว่าแต่ละอย่างให้ข้อมูลอะไร และตรงกับอิลิเมนต์ใด (contacts+identifierBag / categoryBag / businessService)
  • หน้าที่ 2 ประการของ UDDI: การประกาศบริการ + การสืบค้นบริการ
  • ประเภทการสืบค้น 3 ประเภท: Direct, WS-Inspection, Directory Service (UDDI = Directory Service ที่ยืดหยุ่นที่สุด)
  • ความสัมพันธ์ WSDL ↔ UDDI: service↔businessService, port↔bindingTemplate, binding↔tModel, ที่อยู่ WSDL↔overviewURL (ออกสอบบ่อยมาก โดยเฉพาะการจับคู่)
  • เอพีไอ 2 กลุ่ม: ประกาศบริการ (ต้องตรวจสอบตัวตน/มี authInfo/มี save_,delete_) vs สืบค้นบริการ (ไม่ต้องตรวจสอบ/มี find_,get_)
  • ลำดับชั้นการทำงาน: UDDI → WSDL → SOAP (HTTP/SMTP) — UDDI อยู่บนสุด
  • ความสัมพันธ์ 3 ประเภทใน publisherAssertion: parent-child, peer-peer, identity
  • วิธีไบน์ดิง 3 แบบใน accessPoint: SOAP, HTTP, SMTP
  • ข้อจำกัดเอพีไอ: เรียกแบบ SOAP Binding ด้วย HTTP Post เท่านั้น และเป็นเอพีไอแบบขึ้นกับภาษา
  • เครื่องมือแต่ละภาษา: PHP (phpUDDI, pear), Java (UDDI4J, JAXR, JAX-RPC), .NET (UDDI SDK / 5 เนมสเปซ)
  • ปี/องค์กร: uddi.org เริ่ม 2000, OASIS รับรองเวอร์ชัน 2.0 (2003) และ 3.0 (2005)

หน่วยที่ 9: เทคโนโลยีฐานข้อมูลในการบริการผ่านเว็บ

ภาพรวม

หน่วยนี้ชี้ว่า "หัวใจหลักของการบริการผ่านเว็บคือข้อมูล (data)" จึงปูพื้นความรู้ฐานข้อมูล (database) โดยเน้นฐานข้อมูลเชิงสัมพันธ์ (Relational Database) ครอบคลุมแบบจำลองฐานข้อมูล อี-อาร์โมเดล (E-R Model) การใช้ภาษาเอสคิวแอล (SQL) ในการนิยามและจัดการข้อมูล และปิดท้ายด้วยตัวอย่างการประยุกต์ใช้ฐานข้อมูลบนเว็บเซอร์วิสด้วยภาษาจาวา (โปรแกรมโหวตภาษาที่ผู้ใช้ชื่นชอบ ผ่าน JAX-RPC)

แนวคิดสำคัญ

  • ฐานข้อมูล (database): กลุ่มของข้อมูลที่ถูกเก็บรวบรวมและมีความสัมพันธ์ซึ่งกันและกัน ประกอบด้วย ไฟล์/แฟ้มข้อมูล (File), เรคอร์ด/ระเบียน (Record) และฟิลด์/เขตข้อมูล (Field) ถูกจัดการด้วยระบบเดียวกันเพื่อให้ดึงข้อมูลได้รวดเร็ว
  • เว็บเซอร์วิส ใช้ XML ในการสื่อสาร มีอินเทอร์เฟส เช่น WSDL ส่งข้อมูลใน SOAP หรือแนวทาง REST ผ่าน HTTP — ข้อมูลที่ผู้ให้บริการเตรียมไว้มักจัดเก็บในรูปฐานข้อมูล
  • SQL เป็นภาษามาตรฐานและระบบเปิด (open system) ใช้คำสั่งเดียวกันกับฐานข้อมูลชนิดใดก็ได้และได้ผลลัพธ์เหมือนกัน จึงไม่ยึดติดฐานข้อมูลใดฐานข้อมูลหนึ่ง

เนื้อหาหลัก

9.1.1 ความรู้เบื้องต้นเกี่ยวกับฐานข้อมูลเชิงสัมพันธ์

  • แบบจำลองฐานข้อมูล (Database Model) 4 แบบ:

1. ไฮราคี/ลำดับชั้น (Hierarchical): โครงสร้างต้นไม้คว่ำ มีเรคอร์ดพ่อแม่ (parent) และลูก (child) — พ่อแม่มีลูกได้หลายเรคอร์ด แต่ลูกมีพ่อแม่ได้เพียงหนึ่ง

2. เครือข่าย (Network): เชื่อมโยงข้อมูลเป็นชุด ความสัมพันธ์แบบใดก็ได้ (1:1, 1:กลุ่ม, กลุ่ม:กลุ่ม)

3. ออบเจ็กต์ (Object-oriented): มองทุกสิ่งเป็นออบเจ็กต์ มีคลาส (class) กำหนดคุณสมบัติ

4. เชิงสัมพันธ์ (Relational): นิยมที่สุด ใช้ตารางความสัมพันธ์ (รีเลชัน/Relation) แบบสองมิติ มีแถว (row) และคอลัมน์ (column)

  • องค์ประกอบของรีเลชัน: สคีมา/เค้าร่าง (Schema = โครงสร้าง) และอินสแตน (instance = ข้อมูลที่บันทึก ณ ขณะหนึ่ง)
  • คำศัพท์รีเลชัน: รีเลชัน/ตาราง (relation/table), ทัพเพิล (tuple = แต่ละแถว), แอตทริบิวต์ (attribute = คอลัมน์/ฟิลด์), ดีกรี (degree) = จำนวนแอตทริบิวต์, คาร์ดินัลลิตี้ (cardinality) = จำนวนแถว, โดเมน (domain) = ขอบเขต/ชนิดข้อมูลที่เป็นไปได้ของแอตทริบิวต์

9.1 องค์ประกอบของแบบจำลองข้อมูลเชิงสัมพันธ์ (คีย์)

  • ซูเปอร์คีย์ (superkey): เซตของแอตทริบิวต์ที่กำหนดความเป็นหนึ่งเดียวของแถวได้ (อาจเป็นคีย์เดี่ยวหรือคีย์ผสม)
  • คีย์คู่แข่ง (candidate key): ซูเปอร์คีย์ที่มีแอตทริบิวต์น้อยที่สุดเท่าที่เป็นไปได้
  • คีย์หลัก (primary key): คีย์คู่แข่งตัวเดียวที่ถูกเลือกเป็นตัวกำหนดความเป็นหนึ่งเดียว — ห้ามซ้ำ, ห้ามเป็น null, อาจเป็นหนึ่งหรือหลายแอตทริบิวต์
  • คีย์รอง (secondary key): คีย์คู่แข่งที่ไม่ใช่คีย์หลัก ใช้เป็นคีย์เสริมในการค้นหา
  • คีย์นอก (foreign key): เซตของแอตทริบิวต์ที่มีค่าตรงกับคีย์หลักในตารางอื่น ใช้เชื่อมโยง 2 ตาราง

9.1 กฎของคอดด์ (Codd's 12 Rules)

Edgar Frank Codd ผู้คิดค้น Relational Model ตั้งกฎ 12 ข้อสำหรับผลิตภัณฑ์ฐานข้อมูลเชิงสัมพันธ์ ได้แก่: (1) กฎด้านสารสนเทศ, (2) การรับประกันการเข้าถึง, (3) การปฏิบัติกับค่า null อย่างมีแบบแผน, (4) ออนไลน์แค็ตตาล็อกแบบไดนามิก, (5) ด้านภาษาข้อมูล, (6) การปรับปรุงวิว, (7) การแทรก/ปรับปรุง/ลบระดับสูง, (8) ความอิสระของข้อมูลเชิงกายภาพ, (9) ความอิสระของข้อมูลเชิงตรรกะ, (10) ความคงสภาพที่เป็นอิสระ, (11) การกระจายที่เป็นอิสระ, (12) ปราศจากเวอร์ชันย่อย

9.1.2 อี-อาร์โมเดล (E-R Model)

อี-อาร์โมเดล (Entity-Relationship Model) เป็นแบบจำลองแสดงความสัมพันธ์ระหว่างข้อมูล นำเสนอเป็นแผนภาพ อี-อาร์ไดอะแกรม (E-R diagram) มีองค์ประกอบ:

  • เอ็นทิตี้ (Entity): สิ่งที่สนใจเก็บข้อมูล ใช้สัญลักษณ์ สี่เหลี่ยมผืนผ้า
  • แอตทริบิวต์ (Attribute): คุณลักษณะของเอ็นทิตี้ ใช้สัญลักษณ์ รูปไข่ เชื่อมด้วยเส้นตรง; ประเภท: คีย์แอตทริบิวต์ (Primary key — ขีดเส้นใต้), ซิมเพิ้ลแอตทริบิวต์ (Simple — แบ่งย่อยไม่ได้)
  • ความสัมพันธ์ (Relationship): ความสัมพันธ์ระหว่างเอ็นทิตี้ ใช้สัญลักษณ์ สี่เหลี่ยมข้าวหลามตัด (diamond) ตั้งชื่อด้วยคำกริยา
  • คาร์ดินอลิตี้ คอนสเทรน (Cardinality Constraints) 4 ชนิด: 0..* (ไม่มีหรือมากกว่าหนึ่ง), 0..1 (อย่างมากหนึ่ง), 1..* (อย่างน้อยหนึ่ง), 1..1 (เพียงหนึ่งเดียว)
  • การแปลงอี-อาร์ไดอะแกรมเป็นตารางความสัมพันธ์:

- แปลงเอ็นทิตี้: 1 เอ็นทิตี้ → 1 ตาราง, แอตทริบิวต์→แอตทริบิวต์ของตาราง, คีย์→คีย์ของตาราง

- แปลงความสัมพันธ์: ถ้าทั้งสองฝั่งเป็น 0..* หรือ 1..* ให้สร้างตารางใหม่; ถ้ามีฝั่งเดียวเป็น 0..*/1..* ให้นำคีย์ฝั่งนั้นไปเป็นแอตทริบิวต์ของฝั่งตรงข้าม; ถ้าเป็น 0..1/1..1 ทั้งคู่ ให้นำคีย์ฝั่งใดก็ได้ไปเป็นแอตทริบิวต์อีกฝั่ง

9.2.1 ภาษานิยามข้อมูล (DDL)

เอสคิวแอล (SQL: Structured Query Language) มีโปรแกรมรองรับเช่น MySQL (Open Source ยอดนิยม, ใช้ได้ทั้ง Linux/Windows), MsSQL, PostgreSQL, MS Access; ประเภทคำสั่ง SQL 3 ประเภท:

  • DDL (Data Definition Language): นิยามโครงสร้าง เช่น CREATE, ALTER, DROP
  • DML (Data Manipulation Language): จัดการข้อมูล เช่น เพิ่ม ลบ แก้ไข เรียกใช้
  • DCL (Data Control Language): ควบคุมสิทธิ์ เช่น อนุญาต/ยกเลิกการเข้าถึง
  • ชนิดข้อมูล: ตัวอักษร (char(n) ความยาวคงที่, varchar(n) ไม่คงที่ สูงสุด 4,000), ตัวเลข (decimal/dec(m,n), int/integer, number(n))
  • คำสั่ง DDL สำคัญ:

- CREATE TABLE สร้างตาราง พร้อมข้อกำหนด (constraints): NOT NULL (ห้ามว่าง), UNIQUE (ห้ามซ้ำ), PRIMARY KEY (คีย์หลัก — กำหนดคอลัมน์เดียวหรือหลายคอลัมน์ได้), FOREIGN KEY ... REFERENCES (คีย์นอก เชื่อมโยงตาราง)

- ALTER TABLE เปลี่ยนแปลงโครงสร้าง (เพิ่ม/ลบ/เปลี่ยนคอลัมน์)

- DROP TABLE ลบโครงสร้างตาราง (มี CASCADE Constraints ลบข้อจำกัดที่อ้างอิงทั้งหมด)

9.2.2 ภาษาจัดการข้อมูล (DML)

  • INSERT INTO ชื่อตาราง ... VALUES ... — เพิ่มข้อมูล (ได้ทั้งทุกคอลัมน์หรือบางคอลัมน์)
  • UPDATE ชื่อตาราง SET คอลัมน์=ค่า WHERE <เงื่อนไข> — ปรับปรุงข้อมูลทุกแถวที่ตรงเงื่อนไข
  • DELETE FROM ชื่อตาราง WHERE <เงื่อนไข> — ลบข้อมูลทุกแถวที่ตรงเงื่อนไข
  • การเรียกค้น (query) ด้วย SELECT: SELECT * FROM ตาราง (ทุกคอลัมน์), SELECT คอลัมน์ FROM ตาราง (บางคอลัมน์), SELECT ... FROM ... WHERE <เงื่อนไข> (มีเงื่อนไข), เลือกจากหลายตารางพร้อมเชื่อมคีย์นอกได้

9.3 การประยุกต์ใช้ฐานข้อมูลบนเว็บเซอร์วิสด้วยจาวา

  • โมเดลเว็บเซอร์วิส 3 องค์ประกอบ: ผู้ให้บริการ (Service Provider), ผู้ขอใช้บริการ (Service Requestor), นายทะเบียนการให้บริการ (Service Repository/Registry) — ทำงานเป็นขั้น: create → describe (WSDL) → publish (SOAP) → find (SOAP) → retrieve → bind (SOAP)
  • ตัวอย่าง: โปรแกรมโหวตภาษาที่ผู้ใช้ชื่นชอบ พัฒนาด้วย JAX-RPC (ใช้ SOAP เป็น application protocol, HTTP เป็น transport protocol) แบ่งเป็น 2 ฝั่ง (ไคลเอ็นต์/เซิร์ฟเวอร์) เครื่องมือ xrpcc สร้างเอกสาร WSDL จากคลาส
  • 4 ขั้นตอนการพัฒนา: (1) เซอร์วิสอินเทอร์เฟส (Service Interface) — นิยามวิธีที่ไคลเอ็นต์เรียกใช้ (extend java.rmi.remote), (2) เซอร์วิสอิมพลีเมนต์ (Service Implementation) — คลาส VoteImpl ที่เป็นเว็บเซอร์วิสเอ็นพอยต์ เชื่อมต่อฐานข้อมูล, (3) ดีพลอย (Deploy) — นำขึ้น Apache Tomcat ผ่าน Deployment Descriptor (Web.xml), (4) ไคลเอ็นต์แอปพลิเคชัน (Client Application) — โปรแกรมฝั่งไคลเอ็นต์ที่ติดต่อผู้ใช้

  • องค์ประกอบและสัญลักษณ์ของอี-อาร์ไดอะแกรม: เอ็นทิตี้(สี่เหลี่ยมผืนผ้า), แอตทริบิวต์(รูปไข่), ความสัมพันธ์(ข้าวหลามตัด), คีย์(ขีดเส้นใต้)

  • กฎการแปลงอี-อาร์ไดอะแกรมเป็นตาราง: โดยเฉพาะเงื่อนไขว่าเมื่อใดสร้างตารางใหม่ เมื่อใดนำคีย์ไปฝากอีกฝั่ง
  • ประเภทคำสั่ง SQL 3 ประเภท: DDL (CREATE/ALTER/DROP), DML (INSERT/UPDATE/DELETE/SELECT), DCL — จับคู่คำสั่งกับประเภทให้ถูก
  • constraints ใน CREATE TABLE: NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY...REFERENCES
  • รูปแบบคำสั่ง SQL พื้นฐาน: เขียน/อ่าน INSERT, UPDATE...WHERE, DELETE...WHERE, SELECT...FROM...WHERE ได้ (มักให้หาที่ผิดหรือเขียนคำสั่ง)
  • แบบจำลองฐานข้อมูล 4 แบบ: ลำดับชั้น/เครือข่าย/ออบเจ็กต์/เชิงสัมพันธ์ (เชิงสัมพันธ์นิยมที่สุด)

  • 3 องค์ประกอบเว็บเซอร์วิส: Service Provider, Service Requestor, Service Repository และจำว่า "หัวใจของการบริการผ่านเว็บคือข้อมูล"
  • ชนิดข้อมูล SQL: char(n) vs varchar(n), decimal(m,n) ความหมายของ m และ n

หน่วยที่ 10: การพัฒนาและใช้งานการบริการผ่านเว็บ

ภาพรวม

หน่วยนี้ปูพื้นฐานการนำการบริการผ่านเว็บ (Web Service: WS) ไปใช้จริงในองค์กร อธิบายความแตกต่างระหว่างเว็บแอปพลิเคชัน (Web Application) กับเว็บเซอร์วิส แนะนำเครื่องมือพัฒนา 3 กลุ่ม (ดอตเน็ต จาวา โอเพ่นซอร์ส) แล้วลงรายละเอียดวิธีพัฒนาโปรแกรมทั้งฝั่งผู้ให้บริการและผู้ใช้บริการด้วยภาษา PHP สุดท้ายอธิบาย "แพตเทิร์น" การพัฒนา 2 แบบ คือ แบบล่างขึ้นบน (Bottom-up) และแบบบนลงล่าง (Top-down) โดยมีไฟล์ WSDL เป็นหัวใจ

แนวคิดสำคัญ

ทั้งเว็บแอปพลิเคชันและเว็บเซอร์วิสต่างใช้โพรโทคอล HTTP และอินเทอร์เน็ตเป็นช่องทางสื่อสารเหมือนกัน แต่ วัตถุประสงค์ต่างกัน: เว็บแอปพลิเคชันพัฒนามาให้ "คน" ใช้งานติดต่อกับคอมพิวเตอร์ผ่านแอปพลิเคชัน (แสดงผลเป็น HTML ผ่านเบราว์เซอร์) ส่วนเว็บเซอร์วิสพัฒนามาให้ "แอปพลิเคชันอื่น" เรียกใช้งานเพื่อแลกเปลี่ยนข้อมูลระหว่างกัน (ส่งข้อมูลกลับเป็น XML) ทั้งสองเป็นไปตามแนวคิดการประมวลผลแบบกระจาย (Distributed Processing) จุดเด่นของเว็บเซอร์วิสคือทำให้การเรียกใช้ทรัพยากรข้ามองค์กร/ข้ามแพลตฟอร์มสะดวกขึ้น เพราะสื่อสารผ่านเอกสาร XML ที่เกือบทุกระบบรองรับ

เนื้อหาหลัก

10.1 ความรู้พื้นฐานเกี่ยวกับการพัฒนาและใช้งานการบริการผ่านเว็บ

  • รูปแบบการนำ WS มาใช้ 2 แบบ: (1) การแลกเปลี่ยนข้อมูล (Exchange) — ทุกฝ่ายเป็นทั้งผู้ให้และผู้ใช้บริการ (2) การให้บริการข้อมูล (Provide) — แยกบทบาทผู้ให้/ผู้ใช้ชัดเจน องค์กรทั่วไปมักมีบทบาทเป็น "ผู้ให้บริการข้อมูล"
  • ตัวอย่างจริง: ต่างประเทศมี eBay (ฟังก์ชัน FindProducts ค้นหาสินค้า) และ Amazon (มีไลบรารีรองรับ Java, .Net, PHP); ในไทยมี ปตท., กรมสรรพากร (e-Revenue UDDI, ใช้ PHP+NuSOAP), กระทรวงพาณิชย์, กรมอนามัย ส่วนการแลกเปลี่ยนข้อมูลข้ามองค์กรมีโครงการ Th e-GIF (Thailand Electronic Government Interoperability Framework) สำหรับระบบสารบรรณอิเล็กทรอนิกส์
  • เครื่องมือพัฒนา 3 กลุ่ม: (1) ดอตเน็ต (.NET) (2) จาวา (Java) (3) โอเพ่นซอร์ส (เช่น PHP) โดยทั่วไปประกอบด้วย ภาษาพัฒนา + ซอฟต์แวร์ฐานข้อมูล + เว็บเซิร์ฟเวอร์ + ชุดเทมเพลต

- ดอตเน็ต: ทำงานบน Common Language Runtime (CLR) ซึ่งเป็น Virtual Machine โค้ดถูกคอมไพล์เป็นภาษากลาง CIL (Common Intermediate Language) แล้วแปลด้วย JIT (Just-In-Time) ต่อมาขยายเป็น WCF (Windows Communication Foundation) รองรับหลายโพรโทคอล (HTTP, TCP, MSMQ, WS-HTTP); เครื่องมือคือ Visual Studio, WebMatrix; ทำงานร่วมกันได้ตามมาตรฐาน WS-I Basic Profile

- จาวา: ทำงานบน JVM (Java Virtual Machine) ผ่าน JRE เริ่มจาก J2EE 1.4 ใช้ JAX-RPC ต่อมา Java 6 รองรับทั้ง SOAP และ REST

- โอเพ่นซอร์ส: เฟรมเวิร์กจาวา เช่น Metro, Axis, Spring-WS, Eclipse; ภาษายอดนิยมคือ PHP (PHP: Hypertext Preprocessor)

  • REST (Representational State Transfer) vs SOAP: REST เป็นแนวคิดเชิงสถาปัตยกรรม มองทุกอย่างเป็น "ทรัพยากร" อ้างอิงด้วย URI ใช้คำสั่ง HTTP มาตรฐาน ได้แก่ GET (เรียกดู), POST (เพิ่ม), PUT (แก้ไข), DELETE (ลบ) แลกเปลี่ยนข้อมูลเป็น XML ส่วน SOAP เน้นการจัดการ "กิจกรรม" (activity) — Amazon, eBay, Google ใช้ REST

10.2 การพัฒนาโปรแกรมการบริการผ่านเว็บ

  • สภาพแวดล้อมการพัฒนา: ต้องมี (1) เครื่องเซิร์ฟเวอร์ที่ติดตั้งเว็บเซิร์ฟเวอร์ (2) เครื่องสำหรับพัฒนา (3) ซอฟต์แวร์ช่วยพัฒนา (มีหรือไม่ก็ได้) ตัวอย่างใช้ AppServ ซึ่งรวม Apache (เว็บเซิร์ฟเวอร์) + ตัวแปล PHP + MySQL + phpMyAdmin
  • 3 วิธีพัฒนา WS ด้วย PHP:

1. ใช้ไลบรารีเสริม — เช่น NuSOAP, Pear SOAP (สร้างไฟล์ WSDL อัตโนมัติได้)

2. เรียกฟังก์ชันระยะไกล (RPC) — ใช้ XML-RPC ผ่านไลบรารี IXR_Library

3. ไม่ใช่ SOAP (REST) — ไม่ต้องติดตั้งไลบรารีเสริม ใช้ขีดความสามารถของ HTTP/HTML โดยตรง รับค่าผ่าน $_GET

  • ขั้นตอนฝั่งผู้ให้บริการ (nuSOAP, 5 ขั้น): เรียกใช้ไลบรารี → ระบุฟังก์ชันที่ให้บริการ → สร้างตัวแทนเซิร์ฟเวอร์ → ลงทะเบียนฟังก์ชัน → ตรวจสอบคำขอและให้บริการ
  • ขั้นตอนฝั่งผู้ใช้บริการ (4 ขั้น): เรียกใช้ไลบรารี → สร้างผู้ขอใช้บริการ → เรียกใช้บริการ → แสดงผล (การเขียนฝั่งผู้ใช้ง่ายกว่าฝั่งผู้ให้บริการ)

10.3 แพตเทิร์นการพัฒนาการให้บริการผ่านเว็บ

  • แบบล่างขึ้นบน (Bottom-up): เริ่มจาก "เขียนโค้ดก่อน" แล้วใช้เครื่องมือแปลงโค้ดเป็นไฟล์ WSDL เหมาะกับการใช้ประโยชน์จากโปรแกรมที่มีอยู่แล้ว เครื่องมือแปลง PHP→WSDL จะระบุ "ชนิดข้อมูล (type)" จากคอมเมนต์ในโค้ด (เพราะ PHP ไม่ได้ประกาศชนิดตัวแปร)
  • แบบบนลงล่าง (Top-down): เริ่มจากการ "ออกแบบความต้องการ/คลาส" ก่อน → สร้างไฟล์ WSDL → แล้วทั้งฝั่งผู้ให้และผู้ใช้บริการสร้างโค้ดจาก WSDL (พัฒนาคู่ขนานได้) ใช้เครื่องมือ เช่น WSDL2PHP สร้าง "โครงโปรแกรม (stub)" ที่มีคลาส/เมธอด/พารามิเตอร์ครบ แต่ยังไม่มีตรรกะ (นักพัฒนาเติม //TODO เอง) — stub ใช้ทดสอบระบบเป็นส่วนๆ ได้
  • ไฟล์ WSDL คือหัวใจ ของการพัฒนา WS; สร้างได้ด้วย editor พื้นฐาน (Notepad) หรือเครื่องมือกราฟิก เช่น Altova XML; คำสั่งสร้างฝั่งผู้ให้บริการใช้ตัวเลือก -s ส่วนฝั่งผู้ใช้ไม่ต้องมี

ประเด็นที่ควรเน้นเพื่อสอบ

  • ความแตกต่างเว็บแอปพลิเคชัน vs เว็บเซอร์วิส (วัตถุประสงค์: คนใช้ vs แอปฯ เรียกใช้; ผลลัพธ์: HTML vs XML) — ออกสอบบ่อยมาก
  • รูปแบบการนำ WS มาใช้ 2 แบบ (แลกเปลี่ยนข้อมูล vs ให้บริการข้อมูล) และข้อแตกต่างเรื่องบทบาทผู้ให้/ผู้ใช้
  • เครื่องมือพัฒนา 3 กลุ่ม (ดอตเน็ต/จาวา/โอเพ่นซอร์ส) และองค์ประกอบของชุดเครื่องมือ
  • REST vs SOAP: คำสั่ง HTTP 4 ตัว (GET/POST/PUT/DELETE) และความหมาย, แนวคิด "ทุกอย่างคือทรัพยากร/อ้างด้วย URI"
  • 3 วิธีพัฒนา WS ด้วย PHP (ไลบรารีเสริม / RPC-XMLRPC / REST) และไลบรารีที่ต้องใช้ (REST ไม่ต้องติดตั้งไลบรารี)
  • ขั้นตอนพัฒนาฝั่งผู้ให้บริการด้วย nuSOAP (5 ขั้นตอน) และฝั่งผู้ใช้ (4 ขั้นตอน)
  • แพตเทิร์น Bottom-up vs Top-down — จุดเริ่มต่างกัน (โค้ดก่อน vs WSDL ก่อน) และบทบาทของ WSDL/stub
  • ข้อสังเกต: เครื่องมือแปลง PHP→WSDL ระบุชนิดข้อมูลจาก "คอมเมนต์" ในโค้ด
  • บทบาทของ CLR/CIL/JIT ฝั่ง .NET และ JVM/JRE ฝั่ง Java (ทั้งคู่ทำหน้าที่เป็น Virtual Machine/ภาษากลาง)

หน่วยที่ 11: การรักษาความปลอดภัยในการให้บริการผ่านเว็บ

ภาพรวม

หน่วยนี้ว่าด้วยการรักษาความปลอดภัย (Security) ของการบริการผ่านเว็บแบบครบวงจร เริ่มจากเป้าหมายพื้นฐานและการสร้างแบบจำลองภัยคุกคาม (Threat Modeling) ตามแนวทาง STRIDE ของไมโครซอฟต์ ต่อด้วยเกณฑ์วิเคราะห์ช่องโหว่ (เช่น CVSS) แล้วเจาะลึก มาตรฐานความปลอดภัย ที่ต้องรู้ ได้แก่ SSL/TLS, XML Encryption, XML Signature, SAML, WS-Security และมาตรฐานเกี่ยวกับกุญแจ/ความไว้วางใจ ปิดท้ายด้วยการประยุกต์ใช้จริงและรูปแบบการโจมตีระบบ WS

แนวคิดสำคัญ

การรักษาความปลอดภัยของ WS ไม่สามารถป้องกันเฉพาะผู้ให้บริการได้ ต้องครอบคลุมทุกองค์ประกอบที่เกี่ยวข้องตลอดเส้นทาง ตั้งแต่เครื่องแม่ข่ายเว็บ ตัวประมวลผล HTTP ตัวแปลเอกสาร XML ตัวประมวลผล SOAP และโปรแกรมประยุกต์ทั้งสองฝั่ง รูปแบบการรักษาความปลอดภัยพื้นฐานที่ต้องจำคือ CIA = การคงสภาพข้อมูล (Integrity), การรักษาความลับ (Confidentiality), ความพร้อมใช้งาน (Availability) ความท้าทายสำคัญคือสภาพแวดล้อมแบบกระจายและพลวัตที่ไม่มีใครเป็นเจ้าของระบบทั้งหมด ทำให้ต้องอาศัย "มาตรฐาน" เพื่อให้ระบบต่างแพลตฟอร์มทำงานสอดประสานกันได้ และต้องป้องกันแบบ ต้นทางถึงปลายทาง (end-to-end) ไม่ใช่แค่จุดต่อจุด (point-to-point)

เนื้อหาหลัก

11.1 ความรู้พื้นฐานและการสร้างแบบจำลองภัยคุกคาม

  • เป้าหมายความปลอดภัย 3 ข้อ: (1) มีกลไก/เครื่องมือสำหรับคงสภาพและรักษาความลับข้อมูล (2) ให้บริการเฉพาะคำขอที่เป็นไปตามนโยบาย (3) มั่นใจว่าข้อมูลที่ผู้ขอใช้ถูกต้องและเป็นจริง
  • รูปแบบพื้นฐาน (CIA): Integrity (ข้อความไม่ถูกแก้ระหว่างส่ง), Confidentiality (ดักดูเนื้อหาไม่ได้), Availability (ส่งถึงผู้รับตามสิทธิทันที)
  • แบบจำลองการคุกคาม (Threat Modeling): เป็น "กระบวนการทำซ้ำๆ (iterative)" เริ่มตั้งแต่ออกแบบและต่อเนื่องตลอดวงจรชีวิต — เพราะระบุภัยคุกคามทั้งหมดในครั้งเดียวไม่ได้ และต้องรองรับการเปลี่ยนแปลง ไมโครซอฟต์เป็นผู้เสนอแนวคิดนี้ก่อน
  • ที่มาของช่องโหว่ 5 ข้อ: ออกแบบผิด, ประยุกต์ใช้ไม่ถูกต้อง, ดำเนินงานผิด/ขาดการควบคุม, วิธีที่ช่องโหว่ถูกใช้โจมตี, มาตรการป้องกันไม่เหมาะสม
  • ขั้นตอนสร้างแบบจำลอง 7 ขั้น: ระบุทรัพยากรที่มีค่า → กำหนดวัตถุประสงค์ความปลอดภัย → สร้างภาพรวมสถาปัตยกรรม → สร้างโปรไฟล์ความปลอดภัย → ระบุภัยคุกคาม/ช่องโหว่ → จัดทำเอกสาร → จัดอันดับภัยคุกคาม
  • STRIDE (ประเภทภัยคุกคามของไมโครซอฟต์):

- Spoofing — ปลอมแปลงตัวตน (เช่น e-mail spoofing, IP spoofing)

- Tampering — เปลี่ยนแปลง/แก้ไขข้อมูลโดยไม่ได้รับอนุญาต (เช่น แก้ค่า hidden field ราคาสินค้า)

- Repudiation — ปฏิเสธความรับผิดชอบ (รับข้อมูลแล้วปฏิเสธว่าไม่ได้รับ)

- Information Disclosure — เปิดเผยข้อมูลให้ผู้ไม่มีสิทธิ์ (เช่น เห็นเลขบัตรเครดิต)

- Denial of Service — ทำให้ระบบให้บริการไม่ได้ (ส่งคำขอปลอมจำนวนมาก)

- Elevation of Privilege — ยกระดับสิทธิตัวเอง

- ประเมินความเสี่ยง = ความน่าจะเป็นของภัยคุกคาม × ความเสียหายที่อาจเกิด

  • มาตรการตอบโต้ (Countermeasure) แบ่งตามช่วงวงจรชีวิต: ช่วงพัฒนา (code analyzer, สแกนช่องโหว่), ช่วงติดตั้ง (กำหนดค่า config), ช่วงทำงาน (IDS/IDP — Intrusion Detection/Prevention)
  • เกณฑ์วิเคราะห์ช่องโหว่: (1) ไมโครซอฟต์ — เชิงคุณภาพ 4 ระดับ (ร้ายแรง/สำคัญ/ปานกลาง/ต่ำ) (2) US-CERT — เชิงปริมาณ คะแนน 0-180 (ค่ามากกว่า 40 ได้รับความสนใจ) (3) FIRST — พัฒนา CVSS (Common Vulnerability Scoring System) มี 3 กลุ่มเกณฑ์: พื้นฐาน (Base), เวลา (Temporal), สิ่งแวดล้อม (Environmental)

11.2 มาตรฐานการรักษาความปลอดภัย

  • กรอบงาน: เสนอครั้งแรกโดยไมโครซอฟต์ + ไอบีเอ็ม; หลักเกณฑ์พัฒนา 4 ข้อ — ต้องเป็นอิสระจากเทคโนโลยี, ประกอบกันได้, จัดเป็นลำดับชั้น, ขยายได้
  • ที่มาของมาตรฐาน 3 ประเภท: De facto (เกิดจากการใช้งานยอมรับกันเอง), De jure (กำหนดโดยองค์กรกฎหมาย เช่น ISO, ANSI), Consortium recommendation (ข้อเสนอจากกลุ่มบริษัท เช่น W3C)
  • มาตรฐานสำคัญที่ต้องรู้:

1. SSL/TLS (Secure Sockets Layer / Transport Layer Security) — มาตรฐาน de facto พัฒนาโดย Netscape ทำงานระหว่างชั้น Network กับ Application ให้: ความลับ (เข้ารหัสสมมาตร เช่น DES, RC4), คงสภาพ (MAC + MD5), พิสูจน์ตัวตน (certificate + กุญแจสาธารณะ) ข้อจำกัด: ปลอดภัยแค่ "จุดต่อจุด" — โหนดตรงกลางถอดรหัสและแก้ข้อมูลได้ และเข้ารหัสบางส่วนของข้อมูลไม่ได้

2. การรักษาความปลอดภัยข้อมูล XML:

- XML Encryption (การเข้ารหัสเอ็กซ์เอ็มแอล) — W3C แนะนำ ให้ความลับระดับ Application แบบ end-to-end แม้ผ่านหลายโหนด; แยกข้อมูลคีย์/อัลกอริทึมออกจากข้อมูลที่เข้ารหัส แต่ไม่รับประกันการคงสภาพ จึงต้องใช้ลายเซ็นประกอบ; เสี่ยงต่อ plaintext guessing และ DoS จากการเข้ารหัสซ้ำ (recursive)

- XML Signature (ลายเซ็นเอ็กซ์เอ็มแอล) — ข้อกำหนดร่วม W3C + IETF ให้การคงสภาพ + เชื่อมโยงตัวตนผู้ลงนาม (ป้องกันการปฏิเสธ) แทนที่ PKCS#7 เดิมที่เซ็นเฉพาะส่วนไม่ได้; ลายเซ็นเดียวเซ็นได้หลายเอกสาร แต่ลำพังไม่พอ เพราะไม่กำหนดว่าคีย์สัมพันธ์กับบุคคลอย่างไร

3. SAML (Security Assertions Markup Language) — ภาษามาร์กอัปตรวจสอบเงื่อนไขความปลอดภัย กำหนดโดย OASIS เป็นเฟรมเวิร์ก XML แลกเปลี่ยนข้อมูลพิสูจน์ตัวตน/กำหนดสิทธิ มีเงื่อนไข (Assertion) 3 ประเภท: Authentication (พิสูจน์ตัวตน), Attribute (คุณสมบัติ), Authorization (กำหนดสิทธิ); รองรับ SSO (Single Sign-On) — ล็อกอินครั้งเดียวใช้ได้หลายเว็บไซต์; SAML เองไม่รักษาความลับ ต้องใช้ XML Signature + SSL/TLS ประกอบ

4. การรักษาความปลอดภัยข้อความ SOAP — WS-Security:

- WS-Security — มาตรฐาน de facto/OASIS (อนุมัติ มิ.ย. 2002) ระบุส่วนขยายของ SOAP ให้เลือก "บางส่วน" ของข้อความมาเข้ารหัส/ลงนามได้ ใช้ XML Encryption (ความลับ) + XML Signature (คงสภาพ); รองรับ โทเค็นความปลอดภัย (Security Token) หลายแบบ เช่น ใบรับรอง X.509, ตั๋ว Kerberos, SAML — ป้องกันแบบ end-to-end ผ่านหลายฮอป (multi-hop)

- WS-SecureConversation — ขยายจาก WS-Security + WS-Trust ให้รักษาความปลอดภัยการแลกเปลี่ยน "หลายข้อความ" ในเซสชันเดียว โดยสร้าง/แบ่งปันบริบทความปลอดภัย (Security Context)

- WS-Reliability — รับประกันการส่งข้อความเชื่อถือได้ มี 4 ระดับการประกัน: AtMostOnce (มากสุดหนึ่งครั้ง), AtLeastOnce (อย่างน้อยหนึ่งครั้ง), ExactlyOnce (หนึ่งครั้งเท่านั้น), InOrder (ตามลำดับ)

5. มาตรฐานเกี่ยวกับกุญแจและการจัดการความไว้วางใจ:

- PKI (Public Key Infrastructure) + CA (Certification Authority) — โครงสร้างพื้นฐานกุญแจสาธารณะ ครอบคลุมการสร้าง/เก็บ/แจกจ่าย/ใช้/ยกเลิกกุญแจ

- WS-Trust — มาตรฐาน OASIS ขยายจาก WS-Security กำหนดกรอบขอ/แจกจ่าย/ประเมิน/ต่ออายุ/ยกเลิกโทเค็น ผ่านบริการ STS (Security Token Service) ซึ่งอาจทำงานบน Kerberos KDC หรือ PKI

- มาตรฐานเสริมในกรอบ WS-* อื่นๆ: WS-Policy, WS-Privacy, WS-Federation, WS-Authorization

  • ข้อพิจารณา (11.2.3): ปัญหามี 2 ด้าน — เทคนิค (ภัยคุกคามจากตัวมาตรฐานเอง, การทำงานร่วมกัน, ประสิทธิภาพ) และการจัดการ (ค่าใช้จ่าย, การฝึกอบรม) แก้ปัญหา interoperability ด้วยองค์กร WS-I (Web Services Interoperability Organization) ที่ใช้ "โปรไฟล์"; แก้ปัญหาประสิทธิภาพการส่งข้อมูลไบนารีด้วย XOP, MTOM, RRSHB; และใช้ อุปกรณ์ XML (XML Accelerator, XML Firewall) เพื่อเพิ่มประสิทธิภาพและกรองเนื้อหา

11.3 การประยุกต์ใช้และการโจมตีระบบ

  • มาตรฐานมี 2 แบบ: เปิด (หลายฝ่ายร่วมพัฒนา ใช้เสรี) และปิด (เฉพาะองค์กร) องค์ประกอบความปลอดภัยจริงต้องครอบคลุม คน + กระบวนการ + เทคโนโลยี
  • ไมโครซอฟต์: ร่วมพัฒนามาตรฐานเปิด (Code Signing, IETF-TLS, JEPI, SET) ออกชุด WSE (Web Service Enhancement) เสริมความปลอดภัย .NET รองรับ WS-Addressing, WS-Security, WS-Trust ฯลฯ
  • จาวา: รองรับการเข้ารหัสหลายแบบ (RSA, DSA, AES, Triple DES, SHA, RC2, RC4) ผ่าน JCA (Java Cryptography Architecture); รองรับ X.509, OCSP, PKCS; การเข้าถึง/พิสูจน์ตัวตนผ่าน JAAS (Java Authentication and Authorization Service) และ SSO
  • รูปแบบการโจมตี 5 แบบ (ออกสอบบ่อย):

1. Buffer Overflow — ส่งข้อมูล/สตริงขนาดเกินกว่าที่ระบบจัดการได้ ทำให้ระบบเสียหาย/เปิดช่องโหว่

2. Denial of Service (DoS) — ส่งข้อความจำนวนมาก/ไฟล์แนบใหญ่/เข้ารหัสหลายอิลิเมนต์ ทำให้ระบบใช้ทรัพยากรหมด

3. Password Attack (สุ่มรหัสผ่าน) — ลองผิดลองถูกเดารหัสง่ายๆ (เช่น 123, root/root, Administrator/123) อันตรายมากถ้าเป็นบัญชีผู้ดูแลระบบ

4. WSDL Enumeration (แสวงประโยชน์จาก WSDL) — WSDL เปิดเผยตำแหน่ง/ฟังก์ชัน/วิธีเรียกใช้บริการ ทำให้ผู้ร้ายเข้าถึงฟังก์ชันที่ไม่ควรเข้าถึง (เช่น TradeStocks)

5. XML-based Attacks (โจมตีผ่าน XML) — ใช้ความสัมพันธ์แบบ recursive เพิ่มเอนทิตี, ปลอมพารามิเตอร์, ใส่ช่องว่างจำนวนมาก ทำให้ตัวแปล XML โอเวอร์โหลด (เช่น bug การอ้างอิง DTD แบบวงกลมของ Oracle Application Server)

ประเด็นที่ควรเน้นเพื่อสอบ

  • CIA สามเป้าหมายพื้นฐาน (Integrity/Confidentiality/Availability) พร้อมตัวอย่าง — ออกแน่นอน
  • STRIDE ทั้ง 6 ตัว — ต้องจำชื่อภาษาอังกฤษ คำแปล และตัวอย่างของแต่ละแบบ
  • 7 ขั้นตอนการสร้างแบบจำลองภัยคุกคาม และเหตุผลที่เป็น "กระบวนการทำซ้ำ"
  • ที่มาช่องโหว่ 5 ข้อ และ มาตรการตอบโต้ตามช่วงวงจรชีวิต (พัฒนา/ติดตั้ง/ทำงาน → IDS/IDP)
  • เกณฑ์วิเคราะห์ช่องโหว่ 3 องค์กร: ไมโครซอฟต์ (4 ระดับ), US-CERT (0-180), FIRST/CVSS (3 กลุ่มเกณฑ์)
  • SSL/TLS: ให้ความปลอดภัยอะไร 3 อย่าง และ ข้อจำกัด (จุดต่อจุด, เข้ารหัสบางส่วนไม่ได้)
  • XML Encryption vs XML Signature: Encryption=ความลับแต่ไม่คงสภาพ, Signature=คงสภาพ+ป้องกันปฏิเสธ — ต้องใช้คู่กัน
  • WS-Security: ทำไมต้องมีนอกเหนือจาก SSL/TLS (end-to-end, multi-hop, เลือกเข้ารหัสบางส่วน), ใช้ XML Enc + XML Sig + Security Token
  • SAML: 3 ประเภท Assertion (Authentication/Attribute/Authorization) และ SSO
  • WS-Reliability 4 ระดับการประกัน (AtMostOnce/AtLeastOnce/ExactlyOnce/InOrder)
  • 5 รูปแบบการโจมตี (Buffer Overflow, DoS, Password Attack, WSDL Enumeration, XML-based Attack) พร้อมวิธีการ
  • PKI/CA + STS/WS-Trust — บทบาทในการจัดการกุญแจและความไว้วางใจ
  • องค์ประกอบความปลอดภัยจริง = คน + กระบวนการ + เทคโนโลยี และมาตรฐานเปิด/ปิด

ภาพรวม

หน่วยนี้แบ่งเป็น 2 ส่วนใหญ่ คือ นโยบาย (Policy) และ การจัดการ (Management) ของเว็บเซอร์วิส (Web Service: WS) บนสถาปัตยกรรมเชิงบริการ (SOA) ส่วนแรกอธิบายความหมาย ประเภท และภาษานโยบาย พร้อมกรอบงาน WS-Policy และข้อกำหนดที่เกี่ยวข้อง (WS-ReliableMessage, WS-MetadataExchange, WS-Security) ส่วนที่สองว่าด้วยการจัดการระบบประมวลผลแบบกระจาย กรอบงานการจัดการองค์กร (EMS) และกรอบงานการจัดการเว็บเซอร์วิส รวมถึงมาตรฐาน WSDM และ WS-Management กับเครื่องมือจัดการต่างๆ

แนวคิดสำคัญ

นโยบาย (Policy) คือ "หลักและวิธีปฏิบัติที่ถือเป็นแนวดำเนินการ" — เป็นกฎเกณฑ์/ข้อตกลงที่ใช้ควบคุมพฤติกรรมการทำงานของบริการให้เป็นไปตามที่ต้องการ ทำให้ผู้ให้บริการและผู้ใช้บริการสื่อสารด้วยเงื่อนไขที่ตกลงกันได้ ข้อดีคือเมื่อเปลี่ยนกฎเกณฑ์ ไม่ต้องเข้าไปแก้โปรแกรมประยุกต์ในแต่ละระบบ ส่วน การจัดการ (Management) สำคัญขึ้นเรื่อยๆ เพราะระบบแบบกระจายซับซ้อน มีหลายองค์กร/หลายแพลตฟอร์มเชื่อมโยงกัน ทำให้ยากต่อการตรวจสอบหาต้นตอปัญหา การนำเทคโนโลยี WS มาใช้ช่วยให้องค์ประกอบต่างๆ เป็นอิสระต่อกัน (loosely coupled) และจัดการได้ง่ายขึ้น

เนื้อหาหลัก

12.1 นโยบายการให้บริการเว็บเซอร์วิส

12.1.1 ความหมายและประเภทของนโยบาย

  • นโยบายควบคุมบริการให้มีพฤติกรรมตามที่วางไว้ กำหนดได้หลายระดับ (จุดบริการ, โปรแกรมประยุกต์, ขอบเขต/Domain, ระหว่างขอบเขต)
  • ประเภทนโยบาย 3 แบบ:

1. นโยบายด้านเวอร์ชัน (Versioning Policy) — จัดการหลายเวอร์ชันของบริการที่ใช้ชื่อเดียวกันให้ทำงานพร้อมกันได้ (เช่น ลูกค้าใหม่ vs ลูกค้าประจำได้ส่วนลดต่างกัน)

2. นโยบายด้านคุณภาพการให้บริการ (QoS Policy: Quality of Service) — เกี่ยวกับประสิทธิภาพและความเชื่อถือ เช่น Reliability, Scalability, Availability, Notification วัดด้วย Response time, Bandwidth; เกี่ยวข้องกับ WS-ReliableMessage และ WS-Agreement

3. นโยบายด้านความมั่นคง (Security Policy) — เช่น อัลกอริทึมเข้ารหัส, ลายมือชื่อดิจิทัล (Digital Signature), ใบรับรอง X.509; เกี่ยวข้องกับ WS-Security, XML-Signature, XML-Encryption

  • ภาษานโยบาย (Policy Language) ต้องมีคุณสมบัติ: ความยืดหยุ่น (Flexibility), ความเป็นทั่วไป (Genericity), ความสามารถขยายได้ (Extensibility) + นำมาใช้ใหม่ได้ (Reusable) → นำไปสู่การสร้าง WS-Policy

12.1.2 กรอบงานนโยบาย (WS-Policy)

  • WS-Policy เป็นข้อกำหนดของ W3C ให้เว็บเซอร์วิสใช้เอกสาร XML แสดงนโยบายแก่ผู้ใช้บริการได้ เชื่อมโยงกับ WSDL, UDDI, URI ได้ ประกอบด้วย WS-Policy Framework + WS-Policy Attachment
  • ส่วนประกอบการแสดงนโยบาย 5 ส่วน:

1. นิพจน์นโยบาย (Policy Expression) — กลุ่มข้อมูล XML ที่แสดงข้อบังคับนโยบายทั้งหมด

2. ทางเลือกนโยบาย (Policy Alternative) — การรวมกลุ่มข้อบังคับเป็นกลุ่มๆ ตามเงื่อนไข

3. ข้อบังคับนโยบาย (Policy Assertion) — กฎเกณฑ์ในประเด็นหนึ่งๆ (ระบุ type และ parameter ได้)

4. เป้าหมายนโยบาย (Policy Subject) — ทรัพยากรที่นโยบายเชื่อมโยง (รวมกันเรียก Policy Scope)

5. การผูกนโยบาย (Policy Attachment) — เชื่อมนิพจน์นโยบายเข้ากับเป้าหมาย/ขอบเขต

  • ความสัมพันธ์: ข้อบังคับ → รวมเป็นทางเลือก → รวมเป็นนิพจน์ → ผูกกับเป้าหมาย (ตัวอย่าง: เลือกเข้ารหัส Basic128 หรือ Basic256 ที่ใช้อัลกอริทึม AES ต่างกันที่จำนวนบิต)

12.1.3 ข้อกำหนดที่เกี่ยวข้องกับนโยบาย

  1. WS-ReliableMessage (ข่าวสารที่น่าเชื่อถือได้) — เพราะ SOA เป็นแบบ loosely coupled ควบคุมการสื่อสารไม่ได้โดยตรง มาตรฐานนี้ช่วยรับรองว่าข้อมูล (SOAP) ไปถึงปลายทางครบถ้วนและถูกลำดับ → เพิ่ม QoS
  2. WS-MetadataExchange (การแลกเปลี่ยนนิยามข้อมูล) — นิยามข้อมูล (Metadata) คือข้อมูลอธิบายบริการ อยู่ในรูป WSDL; วิธีแลกเปลี่ยนนโยบายมี 3 แบบ: (1) ส่งตรงผ่าน SOAP Header (ง่ายแต่ไม่มีมาตรฐาน) (2) ค้นจากทะเบียน UDDI (3) ค้นด้วย WS-MetadataExchange (ถามตรงไม่ต้องผ่าน UDDI สาธารณะ, ประมวลผลอัตโนมัติ run-time ได้) → เพิ่มความยืดหยุ่นและมาตรฐาน
  3. WS-Security (ความมั่นคง) — ใช้ XML-Encryption + XML-Signature รักษาความปลอดภัยถึงระดับข้อมูล XML บนข้อความ SOAP (เข้ารหัส Header + Body, เพิ่มลายมือชื่อ) ทำให้ได้ คุณสมบัติความมั่นคง 5 ประการ: (1) การรักษาความลับ (Confidentiality) (2) การรักษาความคงสภาพ (Integrity) (3) การระบุตัวตน (Identification) (4) การพิสูจน์ตัวตน (Authentication) (5) การอนุญาตการใช้งาน (Authorization)

12.2 การจัดการในการให้บริการเว็บเซอร์วิส

12.2.1 การจัดการระบบประมวลผลแบบกระจาย

  • วิวัฒนาการ: Client/Server → Distributed → SOA; ระบบแบบกระจาย มีคุณลักษณะ: เป็นระบบเปิด, ใช้ทรัพยากรร่วมกัน, ทำงานพร้อมกัน, ขยายได้, จัดการข้อผิดพลาดได้ (ตัวอย่าง: CORBA)
  • ข้อเสีย: ซับซ้อน, รองรับการโจมตีมากขึ้น, หาต้นตอปัญหายาก (อาจอยู่ที่เครื่อง/เครือข่าย/โปรแกรม)
  • ความสามารถจัดการ 3 ส่วน: (1) การตรวจดู (Monitoring) — ตรวจจับขณะทำงาน/เหตุการณ์อดีต (2) การติดตาม (Tracking) — สังเกตหน่วยงาน/เธรดข้ามทรัพยากร (3) การควบคุม (Control) — ปรับเปลี่ยนพฤติกรรมขณะทำงาน
  • สถาปัตยกรรมการจัดการเว็บเซอร์วิส 4 ส่วน: (1) การตรวจดูผู้ใช้ (งานส่วนหน้า front-end + ส่วนหลัง back-end) (2) การตรวจดูโครงสร้างพื้นฐาน (3) การตรวจดูธุรกรรม (Transaction) (4) การจัดหาทรัพยากร

12.2.2 กรอบงานการจัดการองค์กรและการจัดการเว็บเซอร์วิส

  • EMS (Enterprise Management System) — กรอบงานจัดการอุปกรณ์บนเครือข่ายแบบอิสระ (ไม่ขึ้นกับผู้ผลิต/โพรโทคอล) มี 5 หน้าที่: การจัดการความผิดพลาด (Fault), โครงแบบ (Configuration), บัญชี (Accounting), ประสิทธิภาพ (Performance), ความมั่นคง (Security); ควบคุมด้วย NMS (Network Managers Server)
  • รูปแบบการจัดวาง (Topology) 3 แบบ: รวมศูนย์ (Central — ง่ายแต่จำกัดด้านประสิทธิภาพ/เสี่ยงล่ม), ลำดับชั้น (Hierarchical — ดีขึ้น), กระจาย (Distributed — ดีที่สุดด้านประสิทธิภาพและการขยายตัว)
  • กรอบงานการจัดการเว็บเซอร์วิส = กลุ่มองค์ประกอบที่ทำให้ตรวจสอบ/ตรวจดู/แก้ปัญหา/จัดหา/จัดการโครงสร้าง WS ได้ ต้องมี กิจกรรม 4 ลักษณะ: (1) แยกหน่วยงานเป็นองค์ประกอบ (2) ตรวจวัดในหน่วยงาน (3) รับรู้/ติดตามร่องรอยการใช้งาน (4) ระบุและทำนายปัญหา; ลักษณะเชิงหน้าที่รวมถึง การวัดตัวชี้วัดประสิทธิภาพ (Performance Indicators), Auditing/Monitoring/Troubleshooting, การโยกย้ายบริการ, การจัดการเวอร์ชัน, วงจรชีวิตบริการ, การบำรุงรักษา, การขยายตัว, Dynamic, ความมั่นคง

12.2.3 มาตรฐานและเครื่องมือการจัดการเว็บเซอร์วิส

  • มาตรฐานสำคัญ 2 ตัว:

1. WSDM (Web Services Distributed Management) — อนุมัติโดย OASIS ใช้ WS-Resource Framework + WS-Notification ทำให้ WS เปิดเผยคุณสมบัติ/ทรัพยากร และจัดการเมื่อเกิดข้อผิดพลาดได้ ประกอบด้วย 2 ส่วน:

- MUWS (Management Using Web Services) — ใช้ WS จัดการทรัพยากร/อุปกรณ์ข้ามระบบ (อธิบายด้วย WSDL)

- MOWS (Management of Web Services) — จัดการตัวเว็บเซอร์วิสเอง กำหนดความสามารถ: Identity, Identification, Metrics, Operational State (Up: Busy/Idle, Down: Stop/Crash/Saturated), Processing State (Received→Processing→Completed/Failed)

2. WS-Management — คล้าย WSDM แต่เน้นจัดการ "ฮาร์ดแวร์" ต่างชนิด เสนอโดย DMTF (Distributed Management Task Force) ร่วมกับ AMD, Intel, Microsoft, Sun; ตัวอย่าง WinRM ของวินโดวส์

  • เครื่องมือจัดการ (ไมโครซอฟต์):

- WCF (Windows Communication Foundation Management) — สร้าง/ติดตั้ง/ตรวจดู WS, ติดตามเส้นทางข้อมูล, วัดประสิทธิภาพ

- MOM (Microsoft Operations Manager) — สร้างกฎเกณฑ์ตรวจดู WS และอุปกรณ์ข้ามผู้ผลิต รับการแจ้งเตือน

- ADFS (Active Directory Federation Services) — จัดการความมั่นคง (ระบุตัวตน/สิทธิ) รองรับ SSO (Single Sign-On) ใช้หลายบริการในเซสชันเดียวโดยไม่ต้องพิสูจน์ตัวตนซ้ำ

  • คุณสมบัติความมั่นคง 5 ประการ จาก WS-Security (Confidentiality, Integrity, Identification, Authentication, Authorization)
  • วิธีแลกเปลี่ยนนโยบาย 3 แบบ (SOAP Header / UDDI / WS-MetadataExchange)
  • ความสามารถจัดการระบบแบบกระจาย 3 ส่วน (Monitoring/Tracking/Control) และ สถาปัตยกรรมจัดการ WS 4 ส่วน
  • EMS: 5 หน้าที่ และ 3 รูปแบบการจัดวาง (Central/Hierarchical/Distributed) พร้อมข้อดีข้อเสีย — Distributed ดีที่สุด
  • กิจกรรม 4 ลักษณะของกรอบงานจัดการ WS

  • เครื่องมือไมโครซอฟต์ (WCF/MOM/ADFS) โดยเฉพาะ ADFS + SSO
  • ความแตกต่างการจัดการระบบกระจ่ายในอดีต (เชิงเทคนิค) vs บนพื้นฐาน WS (เชิงบริการ, อิสระต่อกัน)

ภาพรวม

หน่วยนี้อธิบายว่าจะนำแนวคิดพื้นฐานของ "การบริการผ่านเว็บ" (Web Service) มาประยุกต์ใช้แก้ปัญหาการพัฒนาระบบงานและการแลกเปลี่ยนข้อมูลระหว่างองค์กรในเชิงธุรกิจได้อย่างไร โดยแบ่งเป็น 2 ส่วนใหญ่ คือ (1) แนวคิดการประยุกต์ใช้และการบริหารโครงการด้านไอทีเพื่อพัฒนาบริการผ่านเว็บ และ (2) โมเดลธุรกิจที่เกิดขึ้นพร้อมกรณีศึกษาจริงขององค์กรธุรกิจที่นำการบริการผ่านเว็บไปใช้ ทั้งที่สำเร็จและล้มเหลว เน้นให้เห็นว่าเทคโนโลยีนี้เป็นเครื่องมือสร้างโอกาสและความได้เปรียบทางธุรกิจ

แนวคิดสำคัญ

แก่นของหน่วยคือการบริการผ่านเว็บเป็น "มาตรฐานเปิดที่เป็นกลาง" (open standard) ใช้ภาษา XML เป็นแกน สื่อสารผ่านโพรโทคอล SOAP บน HTTP จึงทำให้ระบบที่ต่างแพลตฟอร์มกันทำงานร่วมกันได้ (interoperability) โดยไม่ต้องเปลี่ยนระบบเดิม เพียงเสริมเครื่องมือที่เหมาะสมเข้าไป ด้วยคุณสมบัตินี้องค์กรธุรกิจจึงนำไปประยุกต์ได้ 3 แนวทางหลัก คือ ตอบสนองความต้องการบริการเบ็ดเสร็จ ลดค่าใช้จ่ายในการบูรณาการระบบงานภายใน และทำให้การบูรณาการระหว่างธุรกิจกับธุรกิจง่ายขึ้น นอกจากนี้ความสำเร็จไม่ได้ขึ้นกับเทคโนโลยีอย่างเดียว แต่ต้องอาศัยการบริหารโครงการด้านไอทีที่ดีด้วย

เนื้อหาหลัก

13.1 แนวคิดในการประยุกต์การบริการผ่านเว็บในเชิงธุรกิจ

1) ประยุกต์ใช้เพื่อตอบสนองความต้องการบริการเบ็ดเสร็จ (One Stop Service)

ปัญหาคือเว็บไซต์หนึ่งมักขายสินค้า/บริการอย่างเดียว ลูกค้าที่ต้องการหลายบริการพร้อมกันจึงต้องเข้าหลายเว็บ การบริการผ่านเว็บแก้ปัญหานี้ได้ เพราะแลกเปลี่ยนข้อมูลและประสานงานข้ามระบบได้ ตัวอย่างคลาสสิกคือเว็บ 1stopservicetour.com ที่รวมบริการท่องเที่ยวครบวงจร (จองตั๋วเครื่องบิน จองโรงแรม เช่ารถ ข้อมูลสถานที่ บริการการเงิน) มาไว้จุดเดียว โดยแต่ละธุรกิจประกาศและลงทะเบียนบริการของตนไว้กับ UDDI เรื่องความปลอดภัย การบริการผ่านเว็บผ่านไฟร์วอลล์ได้เพราะ SOAP ส่งผ่าน HTTP และมีมาตรฐาน PKI รองรับ เช่น MD5, SSL, PGP ส่วนเครื่องมือพัฒนาแบ่งเป็น 4 กลุ่ม คือ Microsoft .NET, Sun ONE, IBM (Web Services Toolkit) และเครื่องมืออื่น ๆ ทั้งเชิงพาณิชย์และโอเพนซอร์ส

2) ประยุกต์ใช้เพื่อลดค่าใช้จ่ายในการบูรณาการระบบงานทางธุรกิจ

เมื่อระบบย่อยจำนวน n ระบบเชื่อมต่อแบบจุดต่อจุด จะเกิดโครงสร้างแบบ ร่างแห (Mesh) ซึ่งต้องใช้การเชื่อมต่อถึง n(n-1)/2 คอนเนคชัน ดูแลยากและแพง ทางแก้คือออกแบบส่วนติดต่อแบบ บัส (Bus) ซึ่งเป็นพื้นฐานของ EAI (Enterprise Application Integration) เหลือเพียง n คอนเนคชัน แต่ผลิตภัณฑ์ EAI ในตลาดราคาสูงมาก จึงนำการบริการผ่านเว็บมาทำหน้าที่เป็น Enterprise Service Bus (ESB) แทน เรียกการบูรณาการเชิงบริการนี้ว่า SOI (Service Oriented Integration) ตัวอย่างการบูรณาการ 5 ระบบจาก 10 คอนเนคชัน (แบบร่างแห) ลดเหลือ 5 คอนเนคชัน ช่วยลดต้นทุนได้ราวครึ่งหนึ่ง

3) ประยุกต์ใช้เพื่อให้การบูรณาการระหว่างธุรกิจกับธุรกิจ (B2BI) ง่ายขึ้น

อดีตใช้การเชื่อมแบบ peer-to-peer ทำให้เกิด "สปาเก็ตตีอินทิเกรชัน" (spaghetti integration) ที่ยุ่งเหยิง ส่วนเทคโนโลยี EDI (Electronic Data Interchange) ใช้โครงสร้าง ฮับและสโปค (Hub and Spoke) แต่ยังมีข้อจำกัด เช่น ไม่บอกความหมายของข้อมูลในแมสเสจ และมาตรฐานต่างกันในแต่ละผู้ให้บริการ การบริการผ่านเว็บแก้ปัญหานี้ด้วยคุณสมบัติ 3 ประการ คือ เป็นมาตรฐานเปิด มีความเป็นอิสระในแต่ละส่วน และค้นหาบริการแบบไดนามิกผ่าน UDDI ได้ ช่วยลดค่าใช้จ่ายตั้ง e-Marketplace (การตลาดอิเล็กทรอนิกส์) ได้ด้วย

การบริหารโครงการด้านไอทีเพื่อพัฒนาบริการผ่านเว็บ (13.1.2)

โครงการมีข้อจำกัด 3 ด้านที่ต้องแลกเปลี่ยน (trade-off) กัน คือ ขอบเขต เวลา และค่าใช้จ่าย องค์ความรู้ในการบริหารโครงการตามมาตรฐาน PMI/PMBOK มี 9 ด้าน แบ่งเป็น องค์ความรู้หลัก 4 ประการ (บริหารขอบเขต บริหารเวลา บริหารค่าใช้จ่าย บริหารคุณภาพ) และ องค์ความรู้เสริม 5 ประการ (บริหารการบูรณาการ บริหารทรัพยากรมนุษย์ บริหารการสื่อสาร บริหารความเสี่ยง บริหารการจัดซื้อจัดจ้าง) ซอฟต์แวร์บริหารโครงการแบ่ง 3 ระดับ คือ Low-end, Midrange (เช่น Microsoft Project) และ High-end (Enterprise project management)

13.2 โมเดลธุรกิจและกรณีศึกษา

โมเดลธุรกิจ 3 ลักษณะ

  • เป็นผู้ประกอบการ/ผู้ให้บริการ เช่น ผู้ให้บริการข้อมูลผ่านเว็บโดยตรง หรือผู้ให้บริการทะเบียน (Registry Provider) คล้ายธุรกิจสมุดหน้าเหลือง
  • เป็นตัวแทน/ตัวกลาง เช่น นายหน้าจับคู่สินค้า คนกลางจัดแพ็กเกจ ตัวแทนส่วนบุคคล
  • เป็นคู่ค้า/พันธมิตรธุรกิจ เช่น ผู้รับประกันความถูกต้องของธุรกรรมการเงิน ผู้ให้บริการโฮสติงสำหรับการบริการผ่านเว็บ

วงจรชีวิตที่เกี่ยวข้อง ได้แก่ วงจรชีวิตโครงการ (PLC), วงจรชีวิตการพัฒนาซอฟต์แวร์ (SDLC: วางแผน-วิเคราะห์-ออกแบบ-สร้าง/ติดตั้ง-บำรุงรักษา; มีหลายแบบ เช่น น้ำตก ก้นหอย Prototyping RAD Scrum) และวงจรชีวิตการพัฒนาการบริการผ่านเว็บ (WSDLC)

  • กรณีที่ 2 ข้อมูลประกันสุขภาพ (MedUnite) – กรณีล้มเหลว ก่อตั้งโดยบริษัทประกัน 7 ราย ใช้ XML/SOAP ตรวจสอบสิทธิรักษาพยาบาล ลดต้นทุนติดต่อจาก 5 ดอลลาร์เหลือ 44 เซนต์ แต่ล้มเหลวใน 2 ปี เพราะวางแผนธุรกิจไม่ดี แข่งกับเจ้าตลาด (WebMD) ไม่ได้ และผู้ร่วมก่อตั้งถอนทุน บทเรียนคือเทคโนโลยีดีแต่บริหารโครงการ/ขอบเขต/ผู้มีส่วนได้เสีย (stakeholders) ไม่ดีก็ล้มเหลวได้
  • กรณีที่ 3 ประกันชีวิต (MetLife) – สำเร็จ ให้ลูกค้าดู/แก้ไขบัญชีออนไลน์ ขยายฟังก์ชันไปบริษัทในเครือ ใช้ IBM WebSphere + DB2 ลดต้นทุนต่อรายการเหลือไม่กี่เซนต์
  • กรณีที่ 4 รถเช่า (Dollar Rent A Car) – สำเร็จ เปลี่ยนจาก EDI มาเป็นการบริการผ่านเว็บ (SOAP/WSDL/HTTP) เชื่อมคู่ค้าจองรถเช่าโดยตรง รองรับอุปกรณ์พกพา และคุมความปลอดภัยด้วยการจำกัด IP Address
  • กรณีที่ 5 พาณิชย์อิเล็กทรอนิกส์ (Amazon) – ผู้นำระดับโลก เปิด AWS (Amazon Web Services) ปี 2549 ผสานการบริการผ่านเว็บกับ Cloud Computing แบ่งบริการเป็น 12 กลุ่ม เช่น Compute (EC2), Storage (S3), Database (RDS, DynamoDB), Content Delivery (CloudFront) เป็นต้น

ประเด็นที่ควรเน้นเพื่อสอบ

  • แนวทางประยุกต์การบริการผ่านเว็บในธุรกิจ 3 ข้อ (บริการเบ็ดเสร็จ / ลดค่าใช้จ่ายบูรณาการภายใน / บูรณาการ B2B ง่ายขึ้น) — ออกสอบบ่อย
  • สูตรจำนวนคอนเนคชันแบบร่างแห n(n-1)/2 เทียบกับแบบบัส n และความสัมพันธ์ EAI → ESB → SOI
  • คุณสมบัติ 3 ประการของการบริการผ่านเว็บ: มาตรฐานเปิด / เป็นอิสระแต่ละส่วน / ค้นหาบริการแบบไดนามิก
  • บทบาทของ UDDI, WSDL, SOAP และเหตุที่ SOAP ผ่านไฟร์วอลล์ได้ (ส่งผ่าน HTTP)
  • เครื่องมือพัฒนา 4 กลุ่ม (Microsoft .NET / Sun ONE / IBM / อื่น ๆ)
  • องค์ความรู้บริหารโครงการ 9 ประการ (หลัก 4 + เสริม 5) ตาม PMI และข้อจำกัด 3 ด้าน (ขอบเขต/เวลา/ค่าใช้จ่าย)
  • โมเดลธุรกิจ 3 ลักษณะ (ผู้ประกอบการ / ตัวกลาง / คู่ค้า) พร้อมตัวอย่าง
  • กรณีศึกษาที่ต้องจำให้ได้: MedUnite (ล้มเหลว–สาเหตุ), MetLife, Dollar Rent A Car, และ Amazon AWS (ผสาน Web Service + Cloud, 12 กลุ่มบริการ)
  • ข้อจำกัดของ EDI ที่ทำให้ต้องหันมาใช้การบริการผ่านเว็บ

หน่วยที่ 14: การประยุกต์การบริการผ่านเว็บในภาครัฐ

ภาพรวม

หน่วยนี้ว่าด้วยการที่ภาครัฐนำเทคโนโลยีสารสนเทศและการสื่อสาร (ICT) มาเป็นเครื่องมือบริหารจัดการและให้บริการแก่ประชาชนและภาคธุรกิจ เรียกว่า "รัฐบาลอิเล็กทรอนิกส์" (e-Government) เนื้อหาแบ่งเป็น 3 ตอน คือ ความรู้เบื้องต้นและประโยชน์ของบริการภาครัฐผ่านเว็บ, กระบวนการพัฒนาบริการผ่านเว็บในภาครัฐพร้อมแนวโน้ม, และตัวอย่างบริการจริงด้านการเงินและด้านบุคลากร เน้นแนวคิดบริการแบบเบ็ดเสร็จ ณ จุดเดียว (one stop service) เพื่อเพิ่มประสิทธิภาพและความโปร่งใส

แนวคิดสำคัญ

รัฐบาลอิเล็กทรอนิกส์มุ่งแลกเปลี่ยนข้อมูลข่าวสารระหว่างกันเพื่อให้บริการประชาชนได้ดีที่สุด สะดวก รวดเร็ว และเบ็ดเสร็จ ณ จุดบริการเดียว การให้บริการของภาครัฐแบ่งตามผู้รับบริการได้ 4 ประเภท (G2C, G2B, G2G, G2E) ส่วนตัวบริการที่ให้แก่ประชาชนแบ่งได้ 6 ประเภท การพัฒนาเริ่มจากแผน e-Government Roadmap ของกระทรวง ICT ที่ผลักดันให้แต่ละกรมมีบริการ e-Service และมุ่งสู่เว็บพอร์ทัลกลาง (egov.go.th) นอกจากนี้ยังต้องเข้าใจองค์ประกอบและขั้นตอนการพัฒนาเว็บไซต์ที่มีคุณภาพ รวมถึงแนวโน้มอุปกรณ์พกพาและแนวคิดเว็บ 2.0

เนื้อหาหลัก

14.1 ความรู้เบื้องต้นและประโยชน์ของบริการภาครัฐผ่านเว็บ

ประเภทรัฐบาลอิเล็กทรอนิกส์ 4 ประเภท (แบ่งตามผู้รับบริการ)

  • G2C (Government to Citizens): รัฐสู่ประชาชนโดยตรง ทำธุรกรรมเองได้ตลอด 24 ชม. × 7 วัน เช่น ชำระภาษี จดทะเบียน จ่ายค่าปรับ ต่ออายุบัตรประชาชน
  • G2B (Government to Business): รัฐสู่ภาคธุรกิจ เช่น จดทะเบียนการค้า ส่งเสริมการลงทุน การจัดซื้อจัดจ้างอิเล็กทรอนิกส์ (e-Auction) — การประมูลมี 2 แบบ คือ ประมูลเพื่อขายและประมูลเพื่อซื้อ
  • G2G (Government to Government): รัฐสู่รัฐด้วยกัน เปลี่ยนจากกระดาษ/ลายเซ็นมาใช้เครือข่ายและลายมือชื่ออิเล็กทรอนิกส์ ส่วนใหญ่เป็นระบบงาน Back Office เช่น สารบรรณอิเล็กทรอนิกส์ ระบบบัญชี
  • G2E (Government to Employees): รัฐสู่พนักงานของรัฐ เช่น ระบบสวัสดิการ ระบบพัฒนาบุคลากร

ประเภทบริการของภาครัฐ 6 ประเภท ได้แก่ (1) บริการด้านข้อมูลข่าวสาร (2) บริการเกี่ยวกับไฟฟ้า ประปา โทรศัพท์ ภาษี และการเดินทาง (3) บริการเกี่ยวกับการทะเบียน (4) บริการเกี่ยวกับการซื้อขาย จัดซื้อจัดจ้าง การลงทุนและตลาดหลักทรัพย์ (5) บริการร้องเรียน/ร้องทุกข์ (เช่น สายด่วน 1111) (6) บริการสำหรับเว็บไซต์และผู้ใช้อินเทอร์เน็ต

ประโยชน์ของบริการภาครัฐผ่านเว็บ 6 ข้อ (1) เพิ่มประสิทธิภาพการทำงานของหน่วยงานรัฐ เช่น ต่อทะเบียนรถจาก 1-2 ชม. เหลือ 15-20 นาที, การประปาฯ ใช้อุปกรณ์พกพาจดมาตร-พิมพ์ใบแจ้งหนี้ทันที (2) เพิ่มคุณภาพบริการแก่ประชาชน (เบ็ดเสร็จ ณ จุดเดียว) (3) ลดค่าใช้จ่ายของประชาชนในการขอรับบริการ (4) ลดค่าใช้จ่ายของภาครัฐในการให้บริการ (5) เพิ่มความโปร่งใส ลดการเลือกปฏิบัติ (6) เพิ่มขีดความสามารถในการแข่งขันของประเทศ

14.2 การพัฒนาบริการผ่านเว็บในภาครัฐ

ความเป็นมา เริ่มจากการพัฒนาระบบบริการภาครัฐผ่านระบบอิเล็กทรอนิกส์ (e-Services) ตามแผน e-Government Roadmap ที่กระทรวง ICT เสนอ ครม. อนุมัติเมื่อ 16 ส.ค. 2548 ขอให้แต่ละกรมมีบริการ e-Service อย่างน้อย 1 เว็บ และมุ่งพัฒนา egov.go.th ให้เป็นเว็บพอร์ทัลให้บริการ ณ จุดเดียว (First Stop Service Web Portal) ครอบคลุม G2C/G2B/G2G/G2E ปัจจุบันมีบริการที่พัฒนาแล้ว 21 บริการ (เช่น ระบบจัดหางาน ระบบจดทะเบียนพาณิชย์ผ่านอินเทอร์เน็ต ระบบแจ้งเบาะแส) หมายเหตุสำคัญ: บริการส่วนใหญ่ยังเป็น "แอปพลิเคชันเซอร์วิส" ที่ผู้ใช้ต้องเข้ามาเรียกใช้เอง ส่วน "เว็บเซอร์วิสที่แท้จริง" จะเชื่อมโยงกันอัตโนมัติ เช่น บริการตรวจสอบเลขบัตรประชาชนให้ภาคเอกชน และเครดิตบูโร

องค์ประกอบของบริการผ่านเว็บ อย่างน้อยควรมี 5 ส่วน คือ ชื่อโดเมน (domain name), ที่ตั้งเว็บไซต์ (hosting), การออกแบบและพัฒนา, เนื้อหาของเว็บ (content — สำคัญที่สุด), และการประชาสัมพันธ์ (promotion — เป็นตัวเพิ่มจำนวนผู้เข้าชม ไม่ใช่การออกแบบ) การจัดตั้งเว็บไซต์ทำได้ 3 วิธี: ใช้เว็บโฮสติงฟรี / เช่าเว็บโฮสติง / ลงทุนจัดตั้งเว็บโฮสติงเอง

ขั้นตอนการพัฒนาเว็บไซต์ 6 ขั้นตอน (1) กำหนดเป้าหมายและวางแผน (2) วิเคราะห์และจัดโครงสร้างข้อมูล (ได้ผังโครงสร้างเว็บไซต์และระบบนำทาง/Navigation) (3) ออกแบบเว็บไซต์และเตรียมข้อมูล (4) สร้างและทดสอบ (5) เผยแพร่และส่งเสริมให้เป็นที่รู้จัก (6) ดูแลและพัฒนาเว็บไซต์

แนวโน้ม (1) อุปกรณ์พกพา (mobile device) เช่น โน้ตบุ๊ก เน็ตบุ๊ก แท็บเล็ต สมาร์ตโฟน พ็อกเก็ตพีซี Air Card จะเล็กลง เชื่อมต่อไร้สายได้หลากหลาย ทำให้เข้าถึงบริการได้ทุกอุปกรณ์ (any device) ทุกที่ (anywhere) ทุกเวลา (anytime) (2) แนวทางพัฒนาใช้แนวคิด เว็บ 2.0 ที่เน้นการแบ่งปันข้อมูล การออกแบบที่เน้นผู้ใช้เป็นศูนย์กลาง (user-centered) และผู้ใช้ร่วมสร้างเนื้อหา (ต่างจากเว็บ 1.0 ที่อ่านอย่างเดียว) ตัวอย่างเว็บ 2.0 เช่น บล็อก เครือข่ายสังคม วิกิ คุณสมบัติเด่น 6 อย่าง ได้แก่ การค้นหา (search), การเชื่อมโยง (links), การสร้าง/แก้ไขร่วมกัน (authoring), การติดตามด้วยแท็ก (tags), ส่วนเสริม (extension), การแจ้งเตือน (signals เช่น RSS, Atom)

14.3 ตัวอย่างการบริการผ่านเว็บในภาครัฐ

ด้านการเงิน มีตัวอย่างเว็บเซอร์วิสที่แท้จริง เช่น ธนาคารแห่งประเทศไทย (ธปท.) ให้ข้อมูลอัตราแลกเปลี่ยนแก่ธนาคารพาณิชย์ที่ลงทะเบียน, ตลาดหลักทรัพย์ฯ ให้ข้อมูลมูลค่าหุ้น/ดัชนี, และบริษัทข้อมูลเครดิตแห่งชาติ (Credit Bureau) ให้ตรวจประวัติการชำระสินเชื่อ ส่วนบริการแบบแอปพลิเคชันเซอร์วิส เช่น กรมสรรพากร บริการยื่นแบบและชำระภาษีผ่านอินเทอร์เน็ต (มีบริการย่อย 4 อย่าง: แนะนำบริการ / แนะนำหน่วยรับชำระภาษี / บริการสมาชิก / บริการยื่นแบบฯ) คืนภาษีเร็วขึ้นจาก 60 วันเหลือไม่เกิน 1 เดือน และ ธปท. แบ่งบริการเป็นกลุ่ม: นโยบายการเงิน สถาบันการเงิน ตลาดการเงิน ระบบการชำระเงิน และสถิติ

ด้านบุคลากร ส่วนใหญ่เป็นบริการด้านข้อมูล บริการรับสมัครงาน รับสมัครสอบเข้ารับราชการ รับสมัครนักศึกษา รับสมัครนักศึกษาฝึกงาน และรับสมัครสอบทุนรัฐบาล

  • ความต่างระหว่าง "เว็บเซอร์วิสที่แท้จริง" (เชื่อมโยงอัตโนมัติ) กับ "แอปพลิเคชันเซอร์วิส" (ผู้ใช้เรียกใช้เอง)
  • แนวคิดเว็บ 2.0 คุณสมบัติ 6 อย่าง และความต่างจากเว็บ 1.0
  • ตัวอย่างบริการด้านการเงิน (ธปท. อัตราแลกเปลี่ยน, กรมสรรพากรยื่นภาษี, Credit Bureau) และด้านบุคลากร (รับสมัครงาน/สอบ/ทุน)

ภาพรวม

หน่วยนี้สรุปสถานะปัจจุบันของเว็บเซอร์วิสและฉายภาพแนวโน้มในอนาคต แบ่งเป็น 2 ตอน คือ (1) เว็บเซอร์วิสในปัจจุบัน ครอบคลุมการบูรณาการบนสถาปัตยกรรมเชิงบริการ (SOA) ตัวอย่างการประยุกต์ใช้จริง และแนวคิดคลาวด์คอมพิวติง และ (2) แนวโน้มของเว็บเซอร์วิส เน้นซีแมนติกเว็บ (เว็บ 3.0) เทคโนโลยีหลัก (RDF, OWL, Ontology) และทิศทางการนำไปใช้ ทั้งกับ BPM ตลาดข้อมูล และกริดคอมพิวติง ภาพใหญ่คือธุรกิจจะปฏิรูปจากฐาน "สินค้า" ไปสู่ฐาน "บริการ"

แนวคิดสำคัญ

ปัญหาหลักขององค์กรคือระบบไอทีหลากหลายเทคโนโลยี (Java, .NET, Oracle, SAP) ทำงานร่วมกันยาก แนวคิด สถาปัตยกรรมเชิงบริการ (SOA) จึงออกแบบให้ระบบเป็น "หน่วยบริการ" ที่ทำงานประสานกันได้โดยไม่ยึดติดแพลตฟอร์ม และใช้ เว็บเซอร์วิส (มาตรฐาน XML/SOAP-REST/WSDL/UDDI) เป็นวิธีหลักในการลงมือทำให้เป็นจริง สำคัญต้องจำว่า SOA เป็นรูปแบบ/สไตล์ ส่วนเว็บเซอร์วิสเป็นวิธีการพัฒนา SOA นำไปทำให้เป็นผลได้ด้วยเว็บเซอร์วิส แต่ไม่จำเป็นต้องใช้เว็บเซอร์วิสเสมอ จากนั้นแนวคิดต่อยอดสู่คลาวด์คอมพิวติง (บริการคอมพิวเตอร์แบบสาธารณูปโภค) และซีแมนติกเว็บ (เว็บที่เข้าใจความหมายของข้อมูล)

เนื้อหาหลัก

15.1 เว็บเซอร์วิสในปัจจุบัน

การบูรณาการและสถาปัตยกรรมเชิงบริการ (SOA) เว็บเซอร์วิสใช้มาตรฐาน XML กำหนดรูปแบบข้อมูลเพื่อให้ระบบต่างแพลตฟอร์มทำงานร่วมกันได้ (interoperability) ขั้นตอนการบริการเว็บเซอร์วิสมี 5 ขั้น คือ ผู้ให้บริการพัฒนาบริการ → ลงทะเบียนใน UDDI → ผู้ใช้ค้นหาและได้เอกสาร WSDL → นำ WSDL ไปเรียนรู้การเรียกใช้ → ติดต่อเรียกใช้โดยแลกเปลี่ยนเอกสาร XML ผ่าน SOAP ในไทยกระทรวง ICT นำร่องโครงการ TH e-GIF (Thailand e-Government Interoperability Framework) กำหนดมาตรฐานกลาง เช่น SOAP, WSDL, UDDI, ebMS, WS-Security เพื่อให้หน่วยงานต่างแพลตฟอร์มแลกเปลี่ยนข้อมูลและทำงานร่วมกันได้อัตโนมัติ ประโยชน์ของบริการเว็บเซอร์วิสแบบบูรณาการ ได้แก่ เชื่อมโยงโปรแกรมที่หลากหลายด้วย XML, ลดค่าใช้จ่ายด้วยการนำส่วนประกอบซอฟต์แวร์กลับมาใช้ใหม่ (reuse), เรียกใช้จากโปรแกรมโดยตรงได้ และเรียกใช้ได้ทุกที่ทุกอุปกรณ์ทุกเวลา

เว็บเซอร์วิสกับการประยุกต์ใช้งาน ตัวอย่างเด่น ได้แก่

  • ธุรกิจท่องเที่ยว (Thailand เป็น tourism hub) รวมจองตั๋ว-โรงแรม-รถเช่า-ข้อมูล-การเงิน เช่น บริษัทเซาท์เวสต์
  • ธุรกิจประกันภัย เชื่อมโยงบริษัทประกันและหน่วยงานรัฐ (กรมการขนส่ง กรมการประกันภัย) เกิดธุรกิจแบบพันธมิตร
  • Amazon Web Services ตัวอย่างบริการ เช่น EC2 (เซิร์ฟเวอร์ประมวลผล), S3 (ที่เก็บข้อมูลบนคลาวด์), Amazon Elastic MapReduce (ประมวลผลข้อมูลขนาดใหญ่ราคาประหยัด เช่น ทำดัชนีเว็บ ทำเหมืองข้อมูล)
  • เว็บเซอร์วิสภาครัฐ เช่น กรมสรรพากร เปิด 8 บริการบนแพลตฟอร์ม .NET (เช่น CheckTINPINService, VATService), กรมศุลกากร ระบบ e-Export ไร้เอกสาร (paperless), กระทรวงศึกษาธิการ ระบบ XML Converter/KPI Reporter
  • เทคโนโลยี RFID ใช้คลื่นวิทยุระบุตัวตน/ติดตามสินค้า เหนือกว่าบาร์โค้ด ผสานเว็บเซอร์วิสติดตามการเคลื่อนที่ เช่น Wal-Mart + Gillette (smart shelves), สายการบิน Delta (ติดชิปกระเป๋า)
  • อุปกรณ์สื่อสารพกพา เว็บเซอร์วิสไม่จำกัดแค่เว็บไซต์ แต่เข้าถึงได้ผ่านสมาร์ตโฟน PDA Pocket PC

เว็บเซอร์วิสกับคลาวด์คอมพิวติง แนวคิดคือ "เว็บเซอร์วิสในลักษณะการบริการสาธารณะ" (web service as utility) เหมือนไฟฟ้า-น้ำประปา-โทรศัพท์ ผู้ใช้จ่ายตามการใช้งานจริง ไม่ต้องลงทุนฮาร์ดแวร์เอง คลาวด์คอมพิวติง (Cloud Computing) ย้ายการประมวลผลไปยังกลุ่มเซิร์ฟเวอร์จำนวนมากบนอินเทอร์เน็ต ใช้งานซอฟต์แวร์/ข้อมูลได้ทุกที่ทุกเวลา มีการกำหนด คุณภาพการบริการ (QoS: Quality of Service) ตาม ข้อตกลงระดับการบริการ (SLA: Service Level Agreement) ระหว่างผู้ให้และผู้ใช้บริการ ปัจจัยคุณภาพที่ W3C สรุปไว้ เช่น สมรรถนะ ความน่าเชื่อถือ ความพร้อมใช้งาน ความปลอดภัย ความสามารถทำงานร่วมระหว่างระบบ ตัวอย่างผู้ให้บริการคลาวด์/SaaS ที่นิยม เช่น NetSuite (ERP อันดับหนึ่งแบบ SaaS), Salesforce (CRM), ThinkFree, Zimbra, Zoho, Oracle CRM on Demand และบริการฟรีอย่าง Google Docs

15.2 แนวโน้มของเว็บเซอร์วิส

ซีแมนติกเว็บ (Semantic Web) = เว็บ 3.0 คือพัฒนาการของเวิลด์ไวด์เว็บที่ทำให้เว็บ "เข้าใจความหมาย" ของเนื้อหา ตอบสนองผู้ใช้อย่างชาญฉลาด และแสดงเฉพาะข้อมูลที่ตรงความต้องการ (ต่อยอดจากเว็บ 1.0 อ่านอย่างเดียว → เว็บ 2.0 ผู้ใช้ร่วมสร้าง → เว็บ 3.0 ฉลาดรู้) เสนอแนวคิดโดย Tim Berners-Lee มุ่งบูรณาการเนื้อหาทั่วโลกเป็นฐานข้อมูลขนาดใหญ่โดยใช้ "เมทาดาตา" (metadata) ลักษณะสำคัญคือ อ่าน-เขียน-ปฏิบัติการ, อ่าน-เขียน-เชื่อมโยง และเป็นเว็บเชิงความหมายที่มีลักษณะปัญญาประดิษฐ์ (AI)

เทคโนโลยีหลักสำหรับซีแมนติกเว็บ

  • RDF (Resource Description Framework): กรอบมาตรฐานนิยามแหล่งข้อมูลด้วยรูปแบบ 3 องค์ประกอบ subject-predicate-object หรือ object-property-value เขียนเป็น P(O,V) (ออบเจกต์ O มีคุณสมบัติ P ด้วยค่า V) ข้อมูลทั้งหมดในซีแมนติกเว็บใช้ RDF เป็นภาษาหลักและใช้ XML เป็นไวยากรณ์
  • RDFS (RDF Schema): ใช้นิยามคลาส ซับคลาส และจัดประเภท (taxonomies)
  • OWL (Web Ontology Language): ภาษาอธิบายข้อมูลเชิงออนโทโลจี แสดงความสัมพันธ์ของสิ่งต่าง ๆ เป็นส่วนต่อขยายจาก RDF
  • Ontology (ออนโทโลจี): องค์ความรู้ในขอบเขตที่สนใจ (domain knowledge) แบบลำดับชั้น (hierarchies) ใช้ร่วมกัน (share)/ใช้ซ้ำ (reuse)/ถ่ายทอดคุณสมบัติ (inheritance) ได้ นิยมใช้ในวงการแพทย์ เครื่องมือสร้างเช่น Protégé, Hozo
  • สถาปัตยกรรมซีแมนติกเว็บเป็นชั้น ๆ จากล่างขึ้นบน: URI/Unicode → XML → RDF → RDFS → OWL/Rules(RIF/SWRL) → SPARQL (ภาษาคิวรีคล้าย SQL) → Unifying logic → Proof → Cryptography → Trust → User interface

ทิศทางการนำเว็บเซอร์วิสมาใช้งาน ปฏิรูปธุรกิจจากฐาน "สินค้า" สู่ฐาน "บริการ" (business transformation) มุ่งสร้างนวัตกรรมการบริการแบบเบ็ดเสร็จในขั้นตอนเดียว โดยอาศัยเทคโนโลยีใหม่ ได้แก่ แมชอัป (mashup), SOA, คลาวด์คอมพิวติง, เว็บเซอร์วิส, เว็บ 2.0 และเว็บ 3.0 การประยุกต์สำคัญ ได้แก่

  • การจัดการกระบวนการธุรกิจ (BPM) วิวัฒนาการ SOA 4 ยุค (Single tier → N-tier → ยุคเว็บเซอร์วิส → ยุค SOA) ใช้ภาษา BPEL (Business Process Execution Language) ซึ่งเป็น XML ขับเคลื่อนกระบวนการธุรกิจบนเว็บเซอร์วิส BPM กับเว็บเซอร์วิสเป็นส่วนเติมเต็มกัน
  • ตลาดข้อมูล (Data Market) บริการรูปแบบใหม่บนคลาวด์ที่รวมข้อมูล เว็บเซอร์วิส และการวิเคราะห์มาไว้ที่เดียว สร้างด้วยสถาปัตยกรรม REST รองรับ API เกิด ดาตาแมชอัป (Data mash-up) เช่น รวมข้อมูลกับ Google Maps / Bing Maps
  • กริดคอมพิวติง (Grid Computing) เครือข่ายที่กระจายและแบ่งปันทรัพยากรคอมพิวเตอร์ (CPU, storage, network) เหมือนกริดไฟฟ้า ลดเวลาประมวลผลงานหนักจากเป็นปีเหลือเป็นวัน/ชั่วโมง แนวโน้มมุ่งสู่ "กริดเซอร์วิส" บนมาตรฐาน XML

ประเด็นที่ควรเน้นเพื่อสอบ

  • ความสัมพันธ์ SOA กับเว็บเซอร์วิส — SOA เป็นรูปแบบ/สไตล์, เว็บเซอร์วิสเป็นวิธีการ; SOA ทำให้เป็นจริงด้วยเว็บเซอร์วิสได้แต่ไม่จำเป็นต้องใช้เสมอ (ออกสอบบ่อยและสับสนง่าย)
  • เทคโนโลยีหลัก 3 ของเว็บเซอร์วิส: SOAP/REST (สื่อสาร) + WSDL (อธิบายบริการ) + UDDI (ค้นหา) และ 5 ขั้นตอนการบริการเว็บเซอร์วิส
  • มาตรฐานกลาง TH e-GIF และเป้าหมายบริการร่วม/เบ็ดเสร็จของภาครัฐไทย
  • แนวคิด คลาวด์คอมพิวติงในฐานะสาธารณูปโภค (web service as utility) และความหมาย QoS กับ SLA
  • ตัวอย่าง SaaS: NetSuite (ERP), Salesforce (CRM), Google Docs และบริการ AWS (EC2, S3, MapReduce)
  • ซีแมนติกเว็บ = เว็บ 3.0 ความหมายและพัฒนาการเว็บ 1.0 → 2.0 → 3.0
